       <div class="d-flex flex-items-baseline flex-justify-between mt-3"> 
        <h1 class="border-bottom-0">Managing deploy keys</h1> 
        <div class="d-none d-lg-block ml-2"> <button class="btn-link Link--secondary js-print tooltipped tooltipped-n" aria-label="Print this article"> <!-- From https://heroicons.dev/ --> 
          <svg fill="none" height="18" width="18" stroke-linecap="round" stroke-linejoin="round" stroke-width="2" viewbox="0 0 24 24" stroke="currentColor">
           <path d="M17 17h2a2 2 0 002-2v-4a2 2 0 00-2-2H5a2 2 0 00-2 2v4a2 2 0 002 2h2m2 4h6a2 2 0 002-2v-4a2 2 0 00-2-2H9a2 2 0 00-2 2v4a2 2 0 002 2zm8-12V5a2 2 0 00-2-2H9a2 2 0 00-2 2v4h10z"></path>
          </svg> </button> 
        </div> 
       </div> 
       <div class="lead-mktg">
        <p>Learn different ways to manage SSH keys on your servers when you automate deployment scripts and which way is best for you.</p>
       </div> 
      </div> 
      <div class="article-grid-toc border-bottom border-xl-0 pb-4 mb-5 pb-xl-0 mb-xl-0"> 
       <div class="article-grid-toc-content"> 
        <h2 id="in-this-article" class="f5 mb-2"><a class="Link--primary" href="#in-this-article">In this article</a></h2> 
        <ul class="list-style-none pl-0 f5 mb-0"> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#ssh-agent-forwarding">SSH agent forwarding</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#https-cloning-with-oauth-tokens">HTTPS cloning with OAuth tokens</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#deploy-keys">Deploy keys</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#server-to-server-tokens">Server-to-server tokens</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#machine-users">Machine users</a></li> 
        </ul> 
       </div> 
      </div> 
      <div id="article-contents" class="article-grid-body"> 
       <p>You can manage SSH keys on your servers when automating deployment scripts using SSH agent forwarding, HTTPS with OAuth tokens, deploy keys, or machine users.</p> 
       <h3 id="ssh-agent-forwarding"><a href="#ssh-agent-forwarding">SSH agent forwarding</a></h3> 
       <p>In many cases, especially in the beginning of a project, SSH agent forwarding is the quickest and simplest method to use. Agent forwarding uses the same SSH keys that your local development computer uses.</p> 
       <h5 id="pros"><a href="#pros">Pros</a></h5> 
       <ul> 
        <li>You do not have to generate or keep track of any new keys.</li> 
        <li>There is no key management; users have the same permissions on the server that they do locally.</li> 
        <li>No keys are stored on the server, so in case the server is compromised, you don't need to hunt down and remove the compromised keys.</li> 
       </ul> 
       <h5 id="cons"><a href="#cons">Cons</a></h5> 
       <ul> 
        <li>Users <strong>must</strong> SSH in to deploy; automated deploy processes can't be used.</li> 
        <li>SSH agent forwarding can be troublesome to run for Windows users.</li> 
       </ul> 
       <h5 id="setup"><a href="#setup">Setup</a></h5> 
       <ol> 
        <li>Turn on agent forwarding locally. See <a href="/en/guides/using-ssh-agent-forwarding">our guide on SSH agent forwarding</a> for more information.</li> 
        <li>Set your deploy scripts to use agent forwarding. For example, on a bash script, enabling agent forwarding would look something like this: <code>ssh -A serverA 'bash -s' &lt; deploy.sh</code></li> 
       </ol> 
       <h3 id="https-cloning-with-oauth-tokens"><a href="#https-cloning-with-oauth-tokens">HTTPS cloning with OAuth tokens</a></h3> 
       <p>If you don't want to use SSH keys, you can use <a href="/en/articles/git-automation-with-oauth-tokens">HTTPS with OAuth tokens</a>.</p> 
       <h5 id="pros-1"><a href="#pros-1">Pros</a></h5> 
       <ul> 
        <li>Anyone with access to the server can deploy the repository.</li> 
        <li>Users don't have to change their local SSH settings.</li> 
        <li>Multiple tokens (one for each user) are not needed; one token per server is enough.</li> 
        <li>A token can be revoked at any time, turning it essentially into a one-use password.</li> 
       </ul> 
       <h5 id="cons-1"><a href="#cons-1">Cons</a></h5> 
       <ul> 
        <li>You must make sure that you configure your token with the correct access scopes.</li> 
        <li>Tokens are essentially passwords, and must be protected the same way.</li> 
       </ul> 
       <h5 id="setup-1"><a href="#setup-1">Setup</a></h5> 
       <p>See <a href="/en/articles/git-automation-with-oauth-tokens">our guide on Git automation with tokens</a>.</p> 
       <h3 id="deploy-keys"><a href="#deploy-keys">Deploy keys</a></h3> 
       <p>You can launch projects from a GitHub repository to your server by using a deploy key, which is an SSH key that grants access to a single repository. GitHub attaches the public part of the key directly to your repository instead of a personal user account, and the private part of the key remains on your server. For more information, see "<a href="/en/rest/guides/delivering-deployments">Delivering deployments</a>."</p> 
       <p>Deploy keys with write access can perform the same actions as an organization member with admin access, or a collaborator on a personal repository. For more information, see "<a href="/en/articles/repository-permission-levels-for-an-organization">Repository permission levels for an organization</a>" and "<a href="/en/articles/permission-levels-for-a-user-account-repository">Permission levels for a user account repository</a>."</p> 
       <h5 id="pros-2"><a href="#pros-2">Pros</a></h5> 
       <ul> 
        <li>Anyone with access to the repository and server has the ability to deploy the project.</li> 
        <li>Users don't have to change their local SSH settings.</li> 
        <li>Deploy keys are read-only by default, but you can give them write access when adding them to a repository.</li> 
       </ul> 
       <h5 id="cons-2"><a href="#cons-2">Cons</a></h5> 
       <ul> 
        <li>Deploy keys only grant access to a single repository. More complex projects may have many repositories to pull to the same server.</li> 
        <li>Deploy keys are usually not protected by a passphrase, making the key easily accessible if the server is compromised.</li> 
       </ul> 
       <h5 id="setup-2"><a href="#setup-2">Setup</a></h5> 
       <ol> 
        <li><a href="/en/articles/generating-a-new-ssh-key-and-adding-it-to-the-ssh-agent/#generating-a-new-ssh-key">Run the <code>ssh-keygen</code> procedure</a> on your server, and remember where you save the generated public/private rsa key pair.</li> 
        <li>In the upper-right corner of any GitHub page, click your profile photo, then click <strong>Your profile</strong>. <span class="procedural-image-wrapper"><img src="/assets/images/profile-page.png" alt="Navigation to profile"></span></li> 
        <li>On your profile page, click <strong>Repositories</strong>, then click the name of your repository. <span class="procedural-image-wrapper"><img src="/assets/images/repos.png" alt="Repositories link"></span></li> 
        <li>From your repository, click <strong>Settings</strong>. <span class="procedural-image-wrapper"><img src="/assets/images/repo-settings.png" alt="Repository settings"></span></li> 
        <li>In the sidebar, click <strong>Deploy Keys</strong>, then click <strong>Add deploy key</strong>. <span class="procedural-image-wrapper"><img src="/assets/images/add-deploy-key.png" alt="Add Deploy Keys link"></span></li> 
        <li>Provide a title, paste in your public key. <span class="procedural-image-wrapper"><img src="/assets/images/deploy-key.png" alt="Deploy Key page"></span></li> 
        <li>Select <strong>Allow write access</strong> if you want this key to have write access to the repository. A deploy key with write access lets a deployment push to the repository.</li> 
        <li>Click <strong>Add key</strong>.</li> 
       </ol> 
       <h5 id="using-multiple-repositories-on-one-server"><a href="#using-multiple-repositories-on-one-server">Using multiple repositories on one server</a></h5> 
       <p>If you use multiple repositories on one server, you will need to generate a dedicated key pair for each one. You can't reuse a deploy key for multiple repositories.</p> 
       <p>In the server's SSH configuration file (usually <code>~/.ssh/config</code>), add an alias entry for each repository. For example:</p> 
       <pre><code class="hljs language-bash">Host github.com-repo-0
        Hostname github.com
        IdentityFile=/home/user/.ssh/repo-0_deploy_key

Host github.com-repo-1
        Hostname github.com
        IdentityFile=/home/user/.ssh/repo-1_deploy_key
</code></pre> 
       <ul> 
        <li><code>Host github.com-repo-0</code> - The repository's alias.</li> 
        <li><code>Hostname github.com</code> - Configures the hostname to use with the alias.</li> 
        <li><code>IdentityFile=/home/user/.ssh/repo-0_deploy_key</code> - Assigns a private key to the alias.</li> 
       </ul> 
       <p>You can then use the hostname's alias to interact with the repository using SSH, which will use the unique deploy key assigned to that alias. For example:</p> 
       <pre><code class="hljs language-bash">$ git <span class="hljs-built_in">clone</span> git@github.com-repo-1:OWNER/repo-1.git
</code></pre> 
       <h3 id="server-to-server-tokens"><a href="#server-to-server-tokens">Server-to-server tokens</a></h3> 
       <p>If your server needs to access repositories across one or more organizations, you can use a GitHub App to define the access you need, and then generate <em>tightly-scoped</em>, <em>server-to-server</em> tokens from that GitHub App. The server-to-server tokens can be scoped to single or multiple repositories, and can have fine-grained permissions. For example, you can generate a token with read-only access to a repository's contents. </p> 
       <p>Since GitHub Apps are a first class actor on GitHub, the server-to-server tokens are decoupled from any GitHub user, which makes them comparable to "service tokens". Additionally, server-to-server tokens have dedicated rate limits that scale with the size of the organizations that they act upon. For more information, see <a href="/en/developers/apps/rate-limits-for-github-apps">Rate limits for Github Apps</a>.</p> 
       <h5 id="pros-3"><a href="#pros-3">Pros</a></h5> 
       <ul> 
        <li>Tightly-scoped tokens with well-defined permission sets and expiration times (1 hour, or less if revoked manually using the API).</li> 
        <li>Dedicated rate limits that grow with your organization.</li> 
        <li>Decoupled from GitHub user identities, so they do not consume any licensed seats.</li> 
        <li>Never granted a password, so cannot be directly signed in to.</li> 
       </ul> 
       <h5 id="cons-3"><a href="#cons-3">Cons</a></h5> 
       <ul> 
        <li>Additional setup is needed to create the GitHub App.</li> 
        <li>Server-to-server tokens expire after 1 hour, and so need to be re-generated, typically on-demand using code.</li> 
       </ul> 
       <h5 id="setup-3"><a href="#setup-3">Setup</a></h5> 
       <ol> 
        <li>Determine if your GitHub App should be public or private. If your GitHub App will only act on repositories within your organization, you likely want it private.</li> 
        <li>Determine the permissions your GitHub App requires, such as read-only access to repository contents.</li> 
        <li>Create your GitHub App via your organization's settings page. For more information, see <a href="/en/developers/apps/creating-a-github-app">Creating a GitHub App</a>.</li> 
        <li>Note your GitHub App <code>id</code>.</li> 
        <li>Generate and download your GitHub App's private key, and store this safely. For more information, see <a href="/en/developers/apps/authenticating-with-github-apps#generating-a-private-key">Generating a private key</a>.</li> 
        <li>Install your GitHub App on the repositories it needs to act upon, optionally you may install the GitHub App on all repositories in your organization.</li> 
        <li>Identify the <code>installation_id</code> that represents the connection between your GitHub App and the organization repositories it can access. Each GitHub App and organization pair have at most a single <code>installation_id</code>. You can identify this <code>installation_id</code> via <a href="/en/rest/reference/apps#get-an-organization-installation-for-the-authenticated-app">Get an organization installation for the authenticated app</a>. This requires authenticating as a GitHub App using a JWT, for more information see <a href="/en/developers/apps/authenticating-with-github-apps#authenticating-as-a-github-app">Authenticating as a GitHub App</a>.</li> 
        <li>Generate a server-to-server token using the corresponding REST API endpoint, <a href="/en/rest/reference/apps#create-an-installation-access-token-for-an-app">Create an installation access token for an app</a>. This requires authenticating as a GitHub App using a JWT, for more information see <a href="/en/developers/apps/authenticating-with-github-apps#authenticating-as-a-github-app">Authenticating as a GitHub App</a>, and <a href="/en/developers/apps/authenticating-with-github-apps#authenticating-as-an-installation">Authenticating as an installation</a>.</li> 
        <li>Use this server-to-server token to interact with your repositories, either via the REST or GraphQL APIs, or via a Git client.</li> 
       </ol> 
       <h3 id="machine-users"><a href="#machine-users">Machine users</a></h3> 
       <p>If your server needs to access multiple repositories, you can create a new GitHub account and attach an SSH key that will be used exclusively for automation. Since this GitHub account won't be used by a human, it's called a <em>machine user</em>. You can add the machine user as a <a href="/en/articles/inviting-collaborators-to-a-personal-repository">collaborator</a> on a personal repository (granting read and write access), as an <a href="/en/articles/adding-outside-collaborators-to-repositories-in-your-organization">outside collaborator</a> on an organization repository (granting read, write, or admin access), or to a <a href="/en/articles/adding-organization-members-to-a-team">team</a> with access to the repositories it needs to automate (granting the permissions of the team).</p> 
       <div class="extended-markdown tip border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Tip:</strong> Our <a href="/en/articles/github-terms-of-service">terms of service</a> state:</p> 
        <blockquote> 
         <p><em>Accounts registered by "bots" or other automated methods are not permitted.</em></p> 
        </blockquote> 
        <p>This means that you cannot automate the creation of accounts. But if you want to create a single machine user for automating tasks such as deploy scripts in your project or organization, that is totally cool.</p> 
       </div>

       <ul> 
        <li>Anyone with access to the repository and server has the ability to deploy the project.</li> 
        <li>No (human) users need to change their local SSH settings.</li> 
        <li>Multiple keys are not needed; one per server is adequate.</li>
