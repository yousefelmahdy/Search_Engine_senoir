       <h3 id="introduction"><a href="#introduction">Introduction</a></h3> 
       <p>This guide helps you migrate from Travis CI to GitHub Actions. It compares their concepts and syntax, describes the similarities, and demonstrates their different approaches to common tasks.</p> 
       <h3 id="before-you-start"><a href="#before-you-start">Before you start</a></h3> 
       <p>Before starting your migration to GitHub Actions, it would be useful to become familiar with how it works:</p> 
       <ul> 
        <li>For a quick example that demonstrates a GitHub Actions job, see "<a href="/en/actions/quickstart">Quickstart for GitHub Actions</a>."</li> 
        <li>To learn the essential GitHub Actions concepts, see "<a href="/en/actions/learn-github-actions/introduction-to-github-actions">Introduction to GitHub Actions</a>."</li> 
       </ul> 
       <h3 id="comparing-job-execution"><a href="#comparing-job-execution">Comparing job execution</a></h3> 
       <p>To give you control over when CI tasks are executed, a GitHub Actions <em>workflow</em> uses <em>jobs</em> that run in parallel by default. Each job contains <em>steps</em> that are executed in a sequence that you define. If you need to run setup and cleanup actions for a job, you can define steps in each job to perform these.</p> 
       <h3 id="key-similarities"><a href="#key-similarities">Key similarities</a></h3> 
       <p>GitHub Actions and Travis CI share certain similarities, and understanding these ahead of time can help smooth the migration process.</p> 
       <h4 id="using-yaml-syntax"><a href="#using-yaml-syntax">Using YAML syntax</a></h4> 
       <p>Travis CI and GitHub Actions both use YAML to create jobs and workflows, and these files are stored in the code's repository. For more information on how GitHub Actions uses YAML, see <a href="/en/actions/learn-github-actions/introduction-to-github-actions#create-an-example-workflow">"Creating a workflow file</a>."</p> 
       <h4 id="custom-environment-variables"><a href="#custom-environment-variables">Custom environment variables</a></h4> 
       <p>Travis CI lets you set environment variables and share them between stages. Similarly, GitHub Actions lets you define environment variables for a step, job, or workflow. For more information, see <a href="/en/actions/reference/environment-variables">"Environment variables</a>."</p> 
       <h4 id="default-environment-variables"><a href="#default-environment-variables">Default environment variables</a></h4> 
       <p>Travis CI and GitHub Actions both include default environment variables that you can use in your YAML files. For GitHub Actions, you can see these listed in "<a href="/en/actions/reference/environment-variables#default-environment-variables">Default environment variables</a>."</p> 
       <h4 id="parallel-job-processing"><a href="#parallel-job-processing">Parallel job processing</a></h4> 
       <p>Travis CI can use <code>stages</code> to run jobs in parallel. Similarly, GitHub Actions runs <code>jobs</code> in parallel. For more information, see "<a href="/en/actions/learn-github-actions/managing-complex-workflows#creating-dependent-jobs">Creating dependent jobs</a>."</p> 
       <h4 id="status-badges"><a href="#status-badges">Status badges</a></h4> 
       <p>Travis CI and GitHub Actions both support status badges, which let you indicate whether a build is passing or failing. For more information, see <a href="/en/actions/managing-workflow-runs/adding-a-workflow-status-badge">"Adding a workflow status badge to your repository</a>."</p> 
       <h4 id="using-a-build-matrix"><a href="#using-a-build-matrix">Using a build matrix</a></h4> 
       <p>Travis CI and GitHub Actions both support a build matrix, allowing you to perform testing using combinations of operating systems and software packages. For more information, see "<a href="/en/actions/learn-github-actions/managing-complex-workflows#using-a-build-matrix">Using a build matrix</a>."</p>

       <p>Travis CI and GitHub Actions both allow you to target your CI to a specific branch. For more information, see "<a href="/en/actions/reference/workflow-syntax-for-github-actions#onpushpull_requestbranchestags">Workflow syntax for GitHub Actions</a>."</p>

       <p>Travis CI and GitHub Actions can both add custom environment variables to a test matrix, which allows you to refer to the variable in a later step.</p>

       <h3 id="key-features-in-github-actions"><a href="#key-features-in-github-actions">Key features in GitHub Actions</a></h3> 
       <p>When migrating from Travis CI, consider the following key features in GitHub Actions:</p> 
       <h4 id="storing-secrets"><a href="#storing-secrets">Storing secrets</a></h4> 
       <p>GitHub Actions allows you to store secrets and reference them in your jobs. GitHub Actions organizations can limit which repositories can access organization secrets. Environment protection rules can require manual approval for a workflow to access environment secrets. For more information, see "<a href="/en/actions/reference/encrypted-secrets">Encrypted secrets</a>."</p> 
       <h4 id="sharing-files-between-jobs-and-workflows"><a href="#sharing-files-between-jobs-and-workflows">Sharing files between jobs and workflows</a></h4> 
       <p>GitHub Actions includes integrated support for artifact storage, allowing you to share files between jobs in a workflow. You can also save the resulting files and share them with other workflows. For more information, see "<a href="/en/actions/learn-github-actions/essential-features-of-github-actions#sharing-data-between-jobs">Sharing data between jobs</a>."</p> 
       <h4 id="hosting-your-own-runners"><a href="#hosting-your-own-runners">Hosting your own runners</a></h4> 
       <p>If your jobs require specific hardware or software, GitHub Actions allows you to host your own runners and send your jobs to them for processing. GitHub Actions also lets you use policies to control how these runners are accessed, granting access at the organization or repository level. For more information, see <a href="/en/actions/hosting-your-own-runners">"Hosting your own runners</a>."</p> 
       <h4 id="concurrent-jobs-and-execution-time"><a href="#concurrent-jobs-and-execution-time">Concurrent jobs and execution time</a></h4> 
       <p>The concurrent jobs and workflow execution times in GitHub Actions can vary depending on your GitHub plan. For more information, see "<a href="/en/actions/reference/usage-limits-billing-and-administration">Usage limits, billing, and administration</a>."</p> 
       <h4 id="using-different-languages-in-github-actions"><a href="#using-different-languages-in-github-actions">Using different languages in GitHub Actions</a></h4> 
       <p>When working with different languages in GitHub Actions, you can create a step in your job to set up your language dependencies. For more information about working with a particular language, see the specific guide:</p> 
       <ul> 
        <li><a href="/en/actions/guides/building-and-testing-nodejs">Building and testing Node.js</a></li> 
        <li><a href="/en/actions/guides/building-and-testing-powershell">Building and testing PowerShell</a></li> 
        <li><a href="/en/actions/guides/building-and-testing-python">Building and testing Python</a></li> 
        <li><a href="/en/actions/guides/building-and-testing-java-with-maven">Building and testing Java with Maven</a></li> 
        <li><a href="/en/actions/guides/building-and-testing-java-with-gradle">Building and testing Java with Gradle</a></li>

       <h3 id="executing-scripts"><a href="#executing-scripts">Executing scripts</a></h3> 
       <p>GitHub Actions can use <code>run</code> steps to run scripts or shell commands. To use a particular shell, you can specify the <code>shell</code> type when providing the path to the script. For more information, see "<a href="/en/actions/reference/workflow-syntax-for-github-actions#jobsjob_idstepsrun">Workflow syntax for GitHub Actions</a>."</p> 
       <p>For example:</p> 
       <pre><code class="hljs language-yaml"><span class="hljs-attr">steps:</span>

       <h3 id="error-handling-in-github-actions"><a href="#error-handling-in-github-actions">Error handling in GitHub Actions</a></h3> 
       <p>When migrating to GitHub Actions, there are different approaches to error handling that you might need to be aware of.</p> 
       <h4 id="script-error-handling"><a href="#script-error-handling">Script error handling</a></h4> 
       <p>GitHub Actions stops a job immediately if one of the steps returns an error code. For more information, see "<a href="/en/actions/reference/workflow-syntax-for-github-actions#exit-codes-and-error-action-preference">Workflow syntax for GitHub Actions</a>."</p> 
       <h4 id="job-error-handling"><a href="#job-error-handling">Job error handling</a></h4> 
       <p>GitHub Actions uses <code>if</code> conditionals to execute jobs or steps in certain situations. For example, you can run a step when another step results in a <code>failure()</code>. For more information, see "<a href="/en/actions/reference/workflow-syntax-for-github-actions#example-using-status-check-functions">Workflow syntax for GitHub Actions</a>." You can also use <a href="/en/actions/reference/workflow-syntax-for-github-actions#jobsjob_idcontinue-on-error"><code>continue-on-error</code></a> to prevent a workflow run from stopping when a job fails.</p> 
       <h3 id="migrating-syntax-for-conditionals-and-expressions"><a href="#migrating-syntax-for-conditionals-and-expressions">Migrating syntax for conditionals and expressions</a></h3> 
       <p>To run jobs under conditional expressions, Travis CI and GitHub Actions share a similar <code>if</code> condition syntax. GitHub Actions lets you use the <code>if</code> conditional to prevent a job or step from running unless a condition is met. For more information, see "<a href="/en/actions/reference/context-and-expression-syntax-for-github-actions">Context and expression syntax for GitHub Actions</a>."</p> 
       <p>This example demonstrates how an <code>if</code> conditional can control whether a step is executed:</p> 
       <pre><code class="hljs language-yaml"><span class="hljs-attr">jobs:</span>
  <span class="hljs-attr">conditional:</span>
    <span class="hljs-attr">runs-on:</span> <span class="hljs-string">ubuntu-latest</span>
    <span class="hljs-attr">steps:</span>
      <span class="hljs-bullet">-</span> <span class="hljs-attr">run:</span> <span class="hljs-string">echo</span> <span class="hljs-string">"This step runs with str equals 'ABC' and num equals 123"</span>
        <span class="hljs-attr">if:</span> <span class="hljs-string">env.str</span> <span class="hljs-string">==</span> <span class="hljs-string">'ABC'</span> <span class="hljs-string">&amp;&amp;</span> <span class="hljs-string">env.num</span> <span class="hljs-string">==</span> <span class="hljs-number">123</span>
</code></pre>
