       <h3 id="introduction"><a href="#introduction">Introduction</a></h3> 
       <p>This guide shows you how to set up your Java project in Codespaces. It will take you through an example of opening your project in a codespace, and adding and modifying a dev container configuration from a template.</p> 
       <h4 id="prerequisites"><a href="#prerequisites">Prerequisites</a></h4> 
       <ul> 
        <li>You should have an existing Java project in a repository on GitHub.com. If you don't have a project, you can try this tutorial with the following example: <a href="https://github.com/microsoft/vscode-remote-try-java">https://github.com/microsoft/vscode-remote-try-java</a></li> 
        <li>You must have Codespaces enabled for your organization.</li> 
       </ul>

       <p>When you create a codespace, your project is created on a remote VM that is dedicated to you. By default, the container for your codespace has many languages and runtimes including Java, nvm, npm, and yarn. It also includes a common set of tools like git, wget, rsync, openssh, and nano. </p> 
       <p>You can customize your codespace by adjusting the amount of vCPUs and RAM, <a href="/en/codespaces/setting-up-your-codespace/personalizing-codespaces-for-your-account">adding dotfiles to personalize your environment</a>, or by modifying the tools and scripts installed. </p> 
       <p>Codespaces uses a file called <code>devcontainer.json</code> to store configurations. On launch Codespaces uses the file to install any tools, dependencies, or other set up that might be needed for the project. For more information, see "<a href="/en/codespaces/setting-up-your-codespace/configuring-codespaces-for-your-project">Configuring Codespaces for your project</a>."</p> 
       <h3 id="step-2-add-a-dev-container-to-your-codespace-from-a-template"><a href="#step-2-add-a-dev-container-to-your-codespace-from-a-template">Step 2: Add a dev container to your codespace from a template</a></h3> 
       <p>The default codespaces container comes with the latest Java version, package managers (Maven, Gradle), and other common tools preinstalled. However, we recommend that you set up a custom container to define the tools and scripts that your project needs. This will ensure a fully reproducible environment for all Codespaces users in your repository.</p> 
       <p>To set up your project with a custom container, you will need to use a <code>devcontainer.json</code> file to define the environment. In Codespaces you can add this either from a template or you can create your own. For more information on dev containers, see "<a href="/en/codespaces/setting-up-your-codespace/configuring-codespaces-for-your-project">Configuring Codespaces for your project</a>."</p> 
       <ol> 
        <li>Access the command palette (<code>shift command P</code> / <code>shift control P</code>), then start typing "dev container". Click <strong>Codespaces: Add Development Container Configuration Files...</strong> <span class="procedural-image-wrapper"><img src="/assets/images/help/codespaces/add-prebuilt-container-command.png" alt="&quot;Codespaces: Add Development Container Configuration Files...&quot; in the command palette"></span></li> 
        <li>For this example, click <strong>Java</strong>. In practice, you could select any container that’s specific to Java or a combination of tools such as Java and Azure Functions. <span class="procedural-image-wrapper"><img src="/assets/images/help/codespaces/add-java-prebuilt-container.png" alt="Select Java option from the list"></span></li> 
        <li>Click the recommended version of Java. <span class="procedural-image-wrapper"><img src="/assets/images/help/codespaces/add-java-version.png" alt="Java version selection"></span></li> 
        <li>To rebuild your container, access the command palette (<code>shift command P</code> / <code>shift control P</code>), then start typing "rebuild". Click <strong>Codespaces: Rebuild Container</strong>. <span class="procedural-image-wrapper"><img src="/assets/images/help/codespaces/codespaces-rebuild.png" alt="Rebuild container option"></span></li> 
       </ol> 
       <h4 id="anatomy-of-your-dev-container"><a href="#anatomy-of-your-dev-container">Anatomy of your dev container</a></h4> 
       <p>Adding the Java dev container template adds a <code>.devcontainer</code> folder to the root of your project's repository with the following files:</p> 
       <ul> 
        <li><code>devcontainer.json</code></li> 
        <li>Dockerfile</li> 
       </ul> 
       <p>The newly added <code>devcontainer.json</code> file defines a few properties that are described after the sample.</p>

       <ul> 
        <li><strong>Name</strong> - You can name your dev container anything, this is just the default.</li> 
        <li><strong>Build</strong> - The build properties.
         <ul> 
          <li><strong>Dockerfile</strong> - In the build object, dockerfile is a reference to the Dockerfile that was also added from the template.</li> 
          <li><strong>Args</strong>
           <ul> 
            <li><strong>Variant</strong>: This file only contains one build argument, which is the Java version that is passed into the Dockerfile.</li> 
           </ul></li> 
         </ul></li> 
        <li><strong>Settings</strong> - These are Visual Studio Code settings that you can set.
         <ul> 
          <li><strong>Terminal.integrated.shell.linux</strong> - While bash is the default here, you could use other terminal shells by modifying this.</li> 
         </ul></li> 
        <li><strong>Extensions</strong> - These are extensions included by default.
         <ul> 
          <li><strong>Vscjava.vscode-java-pack</strong> - The Java Extension Pack provides popular extensions for Java development to get you started.</li> 
         </ul></li> 
        <li><strong>forwardPorts</strong> - Any ports listed here will be forwarded automatically.</li> 
        <li><strong>postCreateCommand</strong> - If you want to run anything after you land in your codespace that’s not defined in the Dockerfile, you can do that here.</li> 
        <li><strong>remoteUser</strong> - By default, you’re running as the <code>vscode</code> user, but you can optionally set this to <code>root</code>.</li> 
       </ul>

       <h3 id="step-5-commit-your-changes"><a href="#step-5-commit-your-changes">Step 5: Commit your changes</a></h3> 
       <p>Once you've made changes to your codespace, either new code or configuration changes, you'll want to commit your changes. Committing changes to your repository ensures that anyone else who creates a codespace from this repository has the
