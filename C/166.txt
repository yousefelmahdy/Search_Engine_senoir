              <h3><a id="user-content-setup" class="anchor" aria-hidden="true" href="#setup">

              <p>No matter how you wish to contribute, you'll need <a href="https://github.com/join">a GitHub account</a> if you don't have one already.               <ol>

This is the easiest approach if you just want to make a simple change to a single file, like fixing a typo.</li> 
               <li>Install and use the <a href="https://docs.github.com/en/free-pro-team@latest/github/getting-started-with-github/github-desktop">GitHub Desktop</a></li> 
               <li>Install and use the <a href="https://docs.github.com/en/free-pro-team@latest/github/getting-started-with-github/github-cli">GitHub CLI</a></li> 
               <li><a href="https://git-scm.com/downloads" rel="nofollow">Install <code>git</code></a> and use it from the command line. You might find these resources helpful: 
                <ul> 
                 <li><a href="https://docs.github.com/en/free-pro-team@latest/github/using-git">Using Git</a></li> 
                 <li><a href="https://training.github.com/"><code>git</code> cheatsheet</a></li> 
                </ul> </li> 
              </ol> 
              <p>If you choose an option other than the GitHub UI, you want to install <a href="https://nodejs.org" rel="nofollow">Node.js (version &gt;=12.11)</a> and <a href="https://classic.yarnpkg.com/en/docs/install" rel="nofollow"><code>yarn</code></a>.</p> 
              <h3><a id="user-content-fundamental-concepts" class="anchor" aria-hidden="true" href="#fundamental-concepts">
                <svg class="octicon octicon-link" viewbox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true">
                 <path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path>
                </svg></a>Fundamental concepts</h3> 
              <p>These are some important things to keep in mind about the MDN content.</p> 
              <ul> 
               <li> <p><strong>Documents are folders</strong> -- Documents are always represented by a folder (e.g., <a href="/mdn/content/blob/main/files/en-us/web/javascript"><code>files/en-us/web/javascript</code></a>), and that folder will contain the content of that specific document as an <code>index.html</code> file (e.g., <a href="/mdn/content/blob/main/files/en-us/web/javascript/index.html"><code>files/en-us/web/javascript/index.html</code></a>).</p> </li> 
               <li> <p><strong>Documents are hierarchical</strong> - A document folder may contain other folders, where those folders would represent child documents (e.g., <a href="/mdn/content/blob/main/files/en-us/web/javascript/closures/index.html"><code>files/en-us/web/javascript/closures/index.html</code></a>).</p> </li> 
               <li> <p><strong>Document folders may contain image files</strong> -- A document folder may also contain image files, which are referenced within that document's <code>index.html</code> file.</p> </li> 
               <li> <p><strong>All redirects are specified in a single file</strong> -- All of the redirects are specified within <a href="/mdn/content/blob/main/files/en-us/_redirects.txt"><code>files/en-us/_redirects.txt</code></a>, one redirect per line. Each line specifies a <code>from</code> and <code>to</code> URI separated by whitespace. When you move a document, you'll need to add a redirect to this file specifying that its old URI now redirects to its new URI. Both of these tasks are done using the <code>yarn content move</code> tool  see <a href="#moving-one-or-more-documents">Moving one or more documents</a>. <strong>Don't edit the <code>_redirects.txt</code> file manually!</strong> If both an <code>index.html</code> file and a redirect exist for a document, the document takes precedence and the redirect is ignored.</p> </li> 
               <li> <p><strong>A document's <code>index.html</code> starts with "front-matter"</strong> -- Each document's <code>index.html</code> file must begin with some <a href="https://en.wikipedia.org/wiki/YAML" rel="nofollow">YAML</a> called front-matter that defines some important information about the document: <code>title</code>, <code>slug</code>, and <a href="https://developer.mozilla.org/en-US/docs/MDN/Contribute/Howto/Tag" rel="nofollow"><code>tags</code></a> (if any). Here's an example that shows the front-matter from the <a href="/mdn/content/blob/main/files/en-us/web/javascript/index.html">JavaScript landing page</a>:</p> 
                <div class="highlight highlight-source-yaml position-relative" data-snippet-clipboard-copy-content="---
title: JavaScript
slug: Web/JavaScript
tags:
  - JavaScript
  - Landing
  - Landing page
  - Learn
  - 'l10n:priority'
---
">
                 <pre>---
<span class="pl-ent">title</span>: <span class="pl-s">JavaScript</span>
<span class="pl-ent">slug</span>: <span class="pl-s">Web/JavaScript</span>
<span class="pl-ent">tags</span>:
  - <span class="pl-s">JavaScript</span>
  - <span class="pl-s">Landing</span>
  - <span class="pl-s">Landing page</span>
  - <span class="pl-s">Learn</span>
  - <span class="pl-s"><span class="pl-pds">'</span>l10n:priority<span class="pl-pds">'</span></span>
---</pre>
                </div> </li> 
              </ul> 
              <h3><a id="user-content-simple-changes" class="anchor" aria-hidden="true" href="#simple-changes">
                <svg class="octicon octicon-link" viewbox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true">
                 <path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path>
                </svg></a>Simple changes</h3> 
              <p>If you just want to make a simple change to a single file, like fixing a typo, the GitHub UI is the simplest way to do that. For example, if you've found a typo within the <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript" rel="nofollow">JavaScript landing page</a>, you can sign into GitHub, go to <a href="https://github.com/mdn/content">https://github.com/mdn/content</a>, navigate to the source file <code>files/en-us/web/javascript/index.html</code>, and then click on the edit (pencil) button.</p> 
              <blockquote> 
               <p><strong>Tip:</strong> Click the <strong>Source on GitHub</strong> link in the footer of any MDN page to jump to its source file on GitHub.</p> 
              </blockquote> 
              <p>From there the GitHub UI will take your hand and <a href="https://docs.github.com/en/free-pro-team@latest/github/managing-files-in-a-repository/editing-files-in-your-repository">walk you through the rest</a>, like automatically creating a <a href="https://docs.github.com/en/free-pro-team@latest/github/getting-started-with-github/fork-a-repo">fork</a> and branch to commit your changes to, as well as helping you reach the ultimate goal, a <a href="https://docs.github.com/en/free-pro-team@latest/github/collaborating-with-issues-and-pull-requests/about-pull-requests">pull request</a>. Your pull request represents the work you want to be reviewed, hopefully approved, and then merged into the <code>main</code> branch of this repository.</p> 
              <blockquote> 
               <p><strong>Note:</strong> See the <a href="#pull-request-etiquette">pull request etiquette section</a> for more details on creating and handling pull requests successfully.</p> 
              </blockquote> 
              <p>If you're not certain of the changes that you want to make, get in touch with us first! You can <a href="https://chat.mozilla.org/#/room/#mdn:mozilla.org" rel="nofollow">chat with us</a> or <a href="https://github.com/mdn/content/issues">file an issue</a>.</p> 
              <h3><a id="user-content-more-substantial-changes" class="anchor" aria-hidden="true" href="#more-substantial-changes">
                <svg class="octicon octicon-link" viewbox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true">
                 <path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path>
                </svg></a>More substantial changes</h3> 
              <p>If you need to do some work that requires changes to more than one file, like moving one or more documents, the GitHub UI is not very efficient. You'd have to make a separate pull request for every page you want to change. Instead, you're going to have to use <code>git</code> or one of the other <code>git</code>-based approaches like the <a href="https://docs.github.com/en/free-pro-team@latest/github/getting-started-with-github/github-desktop">GitHub Desktop</a>.</p> 
              <ol>

              <h3><a id="user-content-pull-request-etiquette" class="anchor" aria-hidden="true" href="#pull-request-etiquette">
                <svg class="octicon octicon-link" viewbox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true">
                 <path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path>
                </svg></a>Pull request etiquette</h3> 
              <p>Here are some important rules of etiquette to remember when working with pull requests.</p> 
              <ol> 
               <li> <p>When you submit a pull request, a number of tests are automatically run as GitHub Actions (see <a href="/mdn/content/blob/main/.github/workflows">.github/workflows</a>). If one or more of these tests fail, it is your responsibility to try and resolve the underlying issue(s). If you don't know how to resolve the underlying issue(s), you can ask for help. Your pull request will not be approved and merged if these tests are failing.</p> </li> 
               <li> <p>If your pull request has merge conflicts with the <code>main</code> branch (GitHub checks for this automatically and notifies you), you are responsible for resolving them. You can do this by merging the <code>main</code> branch into your branch (<code>git pull mdn main</code>), and then pushing the updated branch to your fork (<code>git push</code>).</p> </li> 
               <li> <p>Once you've created your pull request, never use <code>git rebase</code> on your branch if you need to make changes. Any changes should be made as additional commits.</p> </li> 
               <li> <p>Each pull request should contain a single logical change, or related set of changes that make sense to submit together. If a pull request becomes too large or contains too many unrelated changes, it becomes too difficult to review, and may begin to look suspicious (it is easier to hide malicious changes in a large pull request). In such cases, the reviewer has the right to close your pull request, and ask that you submit a separate pull request for each logical set of changes that belong together.</p> </li> 
               <li> <p>If your pull request contains any kind of significant complexity (it contains technical changes, and isn't just a typo fix, grammatical improvement, or formatting/structural change), please describe why you're making the change and anything else we need to know about it.</p> 
                <ul> 
                 <li>If the pull request is simple (it is really clear what has been changed and why, and the change is obviously a good thing), you can do this in your pull request's description.</li> 
                 <li>If the pull request is complex (the changes and the reasoning behind them need a bit more explanation), then the requestor should file an issue describing the intended change first, and seek discussion/approval as needed. When the time is right to submit the PR, they should reference the issue (or an existing issue that describes the motivation for the change) in the PR. You can reference an existing issue using <code>#</code> followed by the issue's ID, for example <code>#1234</code>.</li> 
                </ul> </li> 
               <li> <p>Do not re-open a pull request that a reviewer has closed.</p> </li> 
              </ol>

              <p>When you add images to MDN, you should make sure that they are compressed as much as possible without degrading quality, to save on download size for our readers.               <p>For example:</p>

                </svg></a>Adding code examples</h3> 
              <p>There are a lot of code examples on MDN, and you'll probably want to add some as you contribute content to the project. There are four types of code example used on MDN:</p> 
              <ul> 
               <li><a href="https://developer.mozilla.org/en-US/docs/MDN/Structures/Code_examples#Static_examples" rel="nofollow">Static examples</a>  plain code blocks, possibly with a screenshot to statically show the result of such code if it were to be run.</li> 
               <li><a href="https://developer.mozilla.org/en-US/docs/MDN/Structures/Code_examples#Traditional_live_samples" rel="nofollow">Traditional MDN "live samples"</a>  A macro that takes plain code blocks, dynamically puts them into a document inside an <code>&lt;iframe&gt;</code> element, and embeds it into the page to show the code running live.</li> 
               <li><a href="https://developer.mozilla.org/en-US/docs/MDN/Structures/Code_examples#GitHub_live_samples" rel="nofollow">GitHub "live samples"</a>  A macro that takes a document in a GitHub repo inside the <a href="https://github.com/mdn">mdn organization</a>, puts it inside an <code>&lt;iframe&gt;</code> element, and embeds it into the page to show the code running live.</li> 
               <li><a href="https://developer.mozilla.org/en-US/docs/MDN/Structures/Code_examples#Interactive_examples" rel="nofollow">Interactive examples</a>  Our system for creating live interactive examples that show the code running live but also allow you to change code on the fly to see what the effect is.</li> 
              </ul> 
              <p>For a complete description of this topic, see <a href="https://developer.mozilla.org/en-US/docs/MDN/Structures/Code_examples" rel="nofollow">the MDN code examples page</a>.</p> 
              <p>Also make sure that your code examples follow the <a href="https://developer.mozilla.org/en-US/docs/MDN/Guidelines/Code_guidelines" rel="nofollow">MDN code example guidelines</a>.</p>

                <svg class="octicon octicon-link" viewbox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true">
                 <path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path>
                </svg></a>Archiving and unarchiving pages</h3> 
              <p>Pages with the banner "This is an archived page. It's not actively maintained." at the top are archived pages  they are archived because either they are obsolete or out-of-date, or they are not in scope for MDN contributors to actively work on.</p> 
              <p>If you find a page that is marked archived and you think it shouldn't be, you can follow the below steps to unarchive it:</p> 
              <ul> 
               <li>Submit an issue to check why the page is archived and if it should be unarchived.</li> 
               <li>Find the source code location of the archived page. Archived pages live in the <a href="https://github.com/mdn/archived-content/">Archived content</a> repo, the structure of which is the same as this repo (except that there are also locale directories present other than <code>en-US</code>). As an example: 
                <ul> 
                 <li>Take the archived <a href="https://developer.mozilla.org/en-US/docs/Mozilla/Mobile/Mobile_developer_challenge" rel="nofollow">Mobile developer challenge</a> page, which is at the URL <code>https://developer.mozilla.org/en-US/docs/Mozilla/Mobile/Mobile_developer_challenge</code>.</li> 
                 <li>This page's source code is at <a href="https://github.com/mdn/archived-content/blob/main/files/en-us/mozilla/mobile/mobile_developer_challenge/index.html">https://github.com/mdn/archived-content/blob/main/files/en-us/mozilla/mobile/mobile_developer_challenge/index.html</a>.</li> 
                </ul> </li> 
               <li>To unarchive the page, you need to open a pull request to move the page's directory into the appropriate place inside this repo, and update the page's meta data (like <code>slug</code>) if needed.</li> 
              </ul> 
              <blockquote> 
               <p><strong>Note:</strong> To archive a currently-non-archived page, you need to follow the above procedure, but move the page in question the opposite way between repos.</p> 
              </blockquote> 
              <p><strong>Note:</strong> You can have an archived page with non-archived pages below it in the hierarchy, and vice-versa. To achieve this you need to have the same directory structure in both the <code>content</code> and <code>archive-content</code> repos. Whether a page is non-archived or archived depends on which repo its <code>index.html</code> file is in. As an example, compare the <a href="https://github.com/mdn/archived-content/tree/main/files/en-us/mozilla">archived-content mozilla directory</a> with the <a href="https://github.com/mdn/content/tree/main/files/en-us/mozilla">content mozilla directory</a>. Both locations represent the same directory structure, but the <code>index.html</code> file appears in the latter, therefore the page is not archived.</p>

              <p>KumaScript macros are still used on MDN pages, even in the new platform. These are the function names surrounded by handlebars-style double curly braces that you'll see in the source code on occasion, for example <code>{{domxref}}</code> Eventually we have to replace them with something else, but they are here for now. They live in <a href="https://github.com/mdn/yari/tree/main/kumascript/macros">https://github.com/mdn/yari/tree/main/kumascript/macros</a>.</p> 
              <p>Sometimes you'll want to make a change to the content that relies on a macro change. Take <a href="https://github.com/mdn/content/pull/187">https://github.com/mdn/content/pull/187</a>. Florian wanted to add documentation for a new WebGL extension to MDN, but this change relied on the new feature's spec being added to <a href="https://github.com/mdn/yari/blob/main/kumascript/macros/SpecData.json">https://github.com/mdn/yari/blob/main/kumascript/macros/SpecData.json</a>. If not, the specification table on the new page would not render properly because the data it relies on would not be there.</p> 
              <p>In such situations:</p> 
              <ol> 
               <li>Make the required PR to <a href="https://github.com/mdn/yari/tree/main/kumascript/macros">https://github.com/mdn/yari/tree/main/kumascript/macros</a> first, and get that merged.</li> 
               <li>Add the content to this repo.</li> 
              </ol>

              <h3><a id="user-content-when-will-my-change-show-up-on-the-production-mdn-site" class="anchor" aria-hidden="true" href="#when-will-my-change-show-up-on-the-production-mdn-site">

                </svg></a>When will my change show up on the production MDN site?</h3> 
              <p>After your pull request is merged, it may take up to 48 hours before the change goes live on the production <a href="https://developer.mozilla.org/" rel="nofollow">https://developer.mozilla.org/</a> site, but it will usually be quicker.</p> 
              <ul> 
               <li>The complete build runs every 24h at around 7PM US/Eastern time, and is then deployed.</li> 
               <li>Once deployment has occurred, it can take up to 24 hours for the associated CDN caches to replace their previously-cached content with the updates.</li> 
              </ul> 
              <p>You can use <a href="https://whatsdeployed.io/s/DLi/mdn/content" rel="nofollow">https://whatsdeployed.io/s/DLi/mdn/content</a> to see if your change has been deployed to the production site. And use <a href="https://whatsdeployed.io/s/16d/mdn/translated-content" rel="nofollow">https://whatsdeployed.io/s/16d/mdn/translated-content</a> for changes to the <a href="https://github.com/mdn/translated-content">https://github.com/mdn/translated-content</a> repo.</p>

              <p>In general, we do not approve of copying content from other sources and putting it on MDN. MDN should be made up of original content wherever possible. If we receive a pull request and discover that it contains plagiarised content, we will close it and request that the submitter resubmit the change with the content rewritten into their own words.</p> 
              <p>If someone wants to donate an article to MDN that they previously published on their blog or it makes sense to copy a complex reference sheet to MDN there may be justification for republishing it. In these cases you should discuss your plan with the MDN team first:</p> 
              <ol> 
               <li>Include a comment on the relevant repo issue that explains your intention  say what you would like to copy (include the URL) and explain why you think this is appropriate.</li> 
               <li>If the content is published under a closed license: 
                <ul> 
                 <li>If you hold the rights to the content, state this, and say that you are happy to republish it on MDN.</li> 
                 <li>If you do not hold the rights to the content, include the author/publisher on the issue if possible, or include details of how they could be contacted so we can ask them for permission to republish the content.</li> 
                </ul> </li> 
               <li>If the content is published under an open license, say what it is, and link to the license so we can check whether it is compatible with <a href="https://developer.mozilla.org/en-US/docs/MDN/About#using_mdn_web_docs_content" rel="nofollow">MDN's license</a>.</li> 
               <li>@-mention chrisdavidmills on the issue, so he can review the request.</li> 
              </ol> 
              <p>Be warned that unless there is a good reason to republish the content, we will probably say "no". The MDN writing team's decision is final.</p>
