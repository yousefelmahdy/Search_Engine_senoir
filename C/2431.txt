       <h2 id="package-deletion-and-restoration-support-on-github"><a href="#package-deletion-and-restoration-support-on-github">Package deletion and restoration support on GitHub</a></h2> 
       <p>On GitHub if you have the required access, you can delete:</p> 
       <ul> 
        <li>an entire private package</li> 
        <li>an entire public package, if there's not more than 25 downloads of any version of the package</li> 
        <li>a specific version of a private package</li> 
        <li>a specific version of a public package, if the package version doesn't have more than 25 downloads</li> 
       </ul> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> </p> 
        <ul> 
         <li>You cannot delete a public package if any version of the package has more than 25 downloads. In this scenario, contact <a href="https://support.github.com/contact">GitHub support</a> for further assistance.</li> 
         <li>When deleting public packages, be aware that you may break projects that depend on your package.</li> 
        </ul> 
       </div> 
       <p>On GitHub, you can also restore an entire package or package version, if:</p> 
       <ul> 
        <li>You restore the package within 30 days of its deletion.</li> 
        <li>The same package namespace is still available and not used for a new package.</li> 
       </ul> 
       <h2 id="packages-api-support"><a href="#packages-api-support">Packages API support</a></h2> 
       <p>You can use the REST API to manage your packages. For more information, see the "<a href="/en/rest/reference/packages">GitHub Packages API</a>."</p> 
       <p>For packages that inherit their permissions and access from repositories, you can use GraphQL to delete a specific package version. The GitHub Packages GraphQL API does not support containers or Docker images that use the package namespace <code>https://ghcr.io/OWNER/PACKAGE-NAME</code>. For more information about GraphQL support, see "<a href="#deleting-a-version-of-a-repository-scoped-package-with-graphql">Deleting a version of a repository-scoped package with GraphQL</a>."</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> The Container registry is currently in public beta and subject to change. During the beta, storage and bandwidth are free. To use the Container registry, you must enable the feature preview. For more information, see "<a href="/en/packages/learn-github-packages/introduction-to-github-packages">Introduction to GitHub Packages</a>" and "<a href="/en/packages/working-with-a-github-packages-registry/enabling-improved-container-support-with-the-container-registry">Enabling improved container support with the Container registry</a>."</p> 
       </div> 
       <h2 id="required-permissions-to-delete-or-restore-a-package"><a href="#required-permissions-to-delete-or-restore-a-package">Required permissions to delete or restore a package</a></h2> 
       <p>For packages that inherit their access permissions from repositories, you can delete a package if you have admin permissions to the repository.</p> 
       <p>Repository-scoped packages on GitHub Packages include these packages:</p> 
       <ul> 
        <li>npm</li> 
        <li>RubyGems</li> 
        <li>maven</li> 
        <li>Gradle</li> 
        <li>NuGet</li> 
        <li>Docker images at <code>docker.pkg.github.com/OWNER/REPOSITORY/IMAGE-NAME</code></li> 
       </ul> 
       <p>To delete a package that has granular permissions separate from a repository, such as containers or Docker images stored at <code>https://ghcr.io/OWNER/PACKAGE-NAME</code>, you must have admin access to the package.</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> The Container registry is currently in public beta and subject to change. During the beta, storage and bandwidth are free. To use the Container registry, you must enable the feature preview. For more information, see "<a href="/en/packages/learn-github-packages/introduction-to-github-packages">Introduction to GitHub Packages</a>" and "<a href="/en/packages/working-with-a-github-packages-registry/enabling-improved-container-support-with-the-container-registry">Enabling improved container support with the Container registry</a>."</p> 
       </div> 
       <h2 id="automate-package-version-deletion-with-github-actions"><a href="#automate-package-version-deletion-with-github-actions">Automate package version deletion with GitHub Actions</a></h2> 
       <p>You can automate package version deletion using an official action created by GitHub. This action is available in the actions repository or on GitHub Marketplace and only works with repository-scoped packages. For more information, see the "Delete Package Versions" action on <a href="https://github.com/marketplace/actions/delete-package-versions">GitHub Marketplace</a> or in the <a href="https://github.com/actions/delete-package-versions">actions repository</a>.</p>

       <h3 id="deleting-a-version-of-a-repository-scoped-package-with-graphql"><a href="#deleting-a-version-of-a-repository-scoped-package-with-graphql">Deleting a version of a repository-scoped package with GraphQL</a></h3> 
       <p>For packages that inherit their permissions and access from repositories, you can use the GraphQL to delete a specific package version.</p> 
       <p>GraphQL is not supported for containers or Docker images at <code>ghcr.io</code>.</p> 
       <p>Use the <code>deletePackageVersion</code> mutation in the GraphQL API. You must use a token with the <code>read:packages</code>, <code>delete:packages</code>, and <code>repo</code> scopes. For more information about tokens, see "<a href="/en/packages/publishing-and-managing-packages/about-github-packages#authenticating-to-github-packages">About GitHub Packages</a>."</p> 
       <p>The following example demonstrates how to delete a package version, using a <code>packageVersionId</code> of <code>MDIyOlJlZ2lzdHJ5UGFja2FnZVZlcnNpb243MTExNg</code>.</p> 
       <pre><code class="hljs language-shell">curl -X POST \
-H "Accept: application/vnd.github.package-deletes-preview+json" \
-H "Authorization: bearer TOKEN" \
-d '{"query":"mutation { deletePackageVersion(input:{packageVersionId:\"MDIyOlJlZ2lzdHJ5UGFja2FnZVZlcnNpb243MTExNg==\"}) { success }}"}' \
HOSTNAME/graphql</code></pre> 
       <p>To find all of the private packages you have published to GitHub Packages, along with the version IDs for the packages, you can use the <code>packages</code> connection through the <code>repository</code> object. You will need a token with the <code>read:packages</code> and <code>repo</code> scopes. For more information, see the <a href="/en/graphql/reference/objects#repository"><code>packages</code></a> connection or the <a href="/en/graphql/reference/interfaces#packageowner"><code>PackageOwner</code></a> interface.</p> 
       <p>For more information about the <code>deletePackageVersion</code> mutation, see "<a href="/en/graphql/reference/mutations#deletepackageversion"><code>deletePackageVersion</code></a>."</p> 
       <p>You cannot directly delete an entire package using GraphQL, but if you delete every version of a package, the package will no longer show on GitHub.</p>

       <h2 id="restoring-packages"><a href="#restoring-packages">Restoring packages</a></h2> 
       <p>You can restore a deleted package or version if:</p> 
       <ul> 
        <li>You restore the package within 30 days of its deletion.</li> 
        <li>The same package namespace and version is still available and not reused for a new package.</li> 
       </ul> 
       <p>For example, if you have a deleted rubygem package named <code>octo-package</code> that was scoped to the repo <code>octo-repo-owner/octo-repo</code>, then you can only restore the package if the package namespace <code>rubygem.pkg.github.com/octo-repo-owner/octo-repo/octo-package</code> is still available, and 30 days have not yet passed.</p> 
       <p>You must also meet one of these permission requirements:</p> 
       <ul> 
        <li>For repository-scoped packages: You have admin permissions to the repository that owns the deleted package.</li> 
        <li>For user-account scoped packages: Your user account owns the deleted package.</li> 
        <li>For organization-scoped packages: You have admin permissions to the deleted package in the organization that owns the package.</li> 
       </ul> 
       <p>For more information, see "<a href="#required-permissions-to-delete-or-restore-a-package">Required permissions</a>."</p> 
       <p>Once the package is restored, the package will use the same namespace it did before. If the same package namespace is not available, you will not be able to restore your package. In this scenario, to restore the deleted package, you must delete the new package that uses the deleted package's namespace first.</p>
