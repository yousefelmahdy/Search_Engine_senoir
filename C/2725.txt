        <h1 class="border-bottom-0">Getting started with the Checks API</h1> 
        <div class="d-none d-lg-block ml-2"> <button class="btn-link Link--secondary js-print tooltipped tooltipped-n" aria-label="Print this article"> <!-- From https://heroicons.dev/ --> 
          <svg fill="none" height="18" width="18" stroke-linecap="round" stroke-linejoin="round" stroke-width="2" viewbox="0 0 24 24" stroke="currentColor">
           <path d="M17 17h2a2 2 0 002-2v-4a2 2 0 00-2-2H5a2 2 0 00-2 2v4a2 2 0 002 2h2m2 4h6a2 2 0 002-2v-4a2 2 0 00-2-2H9a2 2 0 00-2 2v4a2 2 0 002 2zm8-12V5a2 2 0 00-2-2H9a2 2 0 00-2 2v4h10z"></path>
          </svg> </button> 
        </div> 
       </div> 
       <div class="lead-mktg">
        <p>The Check Runs API enables you to build GitHub Apps that run powerful checks against code changes in a repository. You can create apps that perform continuous integration, code linting, or code scanning services and provide detailed feedback on commits.</p>
       </div>

       <div class="article-grid-toc-content"> 
        <h2 id="in-this-article" class="f5 mb-2"><a class="Link--primary" href="#in-this-article">In this article</a></h2>

         <li class="ml-0  mb-2 lh-condensed"><a href="#overview">Overview</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#about-check-suites">About check suites</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#about-check-runs">About check runs</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#check-runs-and-requested-actions">Check runs and requested actions</a></li> 
        </ul>

      <div id="article-contents" class="article-grid-body"> 
       <h3 id="overview"><a href="#overview">Overview</a></h3> 
       <p>Rather than binary pass/fail build statuses, GitHub Apps can report rich statuses, annotate lines of code with detailed information, and re-run tests. The Checks API functionality is available exclusively to your GitHub Apps.</p> 
       <p>For an example of how to use the Checks API with a GitHub App, see "<a href="/en/apps/quickstart-guides/creating-ci-tests-with-the-checks-api">Creating CI tests with the Checks API</a>."</p> 
       <h3 id="about-check-suites"><a href="#about-check-suites">About check suites</a></h3> 
       <p>When someone pushes code to a repository, GitHub creates a check suite for the last commit. A check suite is a collection of the <a href="/en/rest/reference/checks#check-runs">check runs</a> created by a single GitHub App for a specific commit. Check suites summarize the status and conclusion of the check runs that a suite includes.</p> 
       <p><img src="/assets/images/check_suites.png" alt="Check suites workflow"></p> 
       <p>The check suite reports the highest priority check run <code>conclusion</code> in the check suite's <code>conclusion</code>. For example, if three check runs have conclusions of <code>timed_out</code>, <code>success</code>, and <code>neutral</code> the check suite conclusion will be <code>timed_out</code>.</p> 
       <p>By default, GitHub creates a check suite automatically when code is pushed to the repository. This default flow sends the <code>check_suite</code> event (with <code>requested</code> action) to all GitHub App's that have the <code>checks:write</code> permission. When your GitHub App receives the <code>check_suite</code> event, it can create new check runs for the latest commit. GitHub automatically adds new check runs to the correct <a href="/en/rest/reference/checks#check-suites">check suite</a> based on the check run's repository and SHA.</p> 
       <p>If you don't want to use the default automatic flow, you can control when you create check suites. To change the default settings for the creation of check suites, use the <a href="/en/rest/reference/checks#update-repository-preferences-for-check-suites">Update repository preferences for check suites</a> endpoint. All changes to the automatic flow settings are recorded in the audit log for the repository. If you have disabled the automatic flow, you can create a check suite using the <a href="/en/rest/reference/checks#create-a-check-suite">Create a check suite</a> endpoint. You should continue to use the <a href="/en/rest/reference/checks#create-a-check-run">Create a check run</a> endpoint to provide feedback on a commit.</p> 
       <p>Write permission for the Checks API is only available to GitHub Apps. OAuth Apps and authenticated users can view check runs and check suites, but they are not able to create them. If you aren't building a GitHub App, you might be interested in the <a href="/en/rest/reference/repos#statuses">Statuses API</a>.</p> 
       <p>To use the check suites API, the GitHub App must have the <code>checks:write</code> permission and can also subscribe to the <a href="/en/webhooks/event-payloads/#check_suite">check_suite</a> webhook.</p> 
       <p>For information on how to authenticate as a GitHub App, see "<a href="/en/apps/building-github-apps/authentication-options-for-github-apps#authenticating-as-a-github-app">Authentication Options for GitHub Apps</a>."</p> 
       <h3 id="about-check-runs"><a href="#about-check-runs">About check runs</a></h3> 
       <p>A check run is an individual test that is part of a check suite. Each run includes a status and conclusion.</p> 
       <p><img src="/assets/images/check_runs.png" alt="Check runs workflow"></p> 
       <p>If a check run is in a incomplete state for more than 14 days, then the check run's <code>conclusion</code> becomes <code>stale</code> and appears on GitHub as stale with 
        <svg version="1.1" width="16" height="16" viewbox="0 0 16 16" class="octicon octicon-issue-reopened" aria-label="The issue-reopened icon" role="img">
         <path fill-rule="evenodd" d="M8 1.5a6.491 6.491 0 00-5.285 2.715l1.358 1.358A.25.25 0 013.896 6H.25A.25.25 0 010 5.75V2.104a.25.25 0 01.427-.177l1.216 1.216a8 8 0 0114.315 4.03.748.748 0 01-.668.83.75.75 0 01-.824-.676A6.501 6.501 0 008 1.5zM.712 8.004a.75.75 0 01.822.67 6.501 6.501 0 0011.751 3.111l-1.358-1.358a.25.25 0 01.177-.427h3.646a.25.25 0 01.25.25v3.646a.25.25 0 01-.427.177l-1.216-1.216A8 8 0 01.042 8.827a.75.75 0 01.67-.823zM9 11a1 1 0 11-2 0 1 1 0 012 0zm-.25-6.25a.75.75 0 00-1.5 0v3.5a.75.75 0 001.5 0v-3.5z"></path>
        </svg>. Only GitHub can mark check runs as <code>stale</code>. For more information about possible conclusions of a check run, see the <a href="/en/rest/reference/checks#create-a-check-run--parameters"><code>conclusion</code> parameter</a>.</p> 
       <p>As soon as you receive the <a href="/en/webhooks/event-payloads/#check_suite"><code>check_suite</code></a> webhook, you can create the check run, even if the check is not complete. You can update the <code>status</code> of the check run as it completes with the values <code>queued</code>, <code>in_progress</code>, or <code>completed</code>, and you can update the <code>output</code> as more details become available. A check run can contain timestamps, a link to more details on your external site, detailed annotations for specific lines of code, and information about the analysis performed.</p> 
       <p><img src="/assets/images/check_run_annotations.png" alt="Check run annotation"></p> 
       <p>A check can also be manually re-run in the GitHub UI. See "<a href="/en/articles/about-status-checks#checks">About status checks</a>" for more details. When this occurs, the GitHub App that created the check run will receive the <a href="/en/webhooks/event-payloads/#check_run"><code>check_run</code></a> webhook requesting a new check run. If you create a check run without creating a check suite, GitHub creates the check suite for you automatically.</p> 
       <p>Write permission for the Checks API is only available to GitHub Apps. OAuth Apps and authenticated users can view check runs and check suites, but they are not able to create them. If you aren't building a GitHub App, you might be interested in the <a href="/en/rest/reference/repos#statuses">Statuses API</a>.</p> 
       <p>To use the Check Runs API, the GitHub App must have the <code>checks:write</code> permission and can also subscribe to the <a href="/en/webhooks/event-payloads#check_run">check_run</a> webhook.</p> 
       <h3 id="check-runs-and-requested-actions"><a href="#check-runs-and-requested-actions">Check runs and requested actions</a></h3> 
       <p>When you set up a check run with requested actions (not to be confused with GitHub Actions), you can display a button in the pull request view on GitHub that allows people to request your GitHub App to perform additional tasks.</p> 
       <p>For example, a code linting app could use requested actions to display a button in a pull request to automatically fix detected syntax errors.</p> 
       <p>To create a button that can request additional actions from your app, use the <a href="/en/rest/reference/checks#create-a-check-run--parameters"><code>actions</code> object</a> when you <a href="/en/rest/reference/checks/#create-a-check-run">Create a check run</a>. For example, the <code>actions</code> object below displays a button in a pull request with the label "Fix this." The button appears after the check run completes.</p> 
       <pre><code class="hljs language-json"><span class="hljs-string">"actions"</span>: [{
   <span class="hljs-attr">"label"</span>: <span class="hljs-string">"Fix this"</span>,
   <span class="hljs-attr">"description"</span>: <span class="hljs-string">"Let us fix that for you"</span>,
   <span class="hljs-attr">"identifier"</span>: <span class="hljs-string">"fix_errors"</span>
 }]
</code></pre> 
       <p> <img src="/assets/images/github-apps/github_apps_checks_fix_this_button.png" alt="Check run requested action button"></p> 
       <p>When a user clicks the button, GitHub sends the <a href="/en/webhooks/event-payloads/#check_run"><code>check_run.requested_action</code> webhook</a> to your app. When your app receives a <code>check_run.requested_action</code> webhook event, it can look for the <code>requested_action.identifier</code> key in the webhook payload to determine which button was clicked and perform the requested task.</p> 
       <p>For a detailed example of how to set up requested actions with the Checks API, see "<a href="/en/apps/quickstart-guides/creating-ci-tests-with-the-checks-api/#part-2-creating-the-octo-rubocop-ci-test">Creating CI tests with the Checks API</a>."</p>
