         <h1 id="subgroups">Subgroups<a href="#subgroups" title="Permalink" class="anchor"></a> </h1> 
         <div class="introduced-in"> 
          <p><a href="https://gitlab.com/gitlab-org/gitlab-foss/-/issues/2772">Introduced</a> in GitLab 9.0. </p>
         </div> 
         <p>GitLab supports up to 20 levels of subgroups, also known as nested groups or hierarchical groups. </p>
         <p>By using subgroups you can do the following: </p>
         <ul> 
          <li> <strong>Separate internal / external organizations.</strong> Since every group can have its own visibility level (<a href="../../../development/permissions.html#general-permissions">public, internal, or private</a>), youre able to host groups for different purposes under the same umbrella. </li>
          <li> <strong>Organize large projects.</strong> For large projects, subgroups makes it potentially easier to separate permissions on parts of the source code. </li>
          <li> <strong>Make it easier to manage people and control visibility.</strong> Give people different <a href="../../permissions.html#group-members-permissions">permissions</a> depending on their group <a href="#membership">membership</a>. </li>
         </ul> 
         <p>For more information on allowed permissions in groups and projects, see <a href="../../../development/permissions.html#general-permissions">visibility levels</a>. </p>
         <h2 id="overview">Overview<a href="#overview" title="Permalink" class="anchor"></a> </h2> 
         <p>A group can have many subgroups inside it, and at the same time a group can have only one immediate parent group. It resembles a directory behavior or a nested items list: </p>
         <ul> 
          <li>Group 1 
           <ul> 
            <li>Group 1.1 </li>
            <li>Group 1.2 
             <ul> 
              <li>Group 1.2.1 </li>
              <li>Group 1.2.2 
               <ul> 
                <li>Group 1.2.2.1 </li>
               </ul> </li>
             </ul> </li>
           </ul> </li>
         </ul> 
         <p>In a real world example, imagine maintaining a GNU/Linux distribution with the first group being the name of the distribution, and subsequent groups split as follows: </p>
         <ul> 
          <li>Organization Group - GNU/Linux distro 
           <ul> 
            <li>Category Subgroup - Packages 
             <ul> 
              <li>(project) Package01 </li>
              <li>(project) Package02 </li>
             </ul> </li>
            <li>Category Subgroup - Software 
             <ul> 
              <li>(project) Core </li>
              <li>(project) CLI </li>
              <li>(project) Android app </li>
              <li>(project) iOS app </li>
             </ul> </li>
            <li>Category Subgroup - Infra tools 
             <ul> 
              <li>(project) Ansible playbooks </li>
             </ul> </li>
           </ul> </li>
         </ul> 
         <p>Another example of GitLab as a company would be the following: </p>
         <ul> 
          <li>Organization Group - GitLab 
           <ul> 
            <li>Category Subgroup - Marketing 
             <ul> 
              <li>(project) Design </li>
              <li>(project) General </li>
             </ul> </li>
            <li>Category Subgroup - Software 
             <ul> 
              <li>(project) GitLab CE </li>
              <li>(project) GitLab EE </li>
              <li>(project) Omnibus GitLab </li>
              <li>(project) GitLab Runner </li>
              <li>(project) GitLab Pages daemon </li>
             </ul> </li>
            <li>Category Subgroup - Infra tools 
             <ul> 
              <li>(project) Chef cookbooks </li>
             </ul> </li>
            <li>Category Subgroup - Executive team </li>
           </ul> </li>
         </ul> 
         <hr> 
         <p>When performing actions such as transferring or importing a project between subgroups, the behavior is the same as when performing these actions at the <code class="highlighter-rouge">group/project</code> level. </p>
         <h2 id="creating-a-subgroup">Creating a subgroup<a href="#creating-a-subgroup" title="Permalink" class="anchor"></a> </h2> 
         <p>To create a subgroup you must either be an Owner or a Maintainer of the group, depending on the groups setting. </p>
         <p>By default, groups created in: </p>
         <ul> 
          <li>GitLab 12.2 or later allow both Owners and Maintainers to create subgroups. </li>
          <li>GitLab 12.1 or earlier only allow Owners to create subgroups. </li>
         </ul> 
         <p>The setting can be changed for any group by: </p>
         <ul> 
          <li>A group owner. Select the group, and navigate to <strong>Settings &gt; General &gt; Permissions, LFS, 2FA</strong>. </li>
          <li>An administrator. Navigate to <strong>Admin Area &gt; Overview &gt; Groups</strong>, select the group, and choose <strong>Edit</strong>. </li>
         </ul> 
         <p>For more information check the <a href="../../permissions.html#group-members-permissions">permissions table</a>. For a list of words that are not allowed to be used as group names see the <a href="../../reserved_names.html">reserved names</a>. </p>
         <p>Users can always create subgroups if they are explicitly added as an Owner (or Maintainer, if that setting is enabled) to an immediate parent group, even if group creation is disabled by an administrator in their settings. </p>
         <p>To create a subgroup: </p>
         <ol> 
          <li> <p>In the groups dashboard click the <strong>New subgroup</strong> button. </p><p><a href="img/create_subgroup_button_v13_6.png" target="_blank" rel="noopener noreferrer"><img src="img/create_subgroup_button_v13_6.png" alt="Subgroups page"></a> </p></li>
          <li> <p>Create a new group like you would normally do. Notice that the immediate parent group namespace is fixed under <strong>Group path</strong>. The visibility level can differ from the immediate parent group. </p><p><a href="img/create_new_group.png" target="_blank" rel="noopener noreferrer"><img src="img/create_new_group.png" alt="Subgroups page"></a> </p></li>
          <li> <p>Click the <strong>Create group</strong> button to be redirected to the new groups dashboard page. </p></li>
         </ol> 
         <p>Follow the same process to create any subsequent groups. </p>
         <h2 id="membership">Membership<a href="#membership" title="Permalink" class="anchor"></a> </h2> 
         <p>When you add a member to a group, that member is also added to all subgroups. Permission level is inherited from the groups parent. This model allows access to subgroups if you have membership in one of its parents. </p>
         <p>Jobs for pipelines in subgroups can use <a href="../../../ci/runners/README.html">runners</a> registered to the parent group(s). This means secrets configured for the parent group are available to subgroup jobs. </p>
         <p>In addition, maintainers of projects that belong to subgroups can see the details of runners registered to parent group(s). </p>
         <p>The group permissions for a member can be changed only by Owners, and only on the <strong>Members</strong> page of the group the member was added. </p>
         <p>You can tell if a member has inherited the permissions from a parent group by looking at the groups <strong>Members</strong> page. </p>
         <p><a href="img/group_members_13_7.png" target="_blank" rel="noopener noreferrer"><img src="img/group_members_13_7.png" alt="Group members page"></a> </p>
         <p>From the image above, we can deduce the following things: </p>
         <ul> 
          <li>There are 5 members that have access to the group <code class="highlighter-rouge">four</code>. </li>
          <li>User 0 is a Reporter and has inherited their permissions from group <code class="highlighter-rouge">one</code> which is above the hierarchy of group <code class="highlighter-rouge">four</code>. </li>
          <li>User 1 is a Developer and has inherited their permissions from group <code class="highlighter-rouge">one/two</code> which is above the hierarchy of group <code class="highlighter-rouge">four</code>. </li>
          <li>User 2 is a Developer and has inherited their permissions from group <code class="highlighter-rouge">one/two/three</code> which is above the hierarchy of group <code class="highlighter-rouge">four</code>. </li>
          <li>For User 3 the <strong>Source</strong> column indicates <strong>Direct member</strong>, therefore they belong to group <code class="highlighter-rouge">four</code>, the one were inspecting. </li>
          <li>Administrator is the Owner and member of <strong>all</strong> subgroups and for that reason, as with User 3, the <strong>Source</strong> column indicates <strong>Direct member</strong>. </li>
         </ul> 
         <p>Members can be <a href="../index.html#filter-a-group">filtered by inherited or direct membership</a>. </p>
         <h3 id="overriding-the-ancestor-group-membership">Overriding the ancestor group membership<a href="#overriding-the-ancestor-group-membership" title="Permalink" class="anchor"></a> </h3> 
         <div class="admonition-wrapper note">
          <div class="admonition alert alert-info"> 
           <svg role="img" aria-label="note" class="gl-icon ml-1 mr-1 s16">
            <use href="/13.10/assets/images/icons.svg#information-o" />
            <title>note</title>
           </svg>You must be an Owner of a group to be able to add members to it.
          </div>
         </div> 
         <div class="admonition-wrapper note">
          <div class="admonition alert alert-info"> 
           <svg role="img" aria-label="note" class="gl-icon ml-1 mr-1 s16">
            <use href="/13.10/assets/images/icons.svg#information-o" />
            <title>note</title>
           </svg>A users permissions in a subgroup cannot be lower than in any of its ancestor groups. Therefore, you cannot reduce a users permissions in a subgroup with respect to its ancestor groups.
          </div>
         </div> 
         <p>To override a users membership of an ancestor group (the first group they were added to), add the user to the new subgroup again with a higher set of permissions. </p>
         <p>For example, if User 1 was first added to group <code class="highlighter-rouge">one/two</code> with Developer permissions, then they inherit those permissions in every other subgroup of <code class="highlighter-rouge">one/two</code>. To give them Maintainer access to group <code class="highlighter-rouge">one/two/three/four</code>, you would add them again in that group as Maintainer. Removing them from that group, the permissions fall back to those of the ancestor group. </p>
         <h2 id="mentioning-subgroups">Mentioning subgroups<a href="#mentioning-subgroups" title="Permalink" class="anchor"></a> </h2> 
         <p>Mentioning groups (<code class="highlighter-rouge">@group</code>) in issues, commits and merge requests, would notify all members of that group. Now with subgroups, there is more granular support if you want to split your groups structure. Mentioning works as before and you can choose the group of people to be notified. </p>
         <p><a href="img/mention_subgroups.png" target="_blank" rel="noopener noreferrer"><img src="img/mention_subgroups.png" alt="Mentioning subgroups"></a> </p>
         <h2 id="limitations">Limitations<a href="#limitations" title="Permalink" class="anchor"></a> </h2> 
         <p>Heres a list of what you cant do with subgroups: </p>
         <ul> 
          <li> <a href="../../project/pages/index.html">GitLab Pages</a> supports projects hosted under a subgroup, but not subgroup websites. That means that only the highest-level group supports <a href="../../project/pages/getting_started_part_one.html#gitlab-pages-default-domain-names">group websites</a>, although you can have project websites under a subgroup. </li>
          <li>It is not possible to share a project with a group thats an ancestor of the group the project is in. That means you can only share as you walk down the hierarchy. For example, <code class="highlighter-rouge">group/subgroup01/project</code> <strong>cannot</strong> be shared with <code class="highlighter-rouge">group</code>, but can be shared with <code class="highlighter-rouge">group/subgroup02</code> or <code class="highlighter-rouge">group/subgroup01/subgroup03</code>. </li>
         </ul> 
        </div>
