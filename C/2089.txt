       <h3 id="preparing-to-upgrade"><a href="#preparing-to-upgrade">Preparing to upgrade</a></h3> 
       <ol> 
        <li> <p>Determine an upgrade strategy and choose a version to upgrade to. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/upgrade-requirements">Upgrade requirements</a>."</p> </li> 
        <li> <p>Create a fresh backup of your primary instance with the GitHub Enterprise Server Backup Utilities. For more information, see the <a href="https://github.com/github/backup-utils#readme">GitHub Enterprise Server Backup Utilities README.md file</a>.</p> </li> 
        <li> <p>If you are upgrading using an upgrade package, schedule a maintenance window for GitHub Enterprise Server end users. If you are using a hotpatch, maintenance mode is not required.</p> 
         <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
          <p><strong>Note:</strong> The maintenance window depends on the type of upgrade you perform. Upgrades using a hotpatch usually don't require a maintenance window. Sometimes a reboot is required, which you can perform at a later time. Following the versioning scheme of MAJOR.FEATURE.PATCH, patch releases using an upgrade package typically require less than five minutes of downtime. Feature releases that include data migrations take longer depending on storage performance and the amount of data that's migrated. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/enabling-and-scheduling-maintenance-mode">Enabling and scheduling maintenance mode</a>."</p> 
         </div> </li> 
       </ol>

       <h3 id="taking-a-snapshot"><a href="#taking-a-snapshot">Taking a snapshot</a></h3> 
       <p>A snapshot is a checkpoint of a virtual machine (VM) at a point in time. We highly recommend taking a snapshot before upgrading your virtual machine so that if an upgrade fails, you can revert your VM back to the snapshot. If you're upgrading to a new feature release, you must take a VM snapshot. If you're upgrading to a patch release, you can attach the existing data disk.</p> 
       <p>There are two types of snapshots:</p> 
       <ul> 
        <li> <p><strong>VM snapshots</strong> save your entire VM state, including user data and configuration data. This snapshot method requires a large amount of disk space and is time consuming.</p> </li> 
        <li> <p><strong>Data disk snapshots</strong> only save your user data.</p> 
         <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
          <p><strong>Notes:</strong></p> 
          <ul> 
           <li>Some platforms don't allow you to take a snapshot of just your data disk. For these platforms, you'll need to take a snapshot of the entire VM.</li> 
           <li>If your hypervisor does not support full VM snapshots, you should take a snapshot of the root disk and data disk in quick succession.</li> 
          </ul> 
         </div> </li> 
       </ul>

       <h3 id="upgrading-with-a-hotpatch"><a href="#upgrading-with-a-hotpatch">Upgrading with a hotpatch</a></h3> 
       <p>You can upgrade GitHub Enterprise Server to the latest patch release using a hotpatch, which does not require a maintenance window and usually does not require a reboot. You can use hotpatching to upgrade to a newer patch release, but not a feature release. For example, you can upgrade from <code>2.10.1</code> to <code>2.10.5</code> because they are in the same feature series, but not from <code>2.10.9</code> to <code>2.11.0</code> because they are in a different feature series. Using the Management Console, you can install a hotpatch immediately or schedule it for later installation. You can use the administrative shell to install a hotpatch with the <code>ghe-upgrade</code> utility. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/upgrade-requirements">Upgrade requirements</a>."</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Notes</strong>:</p> 
        <ul> 
         <li> <p>If your GitHub Enterprise Server instance is running a release candidate build, you can't upgrade with a hotpatch.</p> </li> 
         <li> <p>Installing a hotpatch using the Management Console is not available in clustered environments. To install a hotpatch in a clustered environment, see "<a href="/en/enterprise-server@3.1/admin/clustering/upgrading-a-cluster#upgrading-with-a-hotpatch">Upgrading a cluster</a>."</p> </li> 
        </ul> 
       </div>

       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> If you've enabled automatic update checks, you don't need to download the upgrade package and can use the file that was automatically downloaded.        </div> 
       <ol>

       <h4 id="upgrading-an-appliance-that-has-replica-instances-using-a-hotpatch"><a href="#upgrading-an-appliance-that-has-replica-instances-using-a-hotpatch">Upgrading an appliance that has replica instances using a hotpatch</a></h4> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note</strong>: If you are installing a hotpatch, you do not need to enter maintenance mode or stop replication.</p> 
       </div> 
       <p>Appliances configured for high-availability and geo-replication use replica instances in addition to primary instances. To upgrade these appliances, you'll need to upgrade both the primary instance and all replica instances, one at a time.</p> 
       <h5 id="upgrading-the-primary-instance"><a href="#upgrading-the-primary-instance">Upgrading the primary instance</a></h5> 
       <ol> 
        <li>Upgrade the primary instance by following the instructions in "<a href="#installing-a-hotpatch-using-the-administrative-shell">Installing a hotpatch using the administrative shell</a>."</li> 
       </ol> 
       <h5 id="upgrading-a-replica-instance"><a href="#upgrading-a-replica-instance">Upgrading a replica instance</a></h5> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> If you're running multiple replica instances as part of geo-replication, repeat this procedure for each replica instance, one at a time.</p> 
       </div> 
       <ol> 
        <li> <p>Upgrade the replica instance by following the instructions in "<a href="#installing-a-hotpatch-using-the-administrative-shell">Installing a hotpatch using the administrative shell</a>." If you are using multiple replicas for Geo-replication, you must repeat this procedure to upgrade each replica one at a time.</p> </li> 
        <li> <p>Connect to the replica instance over SSH as the "admin" user on port 122:</p> <pre><code class="hljs language-shell">$ ssh -p 122 admin@<em>replica-host</em></code></pre> </li> 
        <li> <p>Verify the upgrade by running:</p> <pre><code class="hljs language-shell">$ ghe-version</code></pre> </li> 
       </ol> 
       <h3 id="upgrading-with-an-upgrade-package"><a href="#upgrading-with-an-upgrade-package">Upgrading with an upgrade package</a></h3> 
       <p>While you can use a hotpatch to upgrade to the latest patch release within a feature series, you must use an upgrade package to upgrade to a newer feature release. For example to upgrade from <code>2.11.10</code> to <code>2.12.4</code> you must use an upgrade package since these are in different feature series. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/upgrade-requirements">Upgrade requirements</a>."</p> 
       <h4 id="upgrading-a-single-appliance-with-an-upgrade-package"><a href="#upgrading-a-single-appliance-with-an-upgrade-package">Upgrading a single appliance with an upgrade package</a></h4> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> If you've enabled automatic update checks, you don't need to download the upgrade package and can use the file that was automatically downloaded. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/enabling-automatic-update-checks">Enabling automatic update checks</a>."</p> 
       </div> 
       <ol> 
        <li> <p>SSH into your GitHub Enterprise Server instance. For more information, see "<a href="/en/enterprise-server@3.1/admin/configuration/accessing-the-administrative-shell-ssh">Accessing the administrative shell (SSH)</a>."</p> <pre><code class="hljs language-shell">$ ssh -p 122 admin@<em>HOSTNAME</em></code></pre> </li> 
        <li> <p>Browse to the <a href="https://enterprise.github.com/releases">GitHub Enterprise Server Releases page</a>. Next to the release you are upgrading to, click <strong>Download</strong>, then click the <strong>Upgrading</strong> tab. Select the appropriate platform and copy the URL for the upgrade package (<em>.pkg</em> file).</p> </li> 
        <li> <p>Download the upgrade package to your GitHub Enterprise Server instance using <code>curl </code>:</p> <pre><code class="hljs language-shell">admin@<em>HOSTNAME</em>:~$ curl -L -O <em>UPGRADE-PKG-URL</em></code></pre> </li> 
        <li> <p>Enable maintenance mode and wait for all active processes to complete on the GitHub Enterprise Server instance. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/enabling-and-scheduling-maintenance-mode">Enabling and scheduling maintenance mode</a>."</p> 
         <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
          <p><strong>Note</strong>: When upgrading the primary appliance in a High Availability configuration, the appliance should already be in maintenance mode if you are following the instructions in "<a href="#upgrading-the-primary-instance">Upgrading the primary instance</a>."</p> 
         </div> </li> 
        <li> <p>Run the <code>ghe-upgrade</code> command using the package file name:</p> <pre><code class="hljs language-shell">admin@<em>HOSTNAME</em>:~$ ghe-upgrade <em>GITHUB-UPGRADE.pkg</em>
*** verifying upgrade package signature...</code></pre> </li> 
        <li> <p>Confirm that you'd like to continue with the upgrade and restart after the package signature verifies. The new root filesystem writes to the secondary partition and the instance automatically restarts in maintenance mode:</p> <pre><code class="hljs language-shell">*** applying update...
This package will upgrade your installation to version <em>version-number</em>
Current root partition: /dev/xvda1 [<em>version-number</em>]
Target root partition:  /dev/xvda2
Proceed with installation? [y/N]</code></pre> </li> 
        <li> <p>For single appliance upgrades, disable maintenance mode so users can use your GitHub Enterprise Server instance.</p> 
         <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
          <p><strong>Note</strong>: When upgrading appliances in a High Availability configuration you should remain in maintenance mode until you have upgraded all of the replicas and replication is current. For more information, see "<a href="#upgrading-a-replica-instance">Upgrading a replica instance</a>."</p> 
         </div> </li> 
       </ol> 
       <h4 id="upgrading-an-appliance-that-has-replica-instances-using-an-upgrade-package"><a href="#upgrading-an-appliance-that-has-replica-instances-using-an-upgrade-package">Upgrading an appliance that has replica instances using an upgrade package</a></h4> 
       <p>Appliances configured for high-availability and geo-replication use replica instances in addition to primary instances. To upgrade these appliances, you'll need to upgrade both the primary instance and all replica instances, one at a time.</p> 
       <h5 id="upgrading-the-primary-instance-1"><a href="#upgrading-the-primary-instance-1">Upgrading the primary instance</a></h5> 
       <div class="extended-markdown warning border rounded-1 mb-4 p-3 color-border-danger color-bg-danger f5"> 
        <p><strong>Warning:</strong> When replication is stopped, if the primary fails, any work that is done before the replica is upgraded and the replication begins again will be lost.</p> 
       </div> 
       <ol> 
        <li>On the primary instance, enable maintenance mode and wait for all active processes to complete. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/enabling-and-scheduling-maintenance-mode">Enabling maintenance mode</a>."</li> 
        <li>Connect to the replica instance over SSH as the "admin" user on port 122:<pre><code class="hljs language-shell">$ ssh -p 122 admin@<em>replica-host</em></code></pre></li> 
        <li>On the replica instance, or on all replica instances if you're running multiple replica instances as part of geo-replication, run <code>ghe-repl-stop</code> to stop replication.</li> 
        <li>Upgrade the primary instance by following the instructions in "<a href="#upgrading-a-single-appliance-with-an-upgrade-package">Upgrading a single appliance with an upgrade package</a>."</li> 
       </ol> 
       <h5 id="upgrading-a-replica-instance-1"><a href="#upgrading-a-replica-instance-1">Upgrading a replica instance</a></h5> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> If you're running multiple replica instances as part of geo-replication, repeat this procedure for each replica instance, one at a time.</p> 
       </div> 
       <ol> 
        <li> <p>Upgrade the replica instance by following the instructions in "<a href="#upgrading-a-single-appliance-with-an-upgrade-package">Upgrading a single appliance with an upgrade package</a>." If you are using multiple replicas for Geo-replication, you must repeat this procedure to upgrade each replica one at a time.</p> </li> 
        <li> <p>Connect to the replica instance over SSH as the "admin" user on port 122:</p> <pre><code class="hljs language-shell">$ ssh -p 122 admin@<em>replica-host</em></code></pre> </li> 
        <li> <p>Verify the upgrade by running:</p> <pre><code class="hljs language-shell">$ ghe-version</code></pre> </li> 
        <li> <p>On the replica instance, to start replication, run <code>ghe-repl-start</code>.</p> </li> 
        <li> <p>On the replica instance, to make sure replication services are running correctly, run <code>ghe-repl-status</code>. This command will return <code>OK</code> for all services when a successful replication is in progress and the replica has upgraded. If the command returns <code>Replication is not running</code>, the replication may still be starting. Wait about one minute before running <code>ghe-repl-status</code> again.</p> 
         <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
          <p> <strong>Note:</strong> While the resync is in progress <code>ghe-repl-status</code> may return expected messages indicating that replication is behind. For example: <code>CRITICAL: git replication is behind the primary by more than 1007 repositories and/or gists</code></p> 
         </div> <p>If <code>ghe-repl-status</code> didn't return <code>OK</code>, follow the steps below to manually start the replication.</p> 
         <ol> 
          <li>On the replica instance, run <code>ghe-repl-setup &lt;primary-instance-ip&gt;</code> again.</li> 
          <li>On the replica instance, to start replication, run <code>ghe-repl-start</code>.</li> 
          <li>On the replica instance, to make sure replication services are running correctly, run <code>ghe-repl-status</code>. This command will return <code>OK</code> for all services when a successful replication is in progress and the replica has upgraded.</li> 
         </ol> </li> 
        <li> <p>When you have completed upgrading the last replica, and the resync is complete, disable maintenance mode so users can use your GitHub Enterprise Server instance.</p> </li> 
       </ol> 
       <h3 id="restoring-from-a-failed-upgrade"><a href="#restoring-from-a-failed-upgrade">Restoring from a failed upgrade</a></h3> 
       <p>If an upgrade fails or is interrupted, you should revert your instance back to its previous state. The process for completing this depends on the type of upgrade.</p> 
       <h4 id="rolling-back-a-patch-release"><a href="#rolling-back-a-patch-release">Rolling back a patch release</a></h4> 
       <p>To roll back a patch release, use the <code>ghe-upgrade</code> command with the <code>--allow-patch-rollback</code> switch. When rolling back an upgrade, you must use an upgrade package file with the <em>.pkg</em> extension. Hotpatch package files with the <em>.hpkg</em> extension are not supported.</p> 
       <pre><code class="hljs language-shell">ghe-upgrade --allow-patch-rollback <em>EARLIER-RELEASE-UPGRADE-PACKAGE</em>.pkg</code></pre> 
       <p>A reboot is required after running the command. Rolling back does not affect the data partition, as migrations are not run on patch releases.</p> 
       <p>For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/command-line-utilities/#ghe-upgrade">Command-line utilities</a>."</p> 
       <h4 id="rolling-back-a-feature-release"><a href="#rolling-back-a-feature-release">Rolling back a feature release</a></h4> 
       <p>To roll back from a feature release, restore from a VM snapshot to ensure that root and data partitions are in a consistent state. For more information, see "<a href="#taking-a-snapshot">Taking a snapshot</a>."</p>
