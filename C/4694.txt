      <p><b>Windows Push Notification Service</b> (commonly referred to as <b>Windows Notification Service</b> or <b>WNS</b>) is a <a href="/wiki/Notification_service" title="Notification service">notification service</a> developed by <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> for all devices running <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> platforms. It allows for developers to send <a href="/wiki/Push_technology" title="Push technology">push data</a> ("toast" and "tile" updates) to Windows and <a href="/wiki/Universal_Windows_Platform" title="Universal Windows Platform">Universal Windows Platform</a> applications which implement the feature.<sup id="cite_ref-:0_1-0" class="reference"><a href="#cite_note-:0-1">[1]</a></sup> Designed as a successor to the <a href="/wiki/Microsoft_Push_Notification_Service" title="Microsoft Push Notification Service">Microsoft Push Notification Service</a>, it was first supported on <a href="/wiki/Windows_8" title="Windows 8">Windows 8</a> and subsequently on <a href="/wiki/Windows_Phone_8.1" title="Windows Phone 8.1">Windows Phone 8.1</a> upon its release.<sup id="cite_ref-WNS_Overview_Runtime_2-0" class="reference"><a href="#cite_note-WNS_Overview_Runtime-2">[2]</a></sup> </p>

      <h2><span class="mw-headline" id="Technical_details">Technical details</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Push_Notification_Service&amp;action=edit&amp;section=1" title="Edit section: Technical details">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <h3><span class="mw-headline" id="Design_and_compatibility">Design and compatibility</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Push_Notification_Service&amp;action=edit&amp;section=2" title="Edit section: Design and compatibility">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The Windows Push Notification Service (WNS) was designed as a successor to the Microsoft Push Notification Service (MPNS), which was only supported natively on the <a href="/wiki/Windows_Phone_8" title="Windows Phone 8">Windows Phone 8</a> Operating System. Developers can still use the MPNS on apps that are installed on newer versions of <a href="/wiki/Microsoft_Windows_Mobile" class="mw-redirect" title="Microsoft Windows Mobile">Windows Mobile</a> (<a href="/wiki/Windows_Phone_8" title="Windows Phone 8">Windows Phone 8</a> or <a href="/wiki/Windows_Phone_8.1" title="Windows Phone 8.1">Windows Phone 8.1</a>), but only if the Windows application was already registered to use the MPNS and has been converted to a <a href="/wiki/Microsoft_Silverlight" title="Microsoft Silverlight">Microsoft Silverlight</a> application and modified to re-target the new platform.<sup id="cite_ref-MPNSvsWNS_3-0" class="reference"><a href="#cite_note-MPNSvsWNS-3">[3]</a></sup> </p>
      <p>In 2015, Microsoft announced that the WNS would be expanded to utilize the <a href="/wiki/Universal_Windows_Platform" title="Universal Windows Platform">Universal Windows Platform</a> architecture, allowing for push data to be sent to <a href="/wiki/Windows_10" title="Windows 10">Windows 10</a>, <a href="/wiki/Windows_10_Mobile" title="Windows 10 Mobile">Windows 10 Mobile</a>, <a href="/wiki/Xbox" title="Xbox">Xbox</a>, as well as other supported platforms using universal API calls and POST requests.<sup id="cite_ref-W10FirstLook_4-0" class="reference"><a href="#cite_note-W10FirstLook-4">[4]</a></sup> </p>
      <p>During the <a href="/wiki/Microsoft_Build" class="mw-redirect" title="Microsoft Build">2015 Build</a> keynote, <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> announced a <a href="/wiki/Universal_Windows_Platform" title="Universal Windows Platform">Universal Windows Platform</a> bridge that would allow <a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a> and <a href="/wiki/IOS" title="IOS">iOS</a> software to be ported to <a href="/wiki/Windows_10_Mobile" title="Windows 10 Mobile">Windows 10 Mobile</a> and published to the <a href="/wiki/Windows_Store" class="mw-redirect" title="Windows Store">Windows Store</a>.<sup id="cite_ref-PCWorldWindowsBridge_5-0" class="reference"><a href="#cite_note-PCWorldWindowsBridge-5">[5]</a></sup> In August 2015, A version of the Microsoft Android bridge toolset was reported to be leaked and available on the internet along with its documentation.<sup id="cite_ref-CWLeak_6-0" class="reference"><a href="#cite_note-CWLeak-6">[6]</a></sup> The leaked toolset required developers to register and use the WNS to send notification data to ported applications, and would not allow for <a href="/wiki/Google_Cloud_Messaging" title="Google Cloud Messaging">Google Cloud Messaging</a> to be used instead. Microsoft later discontinued the Android bridge project in favor of continuing support for iOS application porting instead.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> </p>
      <p>During the <a href="/wiki/Microsoft_Build" class="mw-redirect" title="Microsoft Build">2016 Build</a> keynote, Microsoft announced an update to the WNS and the <a href="/wiki/Windows_10" title="Windows 10">Windows 10</a> Operating System that will allow for Android and iOS devices to forward push notifications received to <a href="/wiki/Windows_10" title="Windows 10">Windows 10</a> to be viewed and discarded.<sup id="cite_ref-Computer.it_8-0" class="reference"><a href="#cite_note-Computer.it-8">[8]</a></sup> </p> 
      <h3><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Push_Notification_Service&amp;action=edit&amp;section=3" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The architecture of the Windows Push Notification Service is similar to that of its predecessor, in that it consists of servers and interfaces that generate, maintain, store, and authenticate unique identifiers (called <i>Channel URI Identifiers</i>) for all devices that register to use the service.<sup id="cite_ref-WNS_Overview_Runtime_2-1" class="reference"><a href="#cite_note-WNS_Overview_Runtime-2">[2]</a></sup> When a device enrolls to receive data and notification information using the WNS, it first sends a device registration request to the WNS network. The WNS network acknowledges the request, and responds with the device's unique <i>Channel URI Identifier</i>.<sup id="cite_ref-WNS_Overview_General_9-0" class="reference"><a href="#cite_note-WNS_Overview_General-9">[9]</a></sup> Typically, the device will then send its identifier to a server owned by the developer so that it can be stored and used for sending notifications.<sup id="cite_ref-:0_1-1" class="reference"><a href="#cite_note-:0-1">[1]</a></sup> When the app developer wishes to transmit a notification or other WNS data to the device, it will transmit a POST request to the WNS network.<sup id="cite_ref-Snoei_10-0" class="reference"><a href="#cite_note-Snoei-10">[10]</a></sup> The network will acknowledge and authenticate the request. If the authentication succeeds, the data to be transmitted is enqueued and then sent to the device from the WNS network using the <i>Channel URI Identifier</i>. </p>
