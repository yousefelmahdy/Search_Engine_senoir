       <h3 id="supported-custom-domains"><a href="#supported-custom-domains">Supported custom domains</a></h3> 
       <p>GitHub Pages works with two types of domains: subdomains and apex domains. For a list of unsupported custom domains, see "<a href="/en/articles/troubleshooting-custom-domains-and-github-pages/#custom-domain-names-that-are-unsupported">Troubleshooting custom domains and GitHub Pages</a>."</p>

          <td><code>www.example.com</code></td>

       <p>You can set up either or both of apex and <code>www</code> subdomain configurations for your site. For more information on apex domains, see "<a href="#using-an-apex-domain-for-your-github-pages-site">Using an apex domain for your GitHub Pages site</a>."</p> 
       <p>We recommend always using a <code>www</code> subdomain, even if you also use an apex domain. When you create a new site with an apex domain, we automatically attempt to secure the <code>www</code> subdomain for use when serving your site's content. If you configure a <code>www</code> subdomain, we automatically attempt to secure the associated apex domain. For more information, see "<a href="/en/articles/managing-a-custom-domain-for-your-github-pages-site">Managing a custom domain for your GitHub Pages site</a>."</p> 
       <p>After you configure a custom domain for a user or organization site, the custom domain will replace the <code>&lt;user&gt;.github.io</code> or <code>&lt;organization&gt;.github.io</code> portion of the URL for any project sites owned by the account that do not have a custom domain configured. For example, if the custom domain for your user site is <code>www.octocat.com</code>, and you have a project site with no custom domain configured that is published from a repository called <code>octo-project</code>, the GitHub Pages site for that repository will be available at <code>www.octocat.com/octo-project</code>.</p> 
       <h3 id="using-a-subdomain-for-your-github-pages-site"><a href="#using-a-subdomain-for-your-github-pages-site">Using a subdomain for your GitHub Pages site</a></h3> 
       <p>A subdomain is the part of a URL before the root domain. You can configure your subdomain as <code>www</code> or as a distinct section of your site, like <code>blog.example.com</code>.</p> 
       <p>Subdomains are configured with a <code>CNAME</code> record through your DNS provider. For more information, see "<a href="/en/articles/managing-a-custom-domain-for-your-github-pages-site#configuring-a-subdomain">Managing a custom domain for your GitHub Pages site</a>."</p> 
       <h4 id="www-subdomains"><a href="#www-subdomains"><code>www</code> subdomains</a></h4> 
       <p>A <code>www</code> subdomain is the most commonly used type of subdomain. For example, <code>www.example.com</code> includes a <code>www</code> subdomain.</p> 
       <p><code>www</code> subdomains are the most stable type of custom domain because <code>www</code> subdomains are not affected by changes to the IP addresses of GitHub's servers.</p> 
       <h4 id="custom-subdomains"><a href="#custom-subdomains">Custom subdomains</a></h4> 
       <p>A custom subdomain is a type of subdomain that doesn't use the standard <code>www</code> variant. Custom subdomains are mostly used when you want two distinct sections of your site. For example, you can create a site called <code>blog.example.com</code> and customize that section independently from <code>www.example.com</code>.</p> 
       <h3 id="using-an-apex-domain-for-your-github-pages-site"><a href="#using-an-apex-domain-for-your-github-pages-site">Using an apex domain for your GitHub Pages site</a></h3> 
       <p>An apex domain is a custom domain that does not contain a subdomain, such as <code>example.com</code>. Apex domains are also known as base, bare, naked, root apex, or zone apex domains.</p> 
       <p>An apex domain is configured with an <code>A</code>, <code>ALIAS</code>, or <code>ANAME</code> record through your DNS provider. For more information, see "<a href="/en/articles/managing-a-custom-domain-for-your-github-pages-site#configuring-an-apex-domain">Managing a custom domain for your GitHub Pages site</a>."</p> 
       <p>If you are using an apex domain as your custom domain, we recommend also setting up a <code>www</code> subdomain. If you configure the correct records for each domain type through your DNS provider, GitHub Pages will automatically create redirects between the domains. For example, if you configure <code>www.example.com</code> as the custom domain for your site, and you have GitHub Pages DNS records set up for the apex and <code>www</code> domains, then <code>example.com</code> will redirect to <code>www.example.com</code>. Note that automatic redirects only apply to the <code>www</code> subdomain. Automatic redirects do not apply to any other subdomains, such as <code>blog</code>. For more information, see "<a href="/en/github/working-with-github-pages/managing-a-custom-domain-for-your-github-pages-site/#configuring-a-subdomain">Managing a custom domain for your GitHub Pages site</a>."</p> 
       <h3 id="updating-custom-domains-when-your-github-pages-site-is-disabled"><a href="#updating-custom-domains-when-your-github-pages-site-is-disabled">Updating custom domains when your GitHub Pages site is disabled</a></h3> 
       <p>If your GitHub Pages site is disabled but has a custom domain set up, you should immediately update or remove your DNS records with your DNS provider to avoid the risk of a domain takeover. Having a custom domain configured with your DNS provider while your site is disabled could result in someone else hosting a site on one of your subdomains. For more information, see "<a href="/en/articles/managing-a-custom-domain-for-your-github-pages-site">Managing a custom domain for your GitHub Pages site</a>." </p>
