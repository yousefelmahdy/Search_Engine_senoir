     <h1 id="requirements">Requirements<a href="#requirements" title="Permalink" class="anchor"></a> </h1> 
     <h2 id="operating-systems">Operating Systems<a href="#operating-systems" title="Permalink" class="anchor"></a> </h2> 
     <h3 id="supported-unix-distributions">Supported Unix distributions<a href="#supported-unix-distributions" title="Permalink" class="anchor"></a> </h3> 
     <ul> 
      <li>Ubuntu </li>
      <li>Debian </li>
      <li>CentOS </li>
      <li>openSUSE </li>
      <li>Red Hat Enterprise Linux (please use the CentOS packages and instructions) </li>
      <li>Scientific Linux (please use the CentOS packages and instructions) </li>
      <li>Oracle Linux (please use the CentOS packages and instructions) </li>
     </ul> 
     <p>For the installations options please see <a href="https://about.gitlab.com/installation/">the installation page on the GitLab website</a>. </p>
     <h3 id="unsupported-unix-distributions">Unsupported Unix distributions<a href="#unsupported-unix-distributions" title="Permalink" class="anchor"></a> </h3> 
     <ul> 
      <li>Arch Linux </li>
      <li>Fedora </li>
      <li>FreeBSD </li>
      <li>Gentoo </li>
      <li>macOS </li>
     </ul> 
     <p>On the above unsupported distributions is still possible to install GitLab yourself. Please see the <a href="installation.html">installation from source guide</a> and the <a href="https://about.gitlab.com/installation/">installation guides</a> for more information. </p>
     <h3 id="non-unix-operating-systems-such-as-windows">Non-Unix operating systems such as Windows<a href="#non-unix-operating-systems-such-as-windows" title="Permalink" class="anchor"></a> </h3> 
     <p>GitLab is developed for Unix operating systems. It does <strong>not</strong> run on Windows, and we have no plans to support it in the near future. For the latest development status view this <a href="https://gitlab.com/gitlab-org/gitlab-ce/issues/46567">issue</a>. Please consider using a virtual machine to run GitLab. </p>
     <h2 id="ruby-versions">Ruby versions<a href="#ruby-versions" title="Permalink" class="anchor"></a> </h2> 
     <p>GitLab requires Ruby (MRI) 2.5. Support for Ruby versions below 2.5 (2.3, 2.4) will stop with GitLab 11.6. </p>
     <p>You will have to use the standard MRI implementation of Ruby. We love <a href="https://www.jruby.org/">JRuby</a> and <a href="https://rubinius.com">Rubinius</a> but GitLab needs several Gems that have native extensions. </p>
     <h2 id="hardware-requirements">Hardware requirements<a href="#hardware-requirements" title="Permalink" class="anchor"></a> </h2> 
     <h3 id="storage">Storage<a href="#storage" title="Permalink" class="anchor"></a> </h3> 
     <p>The necessary hard drive space largely depends on the size of the repos you want to store in GitLab but as a <em>rule of thumb</em> you should have at least as much free space as all your repos combined take up. </p>
     <p>If you want to be flexible about growing your hard drive space in the future consider mounting it using LVM so you can add more hard drives when you need them. </p>
     <p>Apart from a local hard drive you can also mount a volume that supports the network file system (NFS) protocol. This volume might be located on a file server, a network attached storage (NAS) device, a storage area network (SAN) or on an Amazon Web Services (AWS) Elastic Block Store (EBS) volume. </p>
     <p>If you have enough RAM memory and a recent CPU the speed of GitLab is mainly limited by hard drive seek times. Having a fast drive (7200 RPM and up) or a solid state drive (SSD) will improve the responsiveness of GitLab. </p>
     <div class="admonition-wrapper note">
      <div class="admonition alert alert-info"> <i class="fa fa-info-circle fa-fw" aria-hidden="true"></i><strong>Note:</strong> Since file system performance may affect GitLab’s overall performance, we do not recommend using EFS for storage. See the <a href="../administration/high_availability/nfs.html#avoid-using-awss-elastic-file-system-efs">relevant documentation</a> for more details.
      </div>
     </div> 
     <h3 id="cpu">CPU<a href="#cpu" title="Permalink" class="anchor"></a> </h3> 
     <ul> 
      <li>1 core supports up to 100 users but the application can be a bit slower due to having all workers and background jobs running on the same core </li>
      <li> <strong>2 cores</strong> is the <strong>recommended</strong> number of cores and supports up to 500 users </li>
      <li>4 cores supports up to 2,000 users </li>
      <li>8 cores supports up to 5,000 users </li>
      <li>16 cores supports up to 10,000 users </li>
      <li>32 cores supports up to 20,000 users </li>
      <li>64 cores supports up to 40,000 users </li>
      <li>More users? Run it on <a href="https://about.gitlab.com/high-availability/">multiple application servers</a> </li>
     </ul> 
     <h3 id="memory">Memory<a href="#memory" title="Permalink" class="anchor"></a> </h3> 
     <p>You need at least 8GB of addressable memory (RAM + swap) to install and use GitLab! The operating system and any other running applications will also be using memory so keep in mind that you need at least 4GB available before running GitLab. With less memory GitLab will give strange errors during the reconfigure run and 500 errors during usage. </p>
     <ul> 
      <li>4GB RAM + 4GB swap supports up to 100 users but it will be very slow </li>
      <li> <strong>8GB RAM</strong> is the <strong>recommended</strong> memory size for all installations and supports up to 100 users </li>
      <li>16GB RAM supports up to 2,000 users </li>
      <li>32GB RAM supports up to 4,000 users </li>
      <li>64GB RAM supports up to 8,000 users </li>
      <li>128GB RAM supports up to 16,000 users </li>
      <li>256GB RAM supports up to 32,000 users </li>
      <li>More users? Run it on <a href="https://about.gitlab.com/high-availability/">multiple application servers</a> </li>
     </ul> 
     <p>We recommend having at least <a href="https://askubuntu.com/a/505344/310789">2GB of swap on your server</a>, even if you currently have enough available RAM. Having swap will help reduce the chance of errors occurring if your available memory changes. We also recommend <a href="https://askubuntu.com/a/103916">configuring the kernel’s swappiness setting</a> to a low value like <code class="highlighter-rouge">10</code> to make the most of your RAM while still having the swap available when needed. </p>
     <div class="admonition-wrapper note">
      <div class="admonition alert alert-info"> <i class="fa fa-info-circle fa-fw" aria-hidden="true"></i><strong>Note:</strong> The 25 workers of Sidekiq will show up as separate processes in your process overview (such as <code class="highlighter-rouge">top</code> or <code class="highlighter-rouge">htop</code>) but they share the same RAM allocation since Sidekiq is a multithreaded application. Please see the section below about Unicorn workers for information about how many you need of those.
      </div>
     </div> 
     <h2 id="database">Database<a href="#database" title="Permalink" class="anchor"></a> </h2> 
     <p>The server running the database should have <em>at least</em> 5-10 GB of storage available, though the exact requirements depend on the size of the GitLab installation (e.g. the number of users, projects, etc). </p>
     <p>We currently support the following databases: </p>
     <ul> 
      <li>PostgreSQL (highly recommended) </li>
      <li>MySQL/MariaDB (strongly discouraged, not all GitLab features are supported, no support for <a href="https://mariadb.com/kb/en/mariadb/gtid/">MySQL/MariaDB GTID</a>) </li>
     </ul> 
     <p>We highly recommend the use of PostgreSQL instead of MySQL/MariaDB as not all features of GitLab work with MySQL/MariaDB: </p>
     <ol> 
      <li>MySQL support for subgroups was <a href="https://about.gitlab.com/2017/06/22/gitlab-9-3-released/#dropping-support-for-subgroups-in-mysql">dropped with GitLab 9.3</a>. See <a href="https://gitlab.com/gitlab-org/gitlab-ce/issues/30472">issue #30472</a> for more information. </li>
      <li>Geo does <a href="/11.11/ee/administration/geo/replication/database.html">not support MySQL</a>. This means no supported Disaster Recovery solution if using MySQL. <span class="badge-trigger premium-only"></span> </li>
      <li> <a href="../update/README.html#upgrading-without-downtime">Zero downtime migrations</a> do not work with MySQL. </li>
      <li> <a href="/11.11/ee/administration/database_load_balancing.html">Database load balancing</a> is supported only for PostgreSQL. <span class="badge-trigger premium-only"></span> </li>
      <li>GitLab <a href="https://gitlab.com/gitlab-org/gitlab-ce/issues/31806">optimizes the loading of dashboard events</a> using <a href="https://blog.heapanalytics.com/postgresqls-powerful-new-join-type-lateral/">PostgreSQL LATERAL JOINs</a>. </li>
      <li>In general, SQL optimized for PostgreSQL may run much slower in MySQL due to differences in query planners. For example, subqueries that work well in PostgreSQL may not be <a href="https://dev.mysql.com/doc/refman/5.7/en/optimizing-subqueries.html">performant in MySQL</a>. </li>
      <li>Binary column index length is limited to 20 bytes. This is accomplished with <a href="https://gitlab.com/gitlab-org/gitlab-ce/blob/master/config/initializers/mysql_set_length_for_binary_indexes.rb">a hack</a>. </li>
      <li>MySQL requires a variety of hacks to increase limits on various columns, <a href="https://gitlab.com/gitlab-org/gitlab-ce/issues/49583">for example</a>. </li>
      <li> <a href="https://gitlab.com/gitlab-org/gitlab-ce/issues/51173#note_99391731">The milestone filter runs slower queries on MySQL</a>. </li>
      <li>We expect this list to grow over time. </li>
     </ol> 
     <p>Existing users using GitLab with MySQL/MariaDB are advised to <a href="../update/mysql_to_postgresql.html">migrate to PostgreSQL</a> instead. </p>
     <h3 id="postgresql-requirements">PostgreSQL Requirements<a href="#postgresql-requirements" title="Permalink" class="anchor"></a> </h3> 
     <p>As of GitLab 10.0, PostgreSQL 9.6 or newer is required, and earlier versions are not supported. We highly recommend users to use PostgreSQL 9.6 as this is the PostgreSQL version used for development and testing. </p>
     <p>Users using PostgreSQL must ensure the <code class="highlighter-rouge">pg_trgm</code> extension is loaded into every GitLab database. This extension can be enabled (using a PostgreSQL super user) by running the following query for every database: </p>
     <div class="highlighter-rouge">
      <div class="highlight">
       <pre class="highlight"><code>CREATE EXTENSION pg_trgm;
</code></pre>
      </div>
     </div> 
     <p>On some systems you may need to install an additional package (e.g. <code class="highlighter-rouge">postgresql-contrib</code>) for this extension to become available. </p>
     <h4 id="additional-requirements-for-gitlab-geo">Additional requirements for GitLab Geo<a href="#additional-requirements-for-gitlab-geo" title="Permalink" class="anchor"></a> </h4> 
     <p>If you are using <a href="/11.11/ee/development/geo.html">GitLab Geo</a>: </p>
     <ul> 
      <li>We strongly recommend running Omnibus-managed instances as they are actively developed and tested. We aim to be compatible with most external (not managed by Omnibus) databases (for example, AWS RDS) but we do not guarantee compatibility. </li>
      <li>The <a href="/11.11/ee/development/geo.html#geo-tracking-database">tracking database</a> requires the <a href="https://www.postgresql.org/docs/9.6/static/postgres-fdw.html">postgres_fdw</a> extension. </li>
     </ul> 
     <div class="highlighter-rouge">
      <div class="highlight">
       <pre class="highlight"><code>CREATE EXTENSION postgres_fdw;
</code></pre>
      </div>
     </div> 
     <h2 id="unicorn-workers">Unicorn Workers<a href="#unicorn-workers" title="Permalink" class="anchor"></a> </h2> 
     <p>It’s possible to increase the amount of unicorn workers and this will usually help to reduce the response time of the applications and increase the ability to handle parallel requests. </p>
     <p>For most instances we recommend using: CPU cores + 1 = unicorn workers. So for a machine with 2 cores, 3 unicorn workers is ideal. </p>
     <p>For all machines that have 2GB and up we recommend a minimum of three unicorn workers. If you have a 1GB machine we recommend to configure only two Unicorn workers to prevent excessive swapping. </p>
     <p>To change the Unicorn workers when you have the Omnibus package (which defaults to the recommendation above) please see <a href="https://gitlab.com/gitlab-org/omnibus-gitlab/blob/master/doc/settings/unicorn.md#unicorn-settings">the Unicorn settings in the Omnibus GitLab documentation</a>. </p>
     <h2 id="redis-and-sidekiq">Redis and Sidekiq<a href="#redis-and-sidekiq" title="Permalink" class="anchor"></a> </h2> 
     <p>Redis stores all user sessions and the background task queue. The storage requirements for Redis are minimal, about 25kB per user. Sidekiq processes the background jobs with a multithreaded process. This process starts with the entire Rails stack (200MB+) but it can grow over time due to memory leaks. On a very active server (10,000 active users) the Sidekiq process can use 1GB+ of memory. </p>
     <h2 id="prometheus-and-its-exporters">Prometheus and its exporters<a href="#prometheus-and-its-exporters" title="Permalink" class="anchor"></a> </h2> 
     <p>As of Omnibus GitLab 9.0, <a href="https://prometheus.io">Prometheus</a> and its related exporters are enabled by default, to enable easy and in depth monitoring of GitLab. Approximately 200MB of memory will be consumed by these processes, with default settings. </p>
     <p>If you would like to disable Prometheus and it’s exporters or read more information about it, check the <a href="../administration/monitoring/prometheus/index.html">Prometheus documentation</a>. </p>
     <h2 id="gitlab-runner">GitLab Runner<a href="#gitlab-runner" title="Permalink" class="anchor"></a> </h2> 
     <p>We strongly advise against installing GitLab Runner on the same machine you plan to install GitLab on. Depending on how you decide to configure GitLab Runner and what tools you use to exercise your application in the CI environment, GitLab Runner can consume significant amount of available memory. </p>
     <p>Memory consumption calculations, that are available above, will not be valid if you decide to run GitLab Runner and the GitLab Rails application on the same machine. </p>
     <p>It is also not safe to install everything on a single machine, because of the <a href="https://gitlab.com/gitlab-org/gitlab-runner/blob/master/docs/security/index.md">security reasons</a> - especially when you plan to use shell executor with GitLab Runner. </p>
     <p>We recommend using a separate machine for each GitLab Runner, if you plan to use the CI features. </p>
     <h2 id="supported-web-browsers">Supported web browsers<a href="#supported-web-browsers" title="Permalink" class="anchor"></a> </h2> 
     <p>We support the current and the previous major release of: </p>
     <ul> 
      <li>Firefox </li>
      <li>Chrome/Chromium </li>
      <li>Safari </li>
      <li>Microsoft Edge </li>
      <li>Internet Explorer 11 </li>
     </ul> 
     <p>The browser vendors release regular minor version updates with important bug fixes and security updates. Support is only provided for the current minor version of the major version you are running. </p>
     <p>Each time a new browser version is released, we begin supporting that version and stop supporting the third most recent version. </p>
     <div class="admonition-wrapper note">
      <div class="admonition alert alert-info"> <i class="fa fa-info-circle fa-fw" aria-hidden="true"></i><strong>Note:</strong> We do not support running GitLab with JavaScript disabled in the browser and have no plans of supporting that in the future because we have features such as Issue Boards which require JavaScript extensively.
      </div>
