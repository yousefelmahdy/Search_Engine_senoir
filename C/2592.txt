         <h3 id="list-organizations" class="pt-3"> <a href="#list-organizations">List organizations</a> </h3> 
         <p>Lists all organizations, in the order that they were created on GitHub.</p> 
         <p><strong>Note:</strong> Pagination is powered exclusively by the <code>since</code> parameter. Use the <a href="https://docs.github.com/rest/overview/resources-in-the-rest-api#link-header">Link header</a> to get the URL for the next page of organizations.</p> 
        </div> 
        <pre><code><span class="color-bg-info-inverse color-text-inverse rounded-1 px-2 py-1" style="text-transform: uppercase">get</span> /organizations</code></pre> 
        <div>

         <h3 id="get-an-organization" class="pt-3"> <a href="#get-an-organization">Get an organization</a> </h3> 
         <p>To see many of the organization response values, you need to be an authenticated organization owner with the <code>admin:org</code> scope. When the value of <code>two_factor_requirement_enabled</code> is <code>true</code>, the organization requires all members, billing managers, and outside collaborators to enable <a href="https://help.github.com/articles/securing-your-account-with-two-factor-authentication-2fa/">two-factor authentication</a>.</p> 
         <p>GitHub Apps with the <code>Organization plan</code> permission can use this endpoint to retrieve information about an organization's GitHub plan. See "<a href="https://docs.github.com/apps/building-github-apps/authenticating-with-github-apps/">Authenticating with GitHub Apps</a>" for details. For an example response, see 'Response with GitHub plan information' below."</p>

         <h3 id="set-organization-membership-for-a-user" class="pt-3"> <a href="#set-organization-membership-for-a-user">Set organization membership for a user</a> </h3> 
         <p>Only authenticated organization owners can add a member to the organization or update the member's role.</p> 
         <ul> 
          <li>If the authenticated user is <em>adding</em> a member to the organization, the invited user will receive an email inviting them to the organization. The user's <a href="https://docs.github.com/rest/reference/orgs#get-organization-membership-for-a-user">membership status</a> will be <code>pending</code> until they accept the invitation.</li> 
          <li>Authenticated users can <em>update</em> a user's membership by passing the <code>role</code> parameter. If the authenticated user changes a member's role to <code>admin</code>, the affected user will receive an email notifying them that they've been made an organization owner. If the authenticated user changes an owner's role to <code>member</code>, no email will be sent.</li> 
         </ul> 
         <p><strong>Rate limits</strong></p> 
         <p>To prevent abuse, the authenticated user is limited to 50 organization invitations per 24 hour period. If the organization is more than one month old or on a paid plan, the limit is 500 invitations per 24 hour period.</p> 
        </div>

       <h2 id="webhooks"><a href="#webhooks">Webhooks</a></h2> 
       <p>Organization webhooks allow you to receive HTTP <code>POST</code> payloads whenever certain events happen within the organization. Subscribing to these events makes it possible to build integrations that react to events on GitHub. For more information on actions you can subscribe to, see "<a href="/en/developers/webhooks-and-events/github-event-types">GitHub event types</a>."</p> 
       <h3 id="scopes--restrictions"><a href="#scopes--restrictions">Scopes &amp; Restrictions</a></h3> 
       <p>All actions against organization webhooks require the authenticated user to be an admin of the organization being managed. Additionally, OAuth tokens require the <code>admin:org_hook</code> scope. For more information, see "<a href="/en/developers/apps/scopes-for-oauth-apps">Scopes for OAuth Apps</a>."</p> 
       <p>In order to protect sensitive data which may be present in webhook configurations, we also enforce the following access control rules:</p> 
       <ul> 
        <li>OAuth applications cannot list, view, or edit webhooks which they did not create.</li> 
        <li>Users cannot list, view, or edit webhooks which were created by OAuth applications.</li> 
       </ul> 
       <h3 id="receiving-webhooks"><a href="#receiving-webhooks">Receiving Webhooks</a></h3> 
       <p>In order for GitHub to send webhook payloads, your server needs to be accessible from the Internet. We also highly suggest using SSL so that we can send encrypted payloads over HTTPS.</p> 
       <p>For more best practices, <a href="/en/guides/best-practices-for-integrators">see our guide</a>.</p> 
       <h4 id="webhook-headers"><a href="#webhook-headers">Webhook headers</a></h4> 
       <p>GitHub will send along several HTTP headers to differentiate between event types and payload identifiers. See <a href="/en/webhooks/event-payloads/#delivery-headers">webhook headers</a> for details.</p>
