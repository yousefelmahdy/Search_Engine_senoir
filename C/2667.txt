        <div class="d-none d-lg-block ml-2"> <button class="btn-link Link--secondary js-print tooltipped tooltipped-n" aria-label="Print this article"> <!-- From https://heroicons.dev/ -->

      <div id="article-contents" class="article-grid-body"> 
       <p>Let's walk through core API concepts as we tackle some everyday use cases. </p> 
       <h3 id="overview"><a href="#overview">Overview</a></h3> 
       <p>Most applications will use an existing <a href="/en/libraries">wrapper library</a> in the language of your choice, but it's important to familiarize yourself with the underlying API HTTP methods first.</p> 
       <p>There's no easier way to kick the tires than through <a href="http://curl.haxx.se/">cURL</a>. If you are using an alternative client, note that you are required to send a valid <a href="/en/rest/overview/resources-in-the-rest-api#user-agent-required">User Agent header</a> in your request.</p> 
       <h4 id="hello-world"><a href="#hello-world">Hello World</a></h4> 
       <p>Let's start by testing our setup. Open up a command prompt and enter the following command:</p> 
       <pre><code class="hljs language-shell">$ curl https://api.github.com/zen

&gt; Keep it logically awesome.</code></pre> 
       <p>The response will be a random selection from our design philosophies.</p> 
       <p>Next, let's <code>GET</code> <a href="https://github.com/defunkt">Chris Wanstrath's</a> <a href="/en/rest/reference/users#get-a-user">GitHub profile</a>:</p> 
       <pre><code class="hljs language-shell"># GET /users/defunkt
$ curl https://api.github.com/users/defunkt

&gt; {
&gt;   "login": "defunkt",
&gt;   "id": 2,
&gt;   "url": "https://api.github.com/users/defunkt",
&gt;   "html_url": "https://github.com/defunkt",
&gt;   ...
&gt; }</code></pre> 
       <p>Mmmmm, tastes like <a href="http://en.wikipedia.org/wiki/JSON">JSON</a>. Let's add the <code>-i</code> flag to include headers:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/users/defunkt

&gt; HTTP/2 200
&gt; Server: GitHub.com
&gt; Date: Sun, 11 Nov 2012 18:43:28 GMT
&gt; Content-Type: application/json; charset=utf-8
&gt; ETag: "bfd85cbf23ac0b0c8a29bee02e7117c6"
&gt; X-RateLimit-Limit: 60
&gt; X-RateLimit-Remaining: 57
&gt; X-RateLimit-Reset: 1352660008
&gt; X-GitHub-Media-Type: github.v3
&gt; Vary: Accept
&gt; Cache-Control: public, max-age=60, s-maxage=60
&gt; X-Content-Type-Options: nosniff
&gt; Content-Length: 692
&gt; Last-Modified: Tue, 30 Oct 2012 18:58:42 GMT

&gt; {
&gt;   "login": "defunkt",
&gt;   "id": 2,
&gt;   "url": "https://api.github.com/users/defunkt",
&gt;   "html_url": "https://github.com/defunkt",
&gt;   ...
&gt; }</code></pre> 
       <p>There are a few interesting bits in the response headers. As expected, the <code>Content-Type</code> is <code>application/json</code>.</p> 
       <p>Any headers beginning with <code>X-</code> are custom headers, and are not included in the HTTP spec. For example:</p> 
       <ul> 
        <li><code>X-GitHub-Media-Type</code> has a value of <code>github.v3</code>. This lets us know the <a href="/en/rest/overview/media-types">media type</a> for the response. Media types have helped us version our output in API v3. We'll talk more about that later.</li> 
        <li>Take note of the <code>X-RateLimit-Limit</code> and <code>X-RateLimit-Remaining</code> headers. This pair of headers indicate <a href="/en/rest#rate-limiting">how many requests a client can make</a> in a rolling time period (typically an hour) and how many of those requests the client has already spent.</li> 
       </ul> 
       <h3 id="authentication"><a href="#authentication">Authentication</a></h3> 
       <p>Unauthenticated clients can make 60 requests per hour. To get more requests per hour, we'll need to <em>authenticate</em>. In fact, doing anything interesting with the GitHub API requires <a href="/en/rest#authentication">authentication</a>.</p> 
       <h4 id="using-personal-access-tokens"><a href="#using-personal-access-tokens">Using personal access tokens</a></h4> 
       <p>The easiest and best way to authenticate with the GitHub API is by using Basic Authentication <a href="/en/rest/overview/other-authentication-methods#via-oauth-and-personal-access-tokens">via OAuth tokens</a>. OAuth tokens include <a href="/en/articles/creating-an-access-token-for-command-line-use">personal access tokens</a>.</p> 
       <p>Use a <code>-u</code> flag to set your username:</p> 
       <pre><code class="hljs language-shell">$ curl -i -u <em>your_username</em> https://api.github.com/users/octocat
</code></pre> 
       <p>When prompted, you can enter your OAuth token, but we recommend you set up a variable for it:</p> 
       <p>You can use <code>-u "username:$token"</code> and set up a variable for <code>token</code> to avoid leaving your token in shell history, which should be avoided.</p> 
       <pre><code class="hljs language-shell">$ curl -i -u <em>username:$token</em> https://api.github.com/users/octocat
</code></pre> 
       <p>When authenticating, you should see your rate limit bumped to 5,000 requests an hour, as indicated in the <code>X-RateLimit-Limit</code> header. In addition to providing more calls per hour, authentication enables you to read and write private information using the API.</p> 
       <p>You can easily <a href="/en/articles/creating-an-access-token-for-command-line-use">create a <strong>personal access token</strong></a> using your <a href="https://github.com/settings/tokens">Personal access tokens settings page</a>:</p> 
       <p><img src="/assets/images/personal_token.png" alt="Personal Token selection"></p> 
       <h4 id="get-your-own-user-profile"><a href="#get-your-own-user-profile">Get your own user profile</a></h4> 
       <p>When properly authenticated, you can take advantage of the permissions associated with your GitHub account. For example, try getting <a href="/en/rest/reference/users#get-the-authenticated-user">your own user profile</a>:</p> 
       <pre><code class="hljs language-shell">$ curl -i -u <em>your_username</em>:<em>your_token</em> https://api.github.com/user

&gt; {
&gt;   ...
&gt;   "plan": {
&gt;     "space": 2516582,
&gt;    "collaborators": 10,
&gt;    "private_repos": 20,
&gt;    "name": "medium"
&gt;  }
&gt;   ...
&gt; }</code></pre> 
       <p>This time, in addition to the same set of public information we retrieved for <a href="https://github.com/defunkt">@defunkt</a> earlier, you should also see the non-public information for your user profile. For example, you'll see a <code>plan</code> object in the response which gives details about the GitHub plan for the account.</p> 
       <h4 id="using-oauth-tokens-for-apps"><a href="#using-oauth-tokens-for-apps">Using OAuth tokens for apps</a></h4> 
       <p>Apps that need to read or write private information using the API on behalf of another user should use <a href="/en/apps/building-integrations/setting-up-and-registering-oauth-apps">OAuth</a>.</p> 
       <p>OAuth uses <em>tokens</em>. Tokens provide two big features:</p> 
       <ul> 
        <li><strong>Revokable access</strong>: users can revoke authorization to third party apps at any time</li> 
        <li><strong>Limited access</strong>: users can review the specific access that a token will provide before authorizing a third party app</li> 
       </ul> 
       <p>Tokens should be created via a <a href="/en/apps/building-oauth-apps/authorizing-oauth-apps">web flow</a>. An application sends users to GitHub to log in. GitHub then presents a dialog indicating the name of the app, as well as the level of access the app has once it's authorized by the user. After a user authorizes access, GitHub redirects the user back to the application:</p> 
       <p><img src="/assets/images/oauth_prompt.png" alt="GitHub's OAuth Prompt"></p> 
       <p><strong>Treat OAuth tokens like passwords!</strong> Don't share them with other users or store them in insecure places. The tokens in these examples are fake and the names have been changed to protect the innocent.</p> 
       <p>Now that we've got the hang of making authenticated calls, let's move along to the <a href="/en/rest/reference/repos">Repositories API</a>.</p> 
       <h3 id="repositories"><a href="#repositories">Repositories</a></h3> 
       <p>Almost any meaningful use of the GitHub API will involve some level of Repository information. We can <a href="/en/rest/reference/repos#get-a-repository"><code>GET</code> repository details</a> in the same way we fetched user details earlier:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/repos/twbs/bootstrap</code></pre> 
       <p>In the same way, we can <a href="/en/rest/reference/repos#list-repositories-for-the-authenticated-user">view repositories for the authenticated user</a>:</p> 
       <pre><code class="hljs language-shell">$ curl -i -H "Authorization: token ghp_16C7e42F292c6912E7710c838347Ae178B4a" \
    https://api.github.com/user/repos</code></pre> 
       <p>Or, we can <a href="/en/rest/reference/repos#list-repositories-for-a-user">list repositories for another user</a>:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/users/octocat/repos</code></pre> 
       <p>Or, we can <a href="/en/rest/reference/repos#list-organization-repositories">list repositories for an organization</a>:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/orgs/octo-org/repos</code></pre> 
       <p>The information returned from these calls will depend on which scopes our token has when we authenticate:</p> 
       <ul> 
        <li>A token with <code>public_repo</code> <a href="/en/apps/building-oauth-apps/understanding-scopes-for-oauth-apps">scope</a> returns a response that includes all public repositories we have access to see on github.com.</li> 
        <li>A token with <code>repo</code> <a href="/en/apps/building-oauth-apps/understanding-scopes-for-oauth-apps">scope</a> returns a response that includes all public and private repositories we have access to see on GitHub.</li> 
       </ul> 
       <p>As the <a href="/en/rest/reference/repos">docs</a> indicate, these methods take a <code>type</code> parameter that can filter the repositories returned based on what type of access the user has for the repository. In this way, we can fetch only directly-owned repositories, organization repositories, or repositories the user collaborates on via a team.</p> 
       <pre><code class="hljs language-shell">$ curl -i "https://api.github.com/users/octocat/repos?type=owner"</code></pre> 
       <p>In this example, we grab only those repositories that octocat owns, not the ones on which she collaborates. Note the quoted URL above. Depending on your shell setup, cURL sometimes requires a quoted URL or else it ignores the query string.</p> 
       <h4 id="create-a-repository"><a href="#create-a-repository">Create a repository</a></h4> 
       <p>Fetching information for existing repositories is a common use case, but the GitHub API supports creating new repositories as well. To <a href="/en/rest/reference/repos#create-a-repository-for-the-authenticated-user">create a repository</a>, we need to <code>POST</code> some JSON containing the details and configuration options.</p> 
       <pre><code class="hljs language-shell">$ curl -i -H "Authorization: token ghp_16C7e42F292c6912E7710c838347Ae178B4a" \
    -d '{ \
        "name": "blog", \
        "auto_init": true, \
        "private": true, \
        "gitignore_template": "nanoc" \
      }' \
    https://api.github.com/user/repos</code></pre> 
       <p>In this minimal example, we create a new private repository for our blog (to be served on <a href="http://pages.github.com">GitHub Pages</a>, perhaps). Though the blog will be public, we've made the repository private. In this single step, we'll also initialize it with a README and a <a href="http://nanoc.ws/">nanoc</a>-flavored <a href="https://github.com/github/gitignore">.gitignore template</a>.</p> 
       <p>The resulting repository will be found at <code>https://github.com/&lt;your_username&gt;/blog</code>. To create a repository under an organization for which you're an owner, just change the API method from <code>/user/repos</code> to <code>/orgs/&lt;org_name&gt;/repos</code>.</p> 
       <p>Next, let's fetch our newly created repository:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/repos/pengwynn/blog

&gt; HTTP/2 404

&gt; {
&gt;    "message": "Not Found"
&gt; }</code></pre> 
       <p>Oh noes! Where did it go? Since we created the repository as <em>private</em>, we need to authenticate in order to see it. If you're a grizzled HTTP user, you might expect a <code>403</code> instead. Since we don't want to leak information about private repositories, the GitHub API returns a <code>404</code> in this case, as if to say "we can neither confirm nor deny the existence of this repository."</p> 
       <h3 id="issues"><a href="#issues">Issues</a></h3> 
       <p>The UI for Issues on GitHub aims to provide 'just enough' workflow while staying out of your way. With the GitHub <a href="/en/rest/reference/issues">Issues API</a>, you can pull data out or create issues from other tools to create a workflow that works for your team.</p> 
       <p>Just like github.com, the API provides a few methods to view issues for the authenticated user. To <a href="/en/rest/reference/issues#list-issues-assigned-to-the-authenticated-user">see all your issues</a>, call <code>GET /issues</code>:</p> 
       <pre><code class="hljs language-shell">$ curl -i -H "Authorization: token ghp_16C7e42F292c6912E7710c838347Ae178B4a" \
    https://api.github.com/issues</code></pre> 
       <p>To get only the <a href="/en/rest/reference/issues#list-issues-assigned-to-the-authenticated-user">issues under one of your GitHub organizations</a>, call <code>GET /orgs/&lt;org&gt;/issues</code>:</p> 
       <pre><code class="hljs language-shell">$ curl -i -H "Authorization: token ghp_16C7e42F292c6912E7710c838347Ae178B4a" \
    https://api.github.com/orgs/rails/issues</code></pre> 
       <p>We can also get <a href="/en/rest/reference/issues#list-repository-issues">all the issues under a single repository</a>:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/repos/rails/rails/issues</code></pre> 
       <h4 id="pagination"><a href="#pagination">Pagination</a></h4> 
       <p>A project the size of Rails has thousands of issues. We'll need to <a href="/en/rest#pagination">paginate</a>, making multiple API calls to get the data. Let's repeat that last call, this time taking note of the response headers:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/repos/rails/rails/issues

&gt; HTTP/2 200

&gt; ...
&gt; Link: &lt;https://api.github.com/repositories/8514/issues?page=2&gt;; rel="next", &lt;https://api.github.com/repositories/8514/issues?page=30&gt;; rel="last"
&gt; ...</code></pre> 
       <p>The <a href="https://www.w3.org/wiki/LinkHeader"><code>Link</code> header</a> provides a way for a response to link to external resources, in this case additional pages of data. Since our call found more than thirty issues (the default page size), the API tells us where we can find the next page and the last page of results.</p> 
       <h4 id="creating-an-issue"><a href="#creating-an-issue">Creating an issue</a></h4> 
       <p>Now that we've seen how to paginate lists of issues, let's <a href="/en/rest/reference/issues#create-an-issue">create an issue</a> from the API.</p> 
       <p>To create an issue, we need to be authenticated, so we'll pass an OAuth token in the header. Also, we'll pass the title, body, and labels in the JSON body to the <code>/issues</code> path underneath the repository in which we want to create the issue:</p> 
       <pre><code class="hljs language-shell">$ curl -i -H 'Authorization: token ghp_16C7e42F292c6912E7710c838347Ae178B4a' \
$    -d '{ \
$         "title": "New logo", \
$         "body": "We should have one", \
$         "labels": ["design"] \
$       }' \
$    https://api.github.com/repos/pengwynn/api-sandbox/issues

&gt; HTTP/2 201
&gt; Location: https://api.github.com/repos/pengwynn/api-sandbox/issues/17
&gt; X-RateLimit-Limit: 5000

&gt; {
&gt;   "pull_request": {

       <h3 id="conditional-requests"><a href="#conditional-requests">Conditional requests</a></h3> 
       <p>A big part of being a good API citizen is respecting rate limits by caching information that hasn't changed. The API supports <a href="/en/rest#conditional-requests">conditional requests</a> and helps you do the right thing. Consider the first call we made to get defunkt's profile:</p> 
       <pre><code class="hljs language-shell">$ curl -i https://api.github.com/users/defunkt

&gt; HTTP/2 200
&gt; ETag: "bfd85cbf23ac0b0c8a29bee02e7117c6"</code></pre>
