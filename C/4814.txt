     <h1 id="groups">Groups<a href="#groups" title="Permalink" class="anchor"></a> </h1> 
     <p>With GitLab Groups you can assemble related projects together and grant members access to several projects at once. </p>
     <p>Groups can also be nested in <a href="subgroups/index.html">subgroups</a>. </p>
     <p>Find your groups by clicking <strong>Groups</strong> in the top navigation. </p>
     <p><img src="img/groups.png" alt="GitLab Groups"> </p>
     <blockquote> 
      <p>The groups dropdown in the top navigation was [introduced][ce-36234] in <a href="https://about.gitlab.com/2018/07/22/gitlab-11-1-released/#groups-dropdown-in-navigation">GitLab 11.1</a>. </p>
     </blockquote> 
     <p>The Groups page displays all groups you are a member of, how many projects it holds, how many members it has, the group visibility, and, if you have enough permissions, a link to the group settings. By clicking the last button you can leave that group. </p>
     <h2 id="use-cases">Use cases<a href="#use-cases" title="Permalink" class="anchor"></a> </h2> 
     <p>You can create groups for numerous reasons. To name a few: </p>
     <ul> 
      <li>Organize related projects under the same <a href="#namespaces">namespace</a>, add members to that group and grant access to all their projects at once </li>
      <li>Create a group, include members of your team, and make it easier to <code class="highlighter-rouge">@mention</code> all the team at once in issues and merge requests 
       <ul> 
        <li>Create a group for your company members, and create <a href="subgroups/index.html">subgroups</a> for each individual team. Let’s say you create a group called <code class="highlighter-rouge">company-team</code>, and among others, you created subgroups in this group for each individual team <code class="highlighter-rouge">backend-team</code>, <code class="highlighter-rouge">frontend-team</code>, and <code class="highlighter-rouge">production-team</code>: 
         <ol> 
          <li>When you start a new implementation from an issue, you add a comment: <em>”<code class="highlighter-rouge">@company-team</code>, let’s do it! <code class="highlighter-rouge">@company-team/backend-team</code> you’re good to go!”</em> </li>
          <li>When your backend team needs help from frontend, they add a comment: <em>”<code class="highlighter-rouge">@company-team/frontend-team</code> could you help us here please?”</em> </li>
          <li>When the frontend team completes their implementation, they comment: <em>”<code class="highlighter-rouge">@company-team/backend-team</code>, it’s done! Let’s ship it <code class="highlighter-rouge">@company-team/production-team</code>!”</em> </li>
         </ol> </li>
       </ul> </li>
     </ul> 
     <h2 id="namespaces">Namespaces<a href="#namespaces" title="Permalink" class="anchor"></a> </h2> 
     <p>In GitLab, a namespace is a unique name to be used as a user name, a group name, or a subgroup name. </p>
     <ul> 
      <li><code class="highlighter-rouge">http://gitlab.example.com/username</code> </li>
      <li><code class="highlighter-rouge">http://gitlab.example.com/groupname</code> </li>
      <li><code class="highlighter-rouge">http://gitlab.example.com/groupname/subgroup_name</code> </li>
     </ul> 
     <p>For example, consider a user named Alex: </p>
     <ol> 
      <li>Alex creates an account on GitLab.com with the username <code class="highlighter-rouge">alex</code>; their profile will be accessed under <code class="highlighter-rouge">https://gitlab.example.com/alex</code> </li>
      <li>Alex creates a group for their team with the group name <code class="highlighter-rouge">alex-team</code>; the group and its projects will be accessed under <code class="highlighter-rouge">https://gitlab.example.com/alex-team</code> </li>
      <li>Alex creates a subgroup of <code class="highlighter-rouge">alex-team</code> with the subgroup name <code class="highlighter-rouge">marketing</code>; this subgroup and its projects will be accessed under <code class="highlighter-rouge">https://gitlab.example.com/alex-team/marketing</code> </li>
     </ol> 
     <p>By doing so: </p>
     <ul> 
      <li>Any team member mentions Alex with <code class="highlighter-rouge">@alex</code> </li>
      <li>Alex mentions everyone from their team with <code class="highlighter-rouge">@alex-team</code> </li>
      <li>Alex mentions only the marketing team with <code class="highlighter-rouge">@alex-team/marketing</code> </li>
     </ul> 
     <h2 id="issues-and-merge-requests-within-a-group">Issues and merge requests within a group<a href="#issues-and-merge-requests-within-a-group" title="Permalink" class="anchor"></a> </h2> 
     <p>Issues and merge requests are part of projects. For a given group, view all the <a href="../project/issues/index.html#issues-list">issues</a> and <a href="../project/merge_requests/index.html#merge-requests-per-group">merge requests</a> across all the projects in that group, together in a single list view. </p>
     <h2 id="create-a-new-group">Create a new group<a href="#create-a-new-group" title="Permalink" class="anchor"></a> </h2> 
     <blockquote> 
      <p>For a list of words that are not allowed to be used as group names see the <a href="../reserved_names.html">reserved names</a>. </p>
     </blockquote> 
     <p>You can create a group in GitLab from: </p>
     <ol> 
      <li> <p>The Groups page: from the top menu, click <strong>Groups</strong>, and click the green button <strong>New group</strong>: </p><p><img src="img/new_group_from_groups.png" alt="new group from groups page"> </p></li>
      <li> <p>Elsewhere: expand the <code class="highlighter-rouge">plus</code> sign button on the top navbar and choose <strong>New group</strong>: </p><p><img src="img/new_group_from_other_pages.png" alt="new group from elsewhere"> </p></li>
     </ol> 
     <p>Add the following information: </p>
     <p><img src="img/create_new_group_info.png" alt="new group info"> </p>
     <ol> 
      <li>The <strong>Group name</strong> will populate the URL automatically. Optionally, you can change it. This is the name that is displayed in the group views. The name can contain only: 
       <ul> 
        <li>Alphanumeric characters. </li>
        <li>Underscores. </li>
        <li>Dashes and dots. </li>
        <li>Spaces. </li>
       </ul> </li>
      <li>The <strong>Group URL</strong>, which will be the namespace under which your projects will be hosted. The URL can contain only: 
       <ul> 
        <li>Alphanumeric characters. </li>
        <li>Underscores. </li>
        <li>Dashes and dots. It cannot start with dashes or end in dot. </li>
       </ul> </li>
      <li>Optionally, you can add a brief description to tell others what this group is about. </li>
      <li>Optionally, choose an avatar for your group. </li>
      <li>Choose the <a href="../../public_access/public_access.html">visibility level</a>. </li>
     </ol> 
     <h2 id="add-users-to-a-group">Add users to a group<a href="#add-users-to-a-group" title="Permalink" class="anchor"></a> </h2> 
     <p>Add members to a group by navigating to the group’s dashboard, and clicking <strong>Members</strong>: </p>
     <p><img src="img/add_new_members.png" alt="add members to group"> </p>
     <p>Select the <a href="../permissions.html#permissions">permission level</a> and add the new member. You can also set the expiring date for that user, from which they will no longer have access to your group. </p>
     <p>One of the benefits of putting multiple projects in one group is that you can give a user to access to all projects in the group with one action. </p>
     <p>Consider we have a group with two projects: </p>
     <ul> 
      <li>On the <strong>Group Members</strong> page we can now add a new user to the group. </li>
      <li>Now because this user is a <strong>Developer</strong> member of the group, he automatically gets <strong>Developer</strong> access to <strong>all projects</strong> within that group. </li>
     </ul> 
     <p>If necessary, you can increase the access level of an individual user for a specific project, by adding them again as a new member to the project with the new permission levels. </p>
     <h2 id="request-access-to-a-group">Request access to a group<a href="#request-access-to-a-group" title="Permalink" class="anchor"></a> </h2> 
     <p>As a group owner you can enable or disable non members to request access to your group. Go to the group settings and click on <strong>Allow users to request access</strong>. </p>
     <p>As a user, you can request to be a member of a group. Go to the group you’d like to be a member of, and click the <strong>Request Access</strong> button on the right side of your screen. </p>
     <p><img src="img/request_access_button.png" alt="Request access button"> </p>
     <hr> 
     <p>Group owners and maintainers will be notified of your request and will be able to approve or decline it on the members page. </p>
     <p><img src="img/access_requests_management.png" alt="Manage access requests"> </p>
     <hr> 
     <p>If you change your mind before your request is approved, just click the <strong>Withdraw Access Request</strong> button. </p>
     <p><img src="img/withdraw_access_request_button.png" alt="Withdraw access request button"> </p>
     <h2 id="add-projects-to-a-group">Add projects to a group<a href="#add-projects-to-a-group" title="Permalink" class="anchor"></a> </h2>

     <h2 id="transferring-groups">Transferring groups<a href="#transferring-groups" title="Permalink" class="anchor"></a> </h2> 
     <p>From GitLab 10.5, groups can be transferred in the following ways: </p>
     <ul> 
      <li>Top-level groups can be transferred to a group, converting them into subgroups. </li>
      <li>Subgroups can be transferred to a new parent group. </li>
      <li>Subgroups can be transferred out from a parent group, converting them into top-level groups. </li>
     </ul> 
     <p>When transferring groups, note: </p>
     <ul> 
      <li>Changing a group’s parent can have unintended side effects. See <a href="../project/index.html#redirects-when-changing-repository-paths">Redirects when changing repository paths</a>. </li>
      <li>You can only transfer groups to groups you manage. </li>
      <li>You will need to update your local repositories to point to the new location. </li>
      <li>If the parent group’s visibility is lower than the group’s current visibility, visibility levels for subgroups and projects will be changed to match the new parent group’s visibility. </li>
      <li>Only explicit group membership is transferred, not inherited membership. If the group’s owners have only inherited membership, this would leave the group without an owner. In this case, the user transferring the group becomes the group’s owner. </li>
     </ul>

     <p>Changing a group’s path can have unintended side effects. Read <a href="../project/index.html#redirects-when-changing-repository-paths">how redirects will behave</a> before proceeding. </p>
     <p>If you are vacating the path so it can be claimed by another group or user, you may need to rename the group name as well since both names and paths must be unique. </p>
     <p>To change your group path: </p>
     <ol> 
      <li>Navigate to your group’s <strong>Settings &gt; General</strong>. </li>
      <li>Enter a new name under “Group path”. </li>
      <li>Hit <strong>Save group</strong>. </li>
     </ol>

      <div class="admonition alert alert-warning"> <i class="fa fa-exclamation-triangle fa-fw" aria-hidden="true"></i><strong>Caution:</strong> It is currently not possible to rename a namespace if it contains a project with <a href="../project/container_registry.html">Container Registry</a> tags, because the project cannot be moved.
      </div>
     </div> 
     <div class="admonition-wrapper tip">
      <div class="admonition alert alert-success"> <i class="fa fa-pencil fa-fw" aria-hidden="true"></i><strong>TIP:</strong> If you want to retain ownership over the original namespace and protect the URL redirects, then instead of changing a group’s path or renaming a username, you can create a new group and transfer projects to it.

     <h4 id="share-with-group-lock">Share with group lock<a href="#share-with-group-lock" title="Permalink" class="anchor"></a> </h4> 
     <p>Prevent projects in a group from <a href="../project/members/share_project_with_groups.html">sharing a project with another group</a>. This allows for tighter control over project access. </p>
     <p>For example, consider you have two distinct teams (Group A and Group B) working together in a project. To inherit the group membership, you share the project between the two groups A and B. <strong>Share with group lock</strong> prevents any project within the group from being shared with another group. By doing so, you guarantee only the right group members have access to those projects. </p>
     <p>To enable this feature, navigate to the group settings page. Select <strong>Share with group lock</strong> and <strong>Save the group</strong>. </p>
     <p><img src="img/share_with_group_lock.png" alt="Checkbox for share with group lock"> </p>
     <h4 id="member-lock-starter">Member Lock <span class="badge-trigger starter"></span><a href="#member-lock-starter" title="Permalink" class="anchor"></a> </h4> 
     <p>With Member lock, it is possible to lock membership in a project to the level of members in the group. </p>
     <p>Member lock lets a group owner lock down any new project membership to all the projects within the group, allowing tighter control over project membership. </p>
     <p>For instance, if you want to lock the group for an <a href="/11.11/ee/administration/audit_events.html">Audit Event</a>, you enable Member lock to guarantee that membership of a project cannot be modified during that audit. </p>
     <p>To enable this feature: </p>
     <ol> 
      <li>Navigate to the group’s <strong>Settings &gt; General</strong> page. </li>
      <li>Expand the <strong>Permissions, LFS, 2FA</strong> section and select <strong>Member lock</strong>. </li>
      <li>Click the <strong>Save changes</strong> button. </li>
     </ol> 
     <p><img src="img/member_lock.png" alt="Checkbox for membership lock"> </p>
     <p>This will disable the option for all users who previously had permissions to operate project memberships so no new users can be added. Furthermore, any request to add a new user to a project through API will not be possible. </p>
     <h4 id="group-file-templates-premium">Group file templates <span class="badge-trigger premium"></span><a href="#group-file-templates-premium" title="Permalink" class="anchor"></a> </h4> 
     <p>Group file templates allow you to share a set of templates for common file types with every project in a group. It is analogous to the <a href="/11.11/ee/user/admin_area/settings/instance_template_repository.html">instance template repository</a> feature, and the selected project should follow the same naming conventions as are documented on that page. </p>
     <p>Only projects that are in the group may be chosen as the source of templates. This includes projects shared with the group, but <strong>excludes</strong> projects in subgroups or parent groups of the group being configured. </p>
     <p>This feature may be configured for subgroups as well as parent groups. A project in a subgroup will have access to the templates for that subgroup, as well as any parent groups. </p>
     <p><img src="img/group_file_template_dropdown.png" alt="Group file template dropdown"> </p>
     <p>To enable this feature, navigate to the group settings page, expand the <strong>Templates</strong> section, choose a project to act as the template repository, and <strong>Save group</strong>. </p>
     <p><img src="img/group_file_template_settings.png" alt="Group file template settings"> </p>
