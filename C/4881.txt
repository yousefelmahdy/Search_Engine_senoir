         <h1 id="requirements">Requirements <span class="badge-trigger free-self"></span><a href="#requirements" title="Permalink" class="anchor"></a> </h1> 
         <p>This page includes useful information on the supported Operating Systems as well as the hardware requirements that are needed to install and use GitLab. </p>
         <h2 id="operating-systems">Operating Systems<a href="#operating-systems" title="Permalink" class="anchor"></a> </h2> 
         <h3 id="supported-linux-distributions">Supported Linux distributions<a href="#supported-linux-distributions" title="Permalink" class="anchor"></a> </h3> 
         <ul> 
          <li>Ubuntu (16.04/18.04/20.04) </li>
          <li>Debian (9/10) </li>
          <li>CentOS (7/8) </li>
          <li>openSUSE (Leap 15.1/Enterprise Server 12.2) </li>
          <li>Red Hat Enterprise Linux (please use the CentOS packages and instructions) </li>
          <li>Scientific Linux (please use the CentOS packages and instructions) </li>
          <li>Oracle Linux (please use the CentOS packages and instructions) </li>
         </ul> 
         <p>For the installation options, see <a href="index.html">the main installation page</a>. </p>
         <h3 id="unsupported-linux-distributions-and-unix-like-operating-systems">Unsupported Linux distributions and Unix-like operating systems<a href="#unsupported-linux-distributions-and-unix-like-operating-systems" title="Permalink" class="anchor"></a> </h3> 
         <ul> 
          <li>Arch Linux </li>
          <li>Fedora </li>
          <li>FreeBSD </li>
          <li>Gentoo </li>
          <li>macOS </li>
         </ul> 
         <p>Installation of GitLab on these operating systems is possible, but not supported. Please see the <a href="installation.html">installation from source guide</a> and the <a href="https://about.gitlab.com/install/">installation guides</a> for more information. </p>
         <p>Please see <a href="/13.10/omnibus/package-information/deprecated_os.html">OS versions that are no longer supported</a> for Omnibus installs page for a list of supported and unsupported OS versions as well as the last support GitLab version for that OS. </p>
         <h3 id="microsoft-windows">Microsoft Windows<a href="#microsoft-windows" title="Permalink" class="anchor"></a> </h3> 
         <p>GitLab is developed for Linux-based operating systems. It does <strong>not</strong> run on Microsoft Windows, and we have no plans to support it in the near future. For the latest development status view this <a href="https://gitlab.com/gitlab-org/gitlab/-/issues/22337">issue</a>. Please consider using a virtual machine to run GitLab. </p>
         <h2 id="software-requirements">Software requirements<a href="#software-requirements" title="Permalink" class="anchor"></a> </h2> 
         <h3 id="ruby-versions">Ruby versions<a href="#ruby-versions" title="Permalink" class="anchor"></a> </h3> 
         <p>From GitLab 13.6: </p>
         <ul> 
          <li>Ruby 2.7 and later is required. </li>
         </ul> 
         <p>From GitLab 12.2: </p>
         <ul> 
          <li>Ruby 2.6 and later is required. </li>
         </ul> 
         <p>You must use the standard MRI implementation of Ruby. We love <a href="https://www.jruby.org/">JRuby</a> and <a href="https://github.com/rubinius/rubinius#the-rubinius-language-platform">Rubinius</a>, but GitLab needs several Gems that have native extensions. </p>
         <h3 id="go-versions">Go versions<a href="#go-versions" title="Permalink" class="anchor"></a> </h3> 
         <p>The minimum required Go version is 1.13. </p>
         <h3 id="git-versions">Git versions<a href="#git-versions" title="Permalink" class="anchor"></a> </h3> 
         <p>From GitLab 13.6: </p>
         <ul> 
          <li>Git 2.29.x and later is required. </li>
         </ul> 
         <p>From GitLab 13.1: </p>
         <ul> 
          <li>Git 2.24.x and later is required. </li>
          <li>Git 2.28.x and later <a href="https://gitlab.com/gitlab-org/gitaly/-/issues/2959">is recommended</a>. </li>
         </ul> 
         <h3 id="nodejs-versions">Node.js versions<a href="#nodejs-versions" title="Permalink" class="anchor"></a> </h3> 
         <p>Beginning in GitLab 12.9, we only support Node.js 10.13.0 or higher, and we have dropped support for Node.js 8. (Node.js 6 support was dropped in GitLab 11.8) </p>
         <p>We recommend Node 14.x, as its faster. </p>
         <p>GitLab uses <a href="https://webpack.js.org/">webpack</a> to compile frontend assets, which requires a minimum version of Node.js 10.13.0. </p>
         <p>You can check which version youre running with <code class="highlighter-rouge">node -v</code>. If youre running a version older than <code class="highlighter-rouge">v10.13.0</code>, you need to update it to a newer version. You can find instructions to install from community maintained packages or compile from source at the <a href="https://nodejs.org/en/download/">Node.js website</a>. </p>
         <h3 id="redis-versions">Redis versions<a href="#redis-versions" title="Permalink" class="anchor"></a> </h3> 
         <p>GitLab 13.0 and later requires Redis version 4.0 or higher. </p>
         <p>Redis version 5.0 or higher is recommended, as this is what ships with <a href="/13.10/omnibus/">Omnibus GitLab</a> packages starting with GitLab 12.7. </p>
         <h2 id="hardware-requirements">Hardware requirements<a href="#hardware-requirements" title="Permalink" class="anchor"></a> </h2> 
         <h3 id="storage">Storage<a href="#storage" title="Permalink" class="anchor"></a> </h3> 
         <p>The necessary hard drive space largely depends on the size of the repositories you want to store in GitLab but as a <em>rule of thumb</em> you should have at least as much free space as all your repositories combined take up. </p>
         <p>If you want to be flexible about growing your hard drive space in the future consider mounting it using <a href="https://en.wikipedia.org/wiki/Logical_volume_management">logical volume management (LVM)</a> so you can add more hard drives when you need them. </p>
         <p>Apart from a local hard drive you can also mount a volume that supports the network file system (NFS) protocol. This volume might be located on a file server, a network attached storage (NAS) device, a storage area network (SAN) or on an Amazon Web Services (AWS) Elastic Block Store (EBS) volume. </p>
         <p>If you have enough RAM and a recent CPU the speed of GitLab is mainly limited by hard drive seek times. Having a fast drive (7200 RPM and up) or a solid state drive (SSD) will improve the responsiveness of GitLab. </p>
         <div class="admonition-wrapper note">
          <div class="admonition alert alert-info"> 
           <svg role="img" aria-label="note" class="gl-icon ml-1 mr-1 s16">
            <use href="/13.10/assets/images/icons.svg#information-o" />
            <title>note</title>
           </svg>Since file system performance may affect the overall performance of GitLab, <a href="../administration/nfs.html#avoid-using-cloud-based-file-systems">we dont recommend using cloud-based file systems for storage</a>.
          </div>
         </div> 
         <h3 id="cpu">CPU<a href="#cpu" title="Permalink" class="anchor"></a> </h3> 
         <p>CPU requirements are dependent on the number of users and expected workload. Your exact needs may be more, depending on your workload. Your workload is influenced by factors such as - but not limited to - how active your users are, how much automation you use, mirroring, and repository/change size. </p>
         <p>The following is the recommended minimum CPU hardware guidance for a handful of example GitLab user base sizes. </p>
         <ul> 
          <li> <strong>4 cores</strong> is the <strong>recommended</strong> minimum number of cores and supports up to 500 users </li>
          <li>8 cores supports up to 1000 users </li>
          <li>More users? Consult the <a href="../administration/reference_architectures/index.html">reference architectures page</a> </li>
         </ul> 
         <h3 id="memory">Memory<a href="#memory" title="Permalink" class="anchor"></a> </h3> 
         <p>Memory requirements are dependent on the number of users and expected workload. Your exact needs may be more, depending on your workload. Your workload is influenced by factors such as - but not limited to - how active your users are, how much automation you use, mirroring, and repository/change size. </p>
         <p>The following is the recommended minimum Memory hardware guidance for a handful of example GitLab user base sizes. </p>
         <ul> 
          <li> <strong>4GB RAM</strong> is the <strong>required</strong> minimum memory size and supports up to 500 users 
           <ul> 
            <li>Our <a href="https://about.gitlab.com/handbook/engineering/development/enablement/memory/">Memory Team</a> is working to reduce the memory requirement. </li>
           </ul> </li>
          <li>8GB RAM supports up to 1000 users </li>
          <li>More users? Consult the <a href="../administration/reference_architectures/index.html">reference architectures page</a> </li>
         </ul> 
         <p>In addition to the above, we generally recommend having at least 2GB of swap on your server, even if you currently have enough available RAM. Having swap will help reduce the chance of errors occurring if your available memory changes. We also recommend configuring the kernels swappiness setting to a low value like <code class="highlighter-rouge">10</code> to make the most of your RAM while still having the swap available when needed. </p>
         <h2 id="database">Database<a href="#database" title="Permalink" class="anchor"></a> </h2> 
         <p>PostgreSQL is the only supported database, which is bundled with the Omnibus GitLab package. You can also use an <a href="/13.10/omnibus/settings/database.html#using-a-non-packaged-postgresql-database-management-server">external PostgreSQL database</a>. Support for MySQL was removed in GitLab 12.1. Existing users using GitLab with MySQL/MariaDB are advised to <a href="../update/mysql_to_postgresql.html">migrate to PostgreSQL</a> before upgrading. </p>
         <h3 id="postgresql-requirements">PostgreSQL Requirements<a href="#postgresql-requirements" title="Permalink" class="anchor"></a> </h3> 
         <p>The server running PostgreSQL should have <em>at least</em> 5-10 GB of storage available, though the exact requirements <a href="../administration/reference_architectures/index.html">depend on the number of users</a>. </p>
         <p>We highly recommend using the minimum PostgreSQL versions (as specified in the following table) as these were used for development and testing: </p>
         <table> 
          <thead> 
           <tr> 
            <th>GitLab version </th>
            <th>Minimum PostgreSQL version </th>
           </tr>
          </thead>
          <tbody> 
           <tr> 
            <td>10.0 </td>
            <td>9.6 </td>
           </tr>
           <tr> 
            <td>13.0 </td>
            <td>11 </td>
           </tr>
          </tbody>
         </table> 
         <p>You must also ensure the following extensions are <a href="postgresql_extensions.html">loaded into every GitLab database</a>: </p>
         <table> 
          <thead> 
           <tr> 
            <th>Extension </th>
            <th>Minimum GitLab version </th>
           </tr>
          </thead>
          <tbody> 
           <tr> 
            <td><code class="highlighter-rouge">pg_trgm</code> </td>
            <td>8.6 </td>
           </tr>
           <tr> 
            <td><code class="highlighter-rouge">btree_gist</code> </td>
            <td>13.1 </td>
           </tr>
          </tbody>
         </table> 
         <div class="admonition-wrapper note">
          <div class="admonition alert alert-info"> 
           <svg role="img" aria-label="note" class="gl-icon ml-1 mr-1 s16">
            <use href="/13.10/assets/images/icons.svg#information-o" />
            <title>note</title>
           </svg>Support for <a href="https://about.gitlab.com/releases/2020/05/22/gitlab-13-0-released/#postgresql-11-is-now-the-minimum-required-version-to-install-gitlab">PostgreSQL 9.6 and 10 was removed in GitLab 13.0</a> so that GitLab can benefit from PostgreSQL 11 improvements, such as partitioning. For the schedule of transitioning to PostgreSQL 12, see <a href="https://gitlab.com/groups/gitlab-org/-/epics/2184">the related epic</a>.
          </div>
         </div> 
         <h4 id="additional-requirements-for-gitlab-geo">Additional requirements for GitLab Geo<a href="#additional-requirements-for-gitlab-geo" title="Permalink" class="anchor"></a> </h4> 
         <p>If youre using <a href="../administration/geo/index.html">GitLab Geo</a>, we strongly recommend running Omnibus GitLab-managed instances, as we actively develop and test based on those. We try to be compatible with most external (not managed by Omnibus GitLab) databases (for example, <a href="https://aws.amazon.com/rds/">AWS Relational Database Service (RDS)</a>), but we cant guarantee compatibility. </p>
         <h2 id="puma-settings">Puma settings<a href="#puma-settings" title="Permalink" class="anchor"></a> </h2> 
         <p>The recommended settings for Puma are determined by the infrastructure on which its running. Omnibus GitLab defaults to the recommended Puma settings. Regardless of installation method, you can tune the Puma settings. </p>
         <p>If youre using Omnibus GitLab, see <a href="/13.10/omnibus/settings/puma.html">Puma settings</a> for instructions on changing the Puma settings. If youre using the GitLab Helm chart, see the <a href="/13.10/charts/charts/gitlab/webservice/index.html"><code class="highlighter-rouge">webservice</code> chart</a>. </p>
         <h3 id="puma-workers">Puma workers<a href="#puma-workers" title="Permalink" class="anchor"></a> </h3> 
         <p>The recommended number of workers is calculated as the highest of the following: </p>
         <ul> 
          <li><code class="highlighter-rouge">2</code> </li>
          <li>Number of CPU cores - 1 </li>
         </ul> 
         <p>For example a node with 4 cores should be configured with 3 Puma workers. </p>
         <p>You can increase the number of Puma workers, providing enough CPU and memory capacity is available. A higher number of Puma workers will usually help to reduce the response time of the application and increase the ability to handle parallel requests. You must perform testing to verify the optimal settings for your infrastructure. </p>
         <h3 id="puma-threads">Puma threads<a href="#puma-threads" title="Permalink" class="anchor"></a> </h3> 
         <p>The recommended number of threads is dependent on several factors, including total memory, and use of <a href="../administration/gitaly/index.html#direct-access-to-git-in-gitlab">legacy Rugged code</a>. </p>
         <ul> 
          <li>If the operating system has a maximum 2 GB of memory, the recommended number of threads is <code class="highlighter-rouge">1</code>. A higher value will result in excess swapping, and decrease performance. </li>
          <li>If legacy Rugged code is in use, the recommended number of threads is <code class="highlighter-rouge">1</code>. </li>
          <li>In all other cases, the recommended number of threads is <code class="highlighter-rouge">4</code>. We dont recommend setting this higher, due to how <a href="https://en.wikipedia.org/wiki/Global_interpreter_lock">Ruby MRI multi-threading</a> works. </li>
         </ul> 
         <h2 id="unicorn-workers">Unicorn Workers<a href="#unicorn-workers" title="Permalink" class="anchor"></a> </h2> 
         <p>For most instances we recommend using: (CPU cores * 1.5) + 1 = Unicorn workers. For example a node with 4 cores would have 7 Unicorn workers. </p>
         <p>For all machines that have 2GB and up we recommend a minimum of three Unicorn workers. If you have a 1GB machine we recommend to configure only two Unicorn workers to prevent excessive swapping. </p>
         <p>As long as you have enough available CPU and memory capacity, its okay to increase the number of Unicorn workers and this will usually help to reduce the response time of the applications and increase the ability to handle parallel requests. </p>
         <p>To change the Unicorn workers when you have the Omnibus package (which defaults to the recommendation above) please see <a href="/13.10/omnibus/settings/unicorn.html">the Unicorn settings in the Omnibus GitLab documentation</a>. </p>
         <h2 id="redis-and-sidekiq">Redis and Sidekiq<a href="#redis-and-sidekiq" title="Permalink" class="anchor"></a> </h2> 
         <p>Redis stores all user sessions and the background task queue. The storage requirements for Redis are minimal, about 25kB per user. Sidekiq processes the background jobs with a multi-threaded process. This process starts with the entire Rails stack (200MB+) but it can grow over time due to memory leaks. On a very active server (10,000 billable users) the Sidekiq process can use 1GB+ of memory. </p>
         <h2 id="prometheus-and-its-exporters">Prometheus and its exporters<a href="#prometheus-and-its-exporters" title="Permalink" class="anchor"></a> </h2> 
         <p>As of Omnibus GitLab 9.0, <a href="https://prometheus.io">Prometheus</a> and its related exporters are enabled by default, to enable easy and in depth monitoring of GitLab. Approximately 200MB of memory will be consumed by these processes, with default settings. </p>
         <p>If you would like to disable Prometheus and its exporters or read more information about it, check the <a href="../administration/monitoring/prometheus/index.html">Prometheus documentation</a>. </p>
         <h2 id="gitlab-runner">GitLab Runner<a href="#gitlab-runner" title="Permalink" class="anchor"></a> </h2> 
         <p>We strongly advise against installing GitLab Runner on the same machine you plan to install GitLab on. Depending on how you decide to configure GitLab Runner and what tools you use to exercise your application in the CI environment, GitLab Runner can consume significant amount of available memory. </p>
         <p>Memory consumption calculations, that are available above, wont be valid if you decide to run GitLab Runner and the GitLab Rails application on the same machine. </p>
         <p>Its also not safe to install everything on a single machine, because of the <a href="/13.10/runner/security/">security reasons</a>, especially when you plan to use shell executor with GitLab Runner. </p>
         <p>We recommend using a separate machine for each GitLab Runner, if you plan to use the CI features. The GitLab Runner server requirements depend on: </p>
         <ul> 
          <li>The type of <a href="/13.10/runner/executors/">executor</a> you configured on GitLab Runner. </li>
          <li>Resources required to run build jobs. </li>
          <li>Job concurrency settings. </li>
         </ul> 
         <p>Since the nature of the jobs varies for each use case, you will need to experiment by adjusting the job concurrency to get the optimum setting. </p>
         <p>For reference, GitLab.coms <a href="../user/gitlab_com/index.html#shared-runners">auto-scaling shared runner</a> is configured so that a <strong>single job</strong> will run in a <strong>single instance</strong> with: </p>
         <ul> 
          <li>1vCPU. </li>
          <li>3.75GB of RAM. </li>
         </ul>
