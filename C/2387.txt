       <h3 id="about-continuous-integration"><a href="#about-continuous-integration">About continuous integration</a></h3> 
       <p>Continuous integration (CI) is a software practice that requires frequently committing code to a shared repository. Committing code more often detects errors sooner and reduces the amount of code a developer needs to debug when finding the source of an error. Frequent code updates also make it easier to merge changes from different members of a software development team. This is great for developers, who can spend more time writing code and less time debugging errors or resolving merge conflicts.</p> 
       <p>When you commit code to your repository, you can continuously build and test the code to make sure that the commit doesn't introduce errors. Your tests can include code linters (which check style formatting), security checks, code coverage, functional tests, and other custom checks.</p> 
       <p>Building and testing your code requires a server. You can build and test updates locally before pushing code to a repository, or you can use a CI server that checks for new code commits in a repository.</p> 
       <h3 id="about-continuous-integration-using-github-actions"><a href="#about-continuous-integration-using-github-actions">About continuous integration using GitHub Actions</a></h3> 
       <p> CI using GitHub Actions offers workflows that can build the code in your repository and run your tests. Workflows can run on GitHub-hosted virtual machines, or on machines that you host yourself. For more information, see "<a href="/en/actions/automating-your-workflow-with-github-actions/virtual-environments-for-github-hosted-runners">Virtual environments for GitHub-hosted runners</a>" and "<a href="/en/actions/automating-your-workflow-with-github-actions/about-self-hosted-runners">About self-hosted runners</a>."</p> 
       <p>You can configure your CI workflow to run when a GitHub event occurs (for example, when new code is pushed to your repository), on a set schedule, or when an external event occurs using the repository dispatch webhook.</p> 
       <p>GitHub runs your CI tests and provides the results of each test in the pull request, so you can see whether the change in your branch introduces an error. When all CI tests in a workflow pass, the changes you pushed are ready to be reviewed by a team member or merged. When a test fails, one of your changes may have caused the failure.</p> 
       <p>When you set up CI in your repository, GitHub analyzes the code in your repository and recommends CI workflows based on the language and framework in your repository. For example, if you use <a href="https://nodejs.org/en/">Node.js</a>, GitHub will suggest a template file that installs your Node.js packages and runs your tests. You can use the CI workflow template suggested by GitHub, customize the suggested template, or create your own custom workflow file to run your CI tests.</p> 
       <p><img src="/assets/images/help/repository/ci-with-actions-template-picker.png" alt="Screenshot of suggested continuous integration templates"></p> 
       <p>In addition to helping you set up CI workflows for your project, you can use GitHub Actions to create workflows across the full software development life cycle. For example, you can use actions to deploy, package, or release your project. For more information, see "<a href="/en/articles/about-github-actions">About GitHub Actions</a>."</p> 
       <p>For a definition of common terms, see "<a href="/en/github/automating-your-workflow-with-github-actions/core-concepts-for-github-actions">Core concepts for GitHub Actions</a>."</p> 
       <h3 id="supported-languages"><a href="#supported-languages">Supported languages</a></h3> 
       <p>GitHub offers CI workflow templates for a variety of languages and frameworks.</p> 
       <p>Browse the complete list of CI workflow templates offered by GitHub in the <a href="https://github.com/actions/starter-workflows/tree/main/ci">actions/starter-workflows</a> repository.</p> 
       <h3 id="skipping-workflow-runs"><a href="#skipping-workflow-runs">Skipping workflow runs</a></h3> 
       <p>If you want to temporarily prevent a workflow from being triggered, you can add a skip instruction to the commit message. Workflows that would otherwise be triggered <code>on: push</code> or <code>on: pull_request</code>, won't be triggered if you add any of the following strings to the commit message in a push, or the HEAD commit of a pull request:</p> 
       <ul> 
        <li><code>[skip ci]</code></li> 
        <li><code>[ci skip]</code></li> 
        <li><code>[no ci]</code></li> 
        <li><code>[skip actions]</code></li> 
        <li><code>[actions skip]</code></li> 
       </ul> 
       <p>Alternatively, you can end the commit message with two empty lines followed by either <code>skip-checks: true</code> or <code>skip-checks:true</code>.</p> 
       <p>You won't be able to merge the pull request if your repository is configured to require specific checks to pass first. To allow the pull request to be merged you can push a new commit to the pull request without the skip instruction in the commit message.</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> Skip instructions only apply to the <code>push</code> and <code>pull_request</code> events. For example, adding <code>[skip ci]</code> to a commit message won't stop a workflow that's triggered <code>on: pull_request_target</code> from running.</p> 
       </div> 
       <h3 id="notifications-for-workflow-runs"><a href="#notifications-for-workflow-runs">Notifications for workflow runs</a></h3> 
       <p>If you enable email or web notifications for GitHub Actions, you'll receive a notification when any workflow runs that you've triggered have completed. The notification will include the workflow run's status (including successful, failed, neutral, and canceled runs). You can also choose to receive a notification only when a workflow run has failed.</p> 
       <p>Notifications for scheduled workflows are sent to the user who initially created the workflow. If a different user updates the cron syntax in the workflow file, subsequent notifications will be sent to that user instead. If a scheduled workflow is disabled and then re-enabled, notifications will be sent to the user who re-enabled the workflow rather than the user who last modified the cron syntax.</p> 
       <p>
