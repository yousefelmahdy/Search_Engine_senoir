       <div class="d-flex flex-items-baseline flex-justify-between mt-3"> 
        <h1 class="border-bottom-0">Using SAML</h1> 
        <div class="d-none d-lg-block ml-2"> <button class="btn-link Link--secondary js-print tooltipped tooltipped-n" aria-label="Print this article"> <!-- From https://heroicons.dev/ --> 
          <svg fill="none" height="18" width="18" stroke-linecap="round" stroke-linejoin="round" stroke-width="2" viewbox="0 0 24 24" stroke="currentColor">
           <path d="M17 17h2a2 2 0 002-2v-4a2 2 0 00-2-2H5a2 2 0 00-2 2v4a2 2 0 002 2h2m2 4h6a2 2 0 002-2v-4a2 2 0 00-2-2H9a2 2 0 00-2 2v4a2 2 0 002 2zm8-12V5a2 2 0 00-2-2H9a2 2 0 00-2 2v4h10z"></path>
          </svg> </button>

       <div class="lead-mktg">
        <p>SAML is an XML-based standard for authentication and authorization. GitHub Enterprise Server can act as a service provider (SP) with your internal SAML identity provider (IdP).</p>
       </div> 
      </div> 
      <div class="article-grid-toc border-bottom border-xl-0 pb-4 mb-5 pb-xl-0 mb-xl-0"> 
       <div class="article-grid-toc-content"> 
        <h2 id="in-this-article" class="f5 mb-2"><a class="Link--primary" href="#in-this-article">In this article</a></h2> 
        <ul class="list-style-none pl-0 f5 mb-0"> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#supported-saml-services">Supported SAML services</a></li>

         <li class="ml-0  mb-2 lh-condensed"><a href="#saml-attributes">SAML attributes</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#configuring-saml-settings">Configuring SAML settings</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#updating-a-users-saml-nameid">Updating a user's SAML <code>NameID</code></a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#revoking-access-to-your-github-enterprise-server-instance">Revoking access to your GitHub Enterprise Server instance</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#response-message-requirements">Response message requirements</a></li>

      <div id="article-contents" class="article-grid-body"> 
       <p>If you want to authenticate users without adding them to your identity provider, you can configure built-in authentication. For more information, see "<a href="/en/enterprise-server@3.1/admin/guides/user-management/allowing-built-in-authentication-for-users-outside-your-identity-provider">Allowing built-in authentication for users outside your identity provider</a>."</p> 
       <h3 id="supported-saml-services"><a href="#supported-saml-services">Supported SAML services</a></h3> 
       <p>We offer limited support for all identity providers that implement the SAML 2.0 standard. We officially support these identity providers that have been internally tested:</p> 
       <ul> 
        <li>Active Directory Federation Services (AD FS)</li> 
        <li>Azure Active Directory (Azure AD)</li> 
        <li>Okta</li> 
        <li>OneLogin</li> 
        <li>PingOne</li> 
        <li>Shibboleth</li> 
       </ul> 
       <p>GitHub Enterprise Server does not support SAML Single Logout. To terminate an active SAML session, users should log out directly on your SAML IdP.</p> 
       <h3 id="username-considerations-with-saml"><a href="#username-considerations-with-saml">Username considerations with SAML</a></h3> 
       <p>Each GitHub Enterprise Server username is determined by one of the following assertions in the SAML response, ordered by priority:</p> 
       <ul> 
        <li>The custom username attribute, if defined and present</li> 
        <li>An <code>http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name</code> assertion, if present</li> 
        <li>An <code>http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress</code> assertion, if present</li> 
        <li>The <code>NameID</code> element</li> 
       </ul> 
       <p>The <code>NameID</code> element is required even if other attributes are present.</p> 
       <p>A mapping is created between the <code>NameID</code> and the GitHub Enterprise Server username, so the <code>NameID</code> should be persistent, unique, and not subject to change for the lifecycle of the user.</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note</strong>: If the <code>NameID</code> for a user does change on the IdP, the user will see an error message when they try to sign in to your GitHub Enterprise Server instance. To restore the user's access, you'll need to update the user account's <code>NameID</code> mapping. For more information, see "<a href="#updating-a-users-saml-nameid">Updating a user's SAML <code>NameID</code></a>."</p> 
       </div> 
       <p>GitHub Enterprise Server usernames can only contain alphanumeric characters and dashes (<code>-</code>). GitHub Enterprise Server will normalize any non-alphanumeric character in your account's username into a dash. For example, a username of <code>gregory.st.john</code> will be normalized to <code>gregory-st-john</code>. Note that normalized usernames also can't start or end with a dash. They also can't contain two consecutive dashes.</p> 
       <p>Usernames created from email addresses are created from the normalized characters that precede the <code>@</code> character.</p> 
       <p>If multiple accounts are normalized into the same GitHub Enterprise Server username, only the first user account is created. Subsequent users with the same username won't be able to sign in.</p>

       <h3 id="two-factor-authentication"><a href="#two-factor-authentication">Two-factor authentication</a></h3> 
       <p>When using SAML or CAS, two-factor authentication is not supported or managed on the GitHub Enterprise Server appliance, but may be supported by the external authentication provider. Two-factor authentication enforcement on organizations is not available. For more information about enforcing two-factor authentication on organizations, see "<a href="/en/enterprise-server@3.1/articles/requiring-two-factor-authentication-in-your-organization">Requiring two-factor authentication in your organization</a>."</p> 
       <h3 id="saml-metadata"><a href="#saml-metadata">SAML metadata</a></h3>

       <h3 id="response-message-requirements"><a href="#response-message-requirements">Response message requirements</a></h3> 
       <p>The response message must fulfill the following requirements:</p> 
       <ul> 
        <li>The <code>&lt;Destination&gt;</code> element must be provided on the root response document and match the ACS URL only when the root response document is signed. If the assertion is signed, it will be ignored.</li> 
        <li>The <code>&lt;Audience&gt;</code> element must always be provided as part of the <code>&lt;AudienceRestriction&gt;</code> element. It must match the <code>EntityId</code> for GitHub Enterprise Server. This is the URL to the GitHub Enterprise Server instance, such as <code>https://ghe.corp.example.com</code>.</li> 
        <li>Each assertion in the response <strong>must</strong> be protected by a digital signature. This can be accomplished by signing each individual <code>&lt;Assertion&gt;</code> element or by signing the <code>&lt;Response&gt;</code> element.</li> 
        <li>A <code>&lt;NameID&gt;</code> element must be provided as part of the <code>&lt;Subject&gt;</code> element. Any persistent name identifier format may be used.</li> 
        <li>The <code>Recipient</code> attribute must be present and set to the ACS URL. For example:</li>

       <h3 id="troubleshooting-saml-authentication"><a href="#troubleshooting-saml-authentication">Troubleshooting SAML authentication</a></h3> 
       <p>GitHub Enterprise Server logs error messages for failed SAML authentication in the authentication log at <em>/var/log/github/auth.log</em>. For more information about SAML response requirements, see "<a href="#response-message-requirements">Response message requirements</a>."</p> 
       <h4 id="error-another-user-already-owns-the-account"><a href="#error-another-user-already-owns-the-account">Error: "Another user already owns the account"</a></h4> 
       <p>When a user signs in to GitHub Enterprise Server for the first time with SAML authentication, GitHub Enterprise Server creates a user account on the instance and maps the SAML <code>NameID</code> to the account.</p> 
       <p>When the user signs in again, GitHub Enterprise Server compares the account's <code>NameID</code> mapping to the IdP's response. If the <code>NameID</code> in the IdP's response no longer matches the <code>NameID</code> that GitHub Enterprise Server expects for the user, the sign-in will fail. The user will see the following message.</p> 
       <blockquote> 
        <p>Another user already owns the account. Please have your administrator check the authentication log.</p> 
       </blockquote> 
       <p>The message typically indicates that the person's username or email address has changed on the IdP. Ensure that the <code>NameID</code> mapping for the user account on GitHub Enterprise Server matches the user's <code>NameID</code> on your IdP. For more information, see "<a href="#updating-a-users-saml-nameid">Updating a user's SAML <code>NameID</code></a>."</p> 
       <h4 id="error-recipient-in-saml-response-was-blank-or-not-valid"><a href="#error-recipient-in-saml-response-was-blank-or-not-valid">Error: Recipient in SAML response was blank or not valid</a></h4> 
       <p>If the <code>Recipient</code> does not match the ACS URL for your GitHub Enterprise Server instance, one of the following two error messages will appear in the authentication log when a user attempts to authenticate.</p> 
       <pre><code>Recipient in the SAML response must not be blank.
</code></pre> 
       <pre><code>Recipient in the SAML response was not valid.
</code></pre> 
       <p>Ensure that you set the value for <code>Recipient</code> on your IdP to the full ACS URL for your GitHub Enterprise Server instance. For example, <code>https://ghe.corp.example.com/saml/consume</code>.</p> 
       <h4 id="error-saml-response-is-not-signed-or-has-been-modified"><a href="#error-saml-response-is-not-signed-or-has-been-modified">Error: "SAML Response is not signed or has been modified"</a></h4> 
       <p>If your IdP does not sign the SAML response, or the signature does not match the contents, the following error message will appear in the authentication log.</p> 
       <pre><code>SAML Response is not signed or has been modified.
</code></pre> 
       <p>Ensure that you configure signed assertions for the GitHub Enterprise Server application on your IdP.</p> 
       <h4 id="error-audience-is-invalid-or-no-assertion-found"><a href="#error-audience-is-invalid-or-no-assertion-found">Error: "Audience is invalid" or "No assertion found"</a></h4> 
       <p>If the IdP's response has a missing or incorrect value for <code>Audience</code>, the following error message will appear in the authentication log.</p> 
       <pre><code class="hljs language-shell">Audience is invalid. Audience attribute does not match https://<em>YOUR-INSTANCE-URL</em></code></pre> 
       <p>Ensure that you set the value for <code>Audience</code> on your IdP to the <code>EntityId</code> for your GitHub Enterprise Server instance, which is the full URL to your GitHub Enterprise Server instance. For example, <code>https://ghe.corp.example.com</code>.</p>
