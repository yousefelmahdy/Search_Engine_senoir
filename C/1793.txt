       <h3 id="search"><a href="#search">Search</a></h3> 
       <p>Refer to this section of the site admin dashboard to search for users and repositories, and to query the <a href="#audit-log">audit log</a>.</p> 
       <h3 id="management-console"><a href="#management-console">Management Console</a></h3> 
       <p>Here you can launch the Management Console to manage virtual appliance settings such as the domain, authentication, and SSL.</p> 
       <h3 id="explore"><a href="#explore">Explore</a></h3> 
       <p>Data for GitHub's <a href="https://github.com/blog/1585-explore-what-is-trending-on-github">trending page</a> is calculated into daily, weekly, and monthly time spans for both repositories and developers. You can see when this data was last cached and queue up new trending calculation jobs from the <strong>Explore</strong> section.</p> 
       <h3 id="audit-log"><a href="#audit-log">Audit log</a></h3> 
       <p>GitHub Enterprise keeps a running log of audited actions that you can query.</p> 
       <p>By default, the audit log shows you a list of all audited actions in reverse chronological order. You can filter this list by entering key-value pairs in the <strong>Query</strong> text box and then clicking <strong>Search</strong>, as explained in "<a href="/en/enterprise-server@3.1/admin/guides/installation/searching-the-audit-log">Searching the audit log</a>."</p> 
       <p>For more information on audit logging in general, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/audit-logging">Audit logging</a>." For a full list of audited actions, see "<a href="/en/enterprise-server@3.1/admin/guides/installation/audited-actions">Audited actions</a>."</p> 
       <h3 id="reports"><a href="#reports">Reports</a></h3> 
       <p>If you need to get information on the users, organizations, and repositories in your GitHub Enterprise Server instance, you would ordinarily fetch JSON data through the <a href="/en/enterprise-server@3.1/rest">GitHub API</a>. Unfortunately, the API may not provide all of the data that you want and it requires a bit of technical expertise to use. The site admin dashboard offers a <strong>Reports</strong> section as an alternative, making it easy for you to download CSV reports with most of the information that you are likely to need for users, organizations, and repositories.</p> 
       <p>Specifically, you can download CSV reports that list</p> 
       <ul> 
        <li>all users</li> 
        <li>all users who have been active within the last month</li> 
        <li>all users who have been inactive for one month or more</li> 
        <li>all users who have been suspended</li> 
        <li>all organizations</li> 
        <li>all repositories</li> 
       </ul> 
       <p>You can also access these reports programmatically via standard HTTP authentication with a site admin account. You must use a personal access token with the <code>site_admin</code> scope. For more information, see "<a href="/en/enterprise-server@3.1/github/authenticating-to-github/creating-a-personal-access-token">Creating a personal access token</a>."</p> 
       <p>For example, here is how you would download the "all users" report using cURL:</p> 
       <pre><code class="hljs language-shell">curl -L -u <em>username</em>:<em>token</em> http(s)://<em>hostname</em>/stafftools/reports/all_users.csv</code></pre> 
       <p>To access the other reports programmatically, replace <code>all_users</code> with <code>active_users</code>, <code>dormant_users</code>, <code>suspended_users</code>, <code>all_organizations</code>, or <code>all_repositories</code>.</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> The initial <code>curl</code> request will return a 202 HTTP response if there are no cached reports available; a report will be generated in the background. You can send a second request to download the report. You can use a password or an OAuth token with the <code>site_admin</code> scope in place of a password.</p>

       <h3 id="indexing"><a href="#indexing">Indexing</a></h3> 
       <p>GitHub's <a href="https://github.com/blog/1381-a-whole-new-code-search">code search</a> features are powered by <a href="http://www.elasticsearch.org/">ElasticSearch</a>. This section of the site admin dashboard shows you the current status of your ElasticSearch cluster and provides you with several tools to control the behavior of searching and indexing. These tools are split into the following three categories.</p> 
       <h4 id="code-search"><a href="#code-search">Code search</a></h4> 
       <p>This allows you to enable or disable both search and index operations on source code.</p> 
       <h4 id="code-search-index-repair"><a href="#code-search-index-repair">Code search index repair</a></h4> 
       <p>This controls how the code search index is repaired. You can</p> 
       <ul> 
        <li>enable or disable index repair jobs</li> 
        <li>start a new index repair job</li> 
        <li>reset all index repair state</li> 
       </ul> 
       <p>GitHub Enterprise uses repair jobs to reconcile the state of the search index with data stored in a database (issues, pull requests, repositories, and users) and data stored in Git repositories (source code). This happens when</p> 
       <ul> 
        <li>a new search index is created;</li> 
        <li>missing data needs to be backfilled; or</li> 
        <li>old search data needs to be updated.</li> 
       </ul> 
       <p>In other words, repair jobs are started as needed and run in the backgroundthey are not scheduled by site admins in any way.</p> 
       <p>Furthermore, repair jobs use a "repair offset" for parallelization. This is an offset into the database table for the record being reconciled. Multiple background jobs can synchronize work based on this offset.</p> 
       <p>A progress bar shows the current status of a repair job across all of its background workers. It is the percentage difference of the repair offset with the highest record ID in the database. Don't worry about the value shown in the progress bar after a repair job has completed: because it shows the difference between the repair offset and the highest record ID in the database, it will decrease as more repositories are added to your GitHub Enterprise Server instance even though those repositories are actually indexed.</p> 
       <p>You can start a new code-search index repair job at any time. It will use a single CPU as it reconciles the search index with database and Git repository data. To minimize the effects this will have on I/O performance and reduce the chances of operations timing out, try to run a repair job during off-peak hours first. Monitor your system's load averages and CPU usage with a utility like <code>top</code>; if you don't notice any significant changes, it should be safe to run an index repair job during peak hours, as well.</p> 
       <h4 id="issues-index-repair"><a href="#issues-index-repair">Issues index repair</a></h4> 
       <p>This controls how the <a href="https://github.com/blog/831-issues-2-0-the-next-generation">Issues</a> index is repaired. You can</p> 
       <ul> 
        <li>enable or disable index repair jobs</li> 
        <li>start a new index repair job</li> 
        <li>reset all index repair state</li> 
       </ul>
