        <p>Your GitHub App can perform actions on behalf of a user, like creating an issue, creating a deployment, and using other supported endpoints.</p>

      <div id="article-contents" class="article-grid-body"> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> Expiring user tokens are currently an optional feature and subject to change. To opt in or out of the user-to-server token expiration feature, see "<a href="/en/developers/apps/activating-optional-features-for-apps">Activating optional features for apps</a>." For more information, see "<a href="https://developer.github.com/changes/2020-04-30-expiring-user-to-server-access-tokens-for-github-apps">Expiring user-to-server access tokens for GitHub Apps</a>."</p> 
       </div> 
       <p>When your GitHub App acts on behalf of a user, it performs user-to-server requests. These requests must be authorized with a user's access token. User-to-server requests include requesting data for a user, like determining which repositories to display to a particular user. These requests also include actions triggered by a user, like running a build.</p> 
       <p>To keep user-to-server access tokens more secure, you can use access tokens that will expire after 8 hours, and a refresh token that can be exchanged for a new access token. For more information, see "<a href="/en/apps/building-github-apps/refreshing-user-to-server-access-tokens">Refreshing user-to-server access tokens</a>."</p> 
       <h3 id="identifying-users-on-your-site"><a href="#identifying-users-on-your-site">Identifying users on your site</a></h3> 
       <p>To authorize users for standard apps that run in the browser, use the <a href="#web-application-flow">web application flow</a>.</p> 
       <p>To authorize users for headless apps without direct access to the browser, such as CLI tools or Git credential managers, use the <a href="#device-flow">device flow</a>. The device flow uses the OAuth 2.0 <a href="https://tools.ietf.org/html/rfc8628">Device Authorization Grant</a>.</p> 
       <h3 id="web-application-flow"><a href="#web-application-flow">Web application flow</a></h3> 
       <p>Using the web application flow, the process to identify users on your site is:</p> 
       <ol> 
        <li>Users are redirected to request their GitHub identity</li> 
        <li>Users are redirected back to your site by GitHub</li> 
        <li>Your GitHub App accesses the API with the user's access token</li> 
       </ol> 
       <p>If you select <strong>Request user authorization (OAuth) during installation</strong> when creating or modifying your app, step 1 will be completed during app installation. For more information, see "<a href="/en/apps/installing-github-apps/#authorizing-users-during-installation">Authorizing users during installation</a>."</p> 
       <h4 id="1-request-a-users-github-identity"><a href="#1-request-a-users-github-identity">1. Request a user's GitHub identity</a></h4> 
       <pre><code>GET https://github.com/login/oauth/authorize
</code></pre> 
       <p>When your GitHub App specifies a <code>login</code> parameter, it prompts users with a specific account they can use for signing in and authorizing your app.</p> 
       <h5 id="parameters"><a href="#parameters">Parameters</a></h5> 
       <table>
        <thead>
         <tr>
          <th>Name</th>
          <th>Type</th>
          <th>Description</th>
         </tr>
        </thead>
        <tbody>
         <tr>
          <td><code>client_id</code></td>
          <td><code>string</code></td>

         </tr>

         </tr>

       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> You don't need to provide scopes in your authorization request. Unlike traditional OAuth, the authorization token is limited to the permissions associated with your GitHub App and those of the user.</p> 
       </div> 
       <h4 id="2-users-are-redirected-back-to-your-site-by-github"><a href="#2-users-are-redirected-back-to-your-site-by-github">2. Users are redirected back to your site by GitHub</a></h4> 
       <p>If the user accepts your request, GitHub redirects back to your site with a temporary <code>code</code> in a code parameter as well as the state you provided in the previous step in a <code>state</code> parameter. If the states don't match, the request was created by a third party and the process should be aborted.</p> 
       <div class="extended-markdown note border rounded-1 mb-4 p-3 color-border-info color-bg-info f5"> 
        <p><strong>Note:</strong> If you select <strong>Request user authorization (OAuth) during installation</strong> when creating or modifying your app, GitHub returns a temporary <code>code</code> that you will need to exchange for an access token. The <code>state</code> parameter is not returned when GitHub initiates the OAuth flow during app installation.</p> 
       </div> 
       <p>Exchange this <code>code</code> for an access token. When expiring tokens are enabled, the access token expires in 8 hours and the refresh token expires in 6 months. Every time you refresh the token, you get a new refresh token. For more information, see "<a href="/en/developers/apps/refreshing-user-to-server-access-tokens">Refreshing user-to-server access tokens</a>."</p> 
       <p>Expiring user tokens are currently an optional feature and subject to change. To opt-in to the user-to-server token expiration feature, see "<a href="/en/developers/apps/activating-optional-features-for-apps">Activating optional features for apps</a>."</p> 
       <pre><code>POST https://github.com/login/oauth/access_token
</code></pre>

       <h5 id="response"><a href="#response">Response</a></h5> 
       <p>By default, the response takes the following form. The response parameters <code>expires_in</code>, <code>refresh_token</code>, and <code>refresh_token_expires_in</code> are only returned when you enable expiring user-to-server access tokens.</p> 
       <pre><code class="hljs language-json">{
  <span class="hljs-attr">"access_token"</span>: <span class="hljs-string">"ghu_16C7e42F292c6912E7710c838347Ae178B4a"</span>,
  <span class="hljs-attr">"expires_in"</span>: <span class="hljs-number">28800</span>,
  <span class="hljs-attr">"refresh_token"</span>: <span class="hljs-string">"ghr_1B4a2e77838347a7E420ce178F2E7c6912E169246c34E1ccbF66C46812d16D5B1A9Dc86A1498"</span>,
  <span class="hljs-attr">"refresh_token_expires_in"</span>: <span class="hljs-number">15811200</span>,
  <span class="hljs-attr">"scope"</span>: <span class="hljs-string">""</span>,
  <span class="hljs-attr">"token_type"</span>: <span class="hljs-string">"bearer"</span>
}
</code></pre> 
       <h4 id="3-your-github-app-accesses-the-api-with-the-users-access-token"><a href="#3-your-github-app-accesses-the-api-with-the-users-access-token">3. Your GitHub App accesses the API with the user's access token</a></h4> 
       <p>The user's access token allows the GitHub App to make requests to the API on behalf of a user.</p> 
       <pre><code>Authorization: token OAUTH-TOKEN
GET https://api.github.com/user
</code></pre> 
       <p>For example, in curl you can set the Authorization header like this:</p> 
       <pre><code class="hljs language-shell">curl -H "Authorization: token OAUTH-TOKEN" https://api.github.com/user</code></pre> 
       <h3 id="device-flow"><a href="#device-flow">Device flow</a></h3> 
       <p>The device flow allows you to authorize users for a headless app, such as a CLI tool or Git credential manager.</p> 
       <p>For more information about authorizing users using the device flow, see "<a href="/en/developers/apps/authorizing-oauth-apps#device-flow">Authorizing OAuth Apps</a>".</p> 
       <h3 id="check-which-installations-resources-a-user-can-access"><a href="#check-which-installations-resources-a-user-can-access">Check which installation's resources a user can access</a></h3> 
       <p>Once you have an OAuth token for a user, you can check which installations that user can access.</p> 
       <pre><code>Authorization: token OAUTH-TOKEN
GET /user/installations
</code></pre> 
       <p>You can also check which repositories are accessible to a user for an installation.</p> 
       <pre><code>Authorization: token OAUTH-TOKEN
GET /user/installations/:installation_id/repositories
</code></pre> 
       <p>More details can be found in: <a href="/en/rest/reference/apps#list-app-installations-accessible-to-the-user-access-token">List app installations accessible to the user access token</a> and <a href="/en/rest/reference/apps#list-repositories-accessible-to-the-user-access-token">List repositories accessible to the user access token</a>.</p> 
       <h3 id="handling-a-revoked-github-app-authorization"><a href="#handling-a-revoked-github-app-authorization">Handling a revoked GitHub App authorization</a></h3> 
       <p>If a user revokes their authorization of a GitHub App, the app will receive the <a href="/en/webhooks/event-payloads/#github_app_authorization"><code>github_app_authorization</code></a> webhook by default. GitHub Apps cannot unsubscribe from this event. Anyone can revoke their authorization of a GitHub App from their <a href="https://github.com/settings/apps/authorizations">GitHub account settings page</a>. Revoking the authorization of a GitHub App does not uninstall the GitHub App. You should program your GitHub App so that when it receives this webhook, it stops calling the API on behalf of the person who revoked the token. If your GitHub App continues to use a revoked access token, it will receive the <code>401 Bad Credentials</code> error.</p> 
       <h3 id="user-level-permissions"><a href="#user-level-permissions">User-level permissions</a></h3> 
       <p>You can add user-level permissions to your GitHub App to access user resources, such as user emails, that are granted by individual users as part of the <a href="#identifying-users-on-your-site">user authorization flow</a>. User-level permissions differ from <a href="/en/rest/reference/permissions-required-for-github-apps">repository and organization-level permissions</a>, which are granted at the time of installation on an organization or user account.</p> 
       <p>You can select user-level permissions from within your GitHub App's settings in the <strong>User permissions</strong> section of the <strong>Permissions &amp; webhooks</strong> page. For more information on selecting permissions, see "<a href="/en/apps/managing-github-apps/editing-a-github-app-s-permissions">Editing a GitHub App's permissions</a>."</p> 
       <p>When a user installs your app on their account, the installation prompt will list the user-level permissions your app is requesting and explain that the app can ask individual users for these permissions.</p> 
       <p>Because user-level permissions are granted on an individual user basis, you can add them to your existing app without prompting users to upgrade. You will, however, need to send existing users through the user authorization flow to authorize the new permission and get a new user-to-server token for these requests.</p>
