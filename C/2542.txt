       <div class="d-flex flex-items-baseline flex-justify-between mt-3"> 
        <h1 class="border-bottom-0">Migrating OAuth Apps to GitHub Apps</h1> 
        <div class="d-none d-lg-block ml-2"> <button class="btn-link Link--secondary js-print tooltipped tooltipped-n" aria-label="Print this article"> <!-- From https://heroicons.dev/ --> 
          <svg fill="none" height="18" width="18" stroke-linecap="round" stroke-linejoin="round" stroke-width="2" viewbox="0 0 24 24" stroke="currentColor">
           <path d="M17 17h2a2 2 0 002-2v-4a2 2 0 00-2-2H5a2 2 0 00-2 2v4a2 2 0 002 2h2m2 4h6a2 2 0 002-2v-4a2 2 0 00-2-2H9a2 2 0 00-2 2v4a2 2 0 002 2zm8-12V5a2 2 0 00-2-2H9a2 2 0 00-2 2v4h10z"></path>
          </svg> </button> 
        </div> 
       </div> 
       <div class="lead-mktg">
        <p>Learn about the advantages of migrating your OAuth App to a GitHub App and how to migrate an OAuth App that isn't listed on GitHub Marketplace. </p>
       </div> 
      </div> 
      <div class="article-grid-toc border-bottom border-xl-0 pb-4 mb-5 pb-xl-0 mb-xl-0"> 
       <div class="article-grid-toc-content"> 
        <h2 id="in-this-article" class="f5 mb-2"><a class="Link--primary" href="#in-this-article">In this article</a></h2> 
        <ul class="list-style-none pl-0 f5 mb-0"> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#reasons-for-switching-to-github-apps">Reasons for switching to GitHub Apps</a></li> 
         <li class="ml-0  mb-2 lh-condensed"><a href="#converting-an-oauth-app-to-a-github-app">Converting an OAuth App to a GitHub App</a></li> 
        </ul> 
       </div> 
      </div> 
      <div id="article-contents" class="article-grid-body"> 
       <p>This article provides guidelines for existing integrators who are considering migrating from an OAuth App to a GitHub App.</p> 
       <h3 id="reasons-for-switching-to-github-apps"><a href="#reasons-for-switching-to-github-apps">Reasons for switching to GitHub Apps</a></h3> 
       <p><a href="/en/apps">GitHub Apps</a> are the officially recommended way to integrate with GitHub because they offer many advantages over a pure OAuth-based integration:</p> 
       <ul> 
        <li><a href="/en/apps/differences-between-apps/#requesting-permission-levels-for-resources">Fine-grained permissions</a> target the specific information a GitHub App can access, allowing the app to be more widely used by people and organizations with security policies than OAuth Apps, which cannot be limited by permissions.</li> 
        <li><a href="/en/apps/differences-between-apps/#token-based-identification">Short-lived tokens</a> provide a more secure authentication method over OAuth tokens. An OAuth token does not expire until the person who authorized the OAuth App revokes the token. GitHub Apps use tokens that expire quickly, creating a much smaller window of time for compromised tokens to be in use.</li> 
        <li><a href="/en/apps/differences-between-apps/#webhooks">Built-in, centralized webhooks</a> receive events for all repositories and organizations the app can access. Conversely, OAuth Apps require configuring a webhook for each repository and organization accessible to the user.</li> 
        <li><a href="/en/apps/differences-between-apps/#machine-vs-bot-accounts">Bot accounts</a> don't consume a GitHub seat and remain installed even when the person who initially installed the app leaves the organization.</li> 
        <li>Built-in support for OAuth is still available to GitHub Apps using <a href="/en/apps/building-github-apps/identifying-and-authorizing-users-for-github-apps">user-to-server endpoints</a>.</li> 
        <li>Dedicated <a href="/en/apps/building-github-apps/understanding-rate-limits-for-github-apps">API rate limits</a> for bot accounts scale with your integration.</li> 
        <li>Repository owners can <a href="/en/apps/differences-between-apps/#who-can-install-github-apps-and-authorize-oauth-apps">install GitHub Apps</a> on organization repositories. If a GitHub App's configuration has permissions that request an organization's resources, the org owner must approve the installation.</li> 
        <li>Open Source community support is available through <a href="/en/rest/overview/libraries">Octokit libraries</a> and other frameworks such as <a href="https://probot.github.io/">Probot</a>.</li> 
        <li>Integrators building GitHub Apps have opportunities to adopt earlier access to APIs.</li> 
       </ul> 
       <h3 id="converting-an-oauth-app-to-a-github-app"><a href="#converting-an-oauth-app-to-a-github-app">Converting an OAuth App to a GitHub App</a></h3> 
       <p>These guidelines assume that you have a registered OAuth App that may or may not be listed in GitHub Marketplace. At a high level, you'll need to follow these steps:</p> 
       <ol> 
        <li><a href="#review-the-available-api-endpoints-for-github-apps">Review the available API endpoints for GitHub Apps</a></li> 
        <li><a href="#design-to-stay-within-api-rate-limits">Design to stay within API rate limits</a></li> 
        <li><a href="#register-a-new-github-app">Register a new GitHub App</a></li> 
        <li><a href="#determine-the-permissions-your-app-requires">Determine the permissions your app requires</a></li> 
        <li><a href="#subscribe-to-webhooks">Subscribe to webhooks</a></li> 
        <li><a href="#understand-the-different-methods-of-authentication">Understand the different methods of authentication</a></li> 
        <li><a href="#direct-users-to-install-your-github-app-on-repositories">Direct users to install your GitHub App on repositories</a></li> 
        <li><a href="#remove-any-unnecessary-repository-hooks">Remove any unnecessary repository hooks</a></li> 
        <li><a href="#encourage-users-to-revoke-access-to-your-oauth-app">Encourage users to revoke access to your OAuth app</a></li> 
       </ol> 
       <h4 id="review-the-available-api-endpoints-for-github-apps"><a href="#review-the-available-api-endpoints-for-github-apps">Review the available API endpoints for GitHub Apps</a></h4> 
       <p>While the majority of <a href="/en/rest">REST API</a> endpoints and <a href="/en/graphql">GraphQL</a> queries are available to GitHub Apps today, we are still in the process of enabling some endpoints. Review the <a href="/en/rest/overview/endpoints-available-for-github-apps">available REST endpoints</a> to ensure that the endpoints you need are compatible with GitHub Apps. Note that some of the API endpoints enabled for GitHub Apps allow the app to act on behalf of the user. See "<a href="/en/apps/building-github-apps/identifying-and-authorizing-users-for-github-apps/#user-to-server-requests">User-to-server requests</a>" for a list of endpoints that allow a GitHub App to authenticate as a user.</p> 
       <p>We recommend reviewing the list of API endpoints you need as early as possible. Please let Support know if there is an endpoint you require that is not yet enabled for GitHub Apps.</p> 
       <h4 id="design-to-stay-within-api-rate-limits"><a href="#design-to-stay-within-api-rate-limits">Design to stay within API rate limits</a></h4> 
       <p>GitHub Apps use <a href="/en/apps/building-github-apps/understanding-rate-limits-for-github-apps">sliding rules for rate limits</a>, which can increase based on the number of repositories and users in the organization. A GitHub App can also make use of <a href="/en/rest#conditional-requests">conditional requests</a> or consolidate requests by using the <a href="/en/graphql">GraphQL API V4</a>.</p> 
       <h4 id="register-a-new-github-app"><a href="#register-a-new-github-app">Register a new GitHub App</a></h4> 
       <p>Once you've decided to make the switch to GitHub Apps, you'll need to <a href="/en/apps/building-github-apps">create a new GitHub App</a>.</p> 
       <h4 id="determine-the-permissions-your-app-requires"><a href="#determine-the-permissions-your-app-requires">Determine the permissions your app requires</a></h4> 
       <p>When registering your GitHub App, you'll need to select the permissions required by each endpoint used in your app's code. See "<a href="/en/rest/reference/permissions-required-for-github-apps">GitHub App permissions</a>" for a list of the permissions needed for each endpoint available to GitHub Apps.</p> 
       <p>In your GitHub App's settings, you can specify whether your app needs <code>No Access</code>, <code>Read-only</code>, or <code>Read &amp; Write</code> access for each permission type. The fine-grained permissions allow your app to gain targeted access to the subset of data you need. We recommend specifying the smallest set of permissions possible that provides the desired functionality.</p> 
       <h4 id="subscribe-to-webhooks"><a href="#subscribe-to-webhooks">Subscribe to webhooks</a></h4> 
       <p>After you've created a new GitHub App and selected its permissions, you can select the webhook events you wish to subscribe it to. See "<a href="/en/apps/managing-github-apps/editing-a-github-app-s-permissions">Editing a GitHub App's permissions</a>" to learn how to subscribe to webhooks.</p> 
       <h4 id="understand-the-different-methods-of-authentication"><a href="#understand-the-different-methods-of-authentication">Understand the different methods of authentication</a></h4> 
       <p>GitHub Apps primarily use a token-based authentication that expires after a short amount of time, providing more security than an OAuth token that does not expire. It’s important to understand the different methods of authentication available to you and when you need to use them:</p> 
       <ul> 
        <li>A <strong>JSON Web Token (JWT)</strong> <a href="/en/apps/building-github-apps/authenticating-with-github-apps/#authenticating-as-a-github-app">authenticates as the GitHub App</a>. For example, you can authenticate with a <strong>JWT</strong> to fetch application installation details or exchange the <strong>JWT</strong> for an <strong>installation access token</strong>.</li> 
        <li>An <strong>installation access token</strong> <a href="/en/apps/building-github-apps/authenticating-with-github-apps/#authenticating-as-an-installation">authenticates as a specific installation of your GitHub App</a> (also called server-to-server requests). For example, you can authenticate with an <strong>installation access token</strong> to open an issue or provide feedback on a pull request.</li> 
        <li>An <strong>OAuth access token</strong> can <a href="/en/apps/building-github-apps/identifying-and-authorizing-users-for-github-apps/#identifying-users-on-your-site">authenticate as a user of your GitHub App</a> (also called user-to-server requests). For example, you can use an OAuth access token to authenticate as a user when a GitHub App needs to verify a user’s identity or act on a user’s behalf.</li> 
       </ul> 
       <p>The most common scenario is to authenticate as a specific installation using an <strong>installation access token</strong>.</p> 
       <h4 id="direct-users-to-install-your-github-app-on-repositories"><a href="#direct-users-to-install-your-github-app-on-repositories">Direct users to install your GitHub App on repositories</a></h4> 
       <p>Once you've made the transition from an OAuth App to a GitHub App, you will need to let users know that the GitHub App is available to install. For example, you can include an installation link for the GitHub App in a call-to-action banner inside your application. To ease the transition, you can use query parameters to identify the user or organization account that is going through the installation flow for your GitHub App and pre-select any repositories your OAuth App had access to. This allows users to easily install your GitHub App on repositories you already have access to.</p> 
       <h5 id="query-parameters"><a href="#query-parameters">Query parameters</a></h5> 
       <table>
        <thead>
         <tr>
          <th>Name</th>
          <th>Description</th>
         </tr>
        </thead>
        <tbody>
         <tr>
          <td><code>suggested_target_id</code></td>
          <td><strong>Required</strong>: ID of the user or organization that is installing your GitHub App.</td>
         </tr>
         <tr>
          <td><code>repository_ids[]</code></td>
          <td>Array of repository IDs. If omitted, we select all repositories. The maximum number of repositories that can be pre-selected is 100.</td>
         </tr>
        </tbody>
       </table> 
       <h5 id="example-url"><a href="#example-url">Example URL</a></h5> 
       <pre><code>https://github.com/apps/YOUR_APP_NAME/installations/new/permissions?suggested_target_id=ID_OF_USER_OR_ORG&amp;repository_ids[]=REPO_A_ID&amp;repository_ids[]=REPO_B_ID
</code></pre> 
       <p>You'll need to replace <code>YOUR_APP_NAME</code> with the name of your GitHub App, <code>ID_OF_USER_OR_ORG</code> with the ID of your target user or organization, and include up to 100 repository IDs (<code>REPO_A_ID</code> and <code>REPO_B_ID</code>). To get a list of repositories your OAuth App has access to, use the <a href="/en/rest/reference/repos#list-repositories-for-the-authenticated-user">List repositories for the authenticated user</a> and <a href="/en/rest/reference/repos#list-organization-repositories">List organization repositories</a> endpoints.</p> 
       <h4 id="remove-any-unnecessary-repository-hooks"><a href="#remove-any-unnecessary-repository-hooks">Remove any unnecessary repository hooks</a></h4> 
       <p>Once your GitHub App has been installed on a repository, you should remove any unnecessary webhooks that were created by your legacy OAuth App. If both apps are installed on a repository, they may duplicate functionality for the user. To remove webhooks, you can listen for the <a href="/en/webhooks/event-payloads/#installation_repositories"><code>installation_repositories</code> webhook</a> with the <code>repositories_added</code> action and <a href="/en/rest/reference/repos#delete-a-repository-webhook">Delete a repository webhook</a> on those repositories that were created by your OAuth App.</p>
